# 12_06_トランザクション設計

## 1. 目的

本書は、AI Flow Designer BackendにおけるTransaction設計を定義する。

Flow保存では、Lane、Stage、Node、Link、Commentを一括保存する必要がある。一部だけ保存されるとSSOTが破綻し、AIレビュー、Export、Version Snapshot、設計書生成の前提が崩れる。

## 2. 基本方針

- TransactionはService層で開始する。
- RepositoryではTransactionを開始しない。
- RepositoryではSaveChangesを呼ばない。
- UnitOfWorkを用意してSaveChangesとTransactionを集約する。
- 失敗時は全件Rollbackする。
- ネストしたTransactionは禁止する。
- Transaction範囲は必要最小限にする。

## 3. UnitOfWork

UnitOfWorkはDbContextのSaveChangesとTransactionを抽象化する。

```csharp
public interface IUnitOfWork
{
    Task<int> SaveChangesAsync(CancellationToken ct);
    Task<IDbContextTransaction> BeginTransactionAsync(CancellationToken ct);
}
```

実装はInfrastructure層に配置する。

```text
FlowDesigner.Infrastructure/Persistence/EfCoreUnitOfWork.cs
```

## 4. Transaction対象処理

Transactionを必須とする処理:

- Project作成とProjectMember作成
- Flow作成
- Flow一括保存
- 自動保存
- Version Snapshot作成
- Version復元
- Template適用
- Template複製
- 画像メタデータ登録
- Project削除
- Flow削除

## 5. Flow保存Transaction

Flow保存は最重要Transactionである。

保存対象:

1. Flow
2. Lane
3. Stage
4. FlowNode
5. FlowLink
6. FlowComment
7. Revision
8. 必要に応じてSnapshot

処理フロー:

```text
1. JWT認証
2. Project権限確認
3. 編集ロック確認
4. Revision確認
5. DTO Validation
6. 構造整合性Validation
7. Transaction開始
8. 差分分類
9. Lane反映
10. Stage反映
11. Node反映
12. Link反映
13. Comment反映
14. Flow Revision更新
15. SaveChanges
16. Commit
17. Response返却
```

## 6. Rollback方針

以下の場合はRollbackする。

- Validation通過後のDB更新中に例外
- SaveChanges失敗
- Link整合性エラーが更新中に検出された場合
- Snapshot作成失敗
- Template適用失敗
- 画像メタデータ登録失敗

Rollback後、Serviceは適切な例外を送出し、Middlewareが標準ErrorResponseへ変換する。

## 7. SaveChanges方針

原則として、1つのUseCaseにつきSaveChangesは1回に集約する。

例外:

- 物理ファイル保存とDBメタデータ保存のように、外部ストレージとの整合を取る場合
- 将来のOutbox Pattern導入時

Repository内でSaveChangesしない。

## 8. Template適用Transaction

Template適用時は、TemplateJsonからLane / Stage / Node / Link / Commentを生成し、IDを再採番して現在Flowへ追加または置換する。

Transaction内で行うこと:

- Template存在確認
- Project権限確認
- ID再採番
- Flow構造反映
- Revision更新
- SaveChanges

失敗時は適用前に戻す。

## 9. Version Snapshot Transaction

Snapshot作成時は、保存済みFlowを読み込み、SnapshotJsonを生成し、VersionNoを採番する。

Transaction内で行うこと:

- Flow存在確認
- VersionNo採番
- SnapshotJson作成
- FlowVersionSnapshot追加
- SaveChanges

VersionNo重複を避けるため、Flow単位で最新VersionNoを確認する。

## 10. Version復元 Transaction

Version復元は破壊的変更になり得るため、Transaction必須とする。

処理方針:

1. 現在Flowを復元前Snapshotとして保存する。
2. 指定VersionのSnapshotJsonを読み込む。
3. 現在Flowへ反映する。
4. Revision更新。
5. Commit。

失敗時は現在Flowを変更しない。

## 11. 画像管理Transaction

画像アップロードでは、物理ファイル保存とDBメタデータ保存の順序に注意する。

初期方針:

1. ファイル検証
2. 物理ファイル一時保存
3. Transaction開始
4. IMAGE_FILE登録
5. SaveChanges
6. Commit
7. 一時ファイルを正式配置へ移動

移動失敗時の補償処理を用意する。

## 12. 禁止事項

- RepositoryでBeginTransactionする。
- RepositoryでSaveChangesする。
- Transaction内で外部APIを長時間呼び出す。
- Transaction内でPDF生成のような重い処理を行う。
- ネストTransactionを行う。
- Flow保存中に部分Commitする。

## 13. テスト観点

- Flow保存成功時に全要素が保存される。
- Node保存後、Link保存失敗時にNodeもRollbackされる。
- Repository単体ではSaveChangesされない。
- Template適用失敗時にFlowが変更されない。
- Version復元失敗時に現在Flowが保持される。
- Transaction失敗時に標準エラーが返る。

## 14. 完了条件

- Transaction境界がService層に定義されている。
- UnitOfWork方針が定義されている。
- Flow保存、Snapshot、Template、画像管理のTransaction方針が明確である。
- RepositoryでSaveChangesしない設計になっている。
- AIが本書を読んでTransaction実装に着手できる。
