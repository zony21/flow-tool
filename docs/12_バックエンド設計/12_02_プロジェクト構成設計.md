# 12_02_プロジェクト構成設計

## 1. 目的

本書は、AI Flow Designer Backendのプロジェクト構成を定義する。

BackendはASP.NET Core .NET 8を用いて実装する。SSOTの保存、検証、Export、Version Snapshot、認証・認可を扱うため、単一プロジェクトにすべてを詰め込まず、責務ごとに分割する。

## 2. 推奨Solution構成

```text
backend/
 ├─ FlowDesigner.sln
 ├─ FlowDesigner.Api/
 ├─ FlowDesigner.Application/
 ├─ FlowDesigner.Domain/
 ├─ FlowDesigner.Infrastructure/
 └─ FlowDesigner.Tests/
```

## 3. FlowDesigner.Api

HTTP APIの入口を担当する。

配置例:

```text
FlowDesigner.Api/
 ├─ Controllers/
 ├─ Middlewares/
 ├─ Filters/
 ├─ Extensions/
 ├─ Program.cs
 └─ appsettings.json
```

責務:

- Controller定義
- Middleware登録
- 認証・認可設定
- DI登録呼び出し
- CORS設定
- Swagger設定
- 共通例外Middleware登録

禁止事項:

- 業務ロジックを置かない。
- Repositoryを直接呼ばない。
- DbContextを直接操作しない。

## 4. FlowDesigner.Application

ユースケースと業務処理の中心を担当する。

配置例:

```text
FlowDesigner.Application/
 ├─ Services/
 ├─ DTOs/
 ├─ Validators/
 ├─ Interfaces/
 │   ├─ Repositories/
 │   ├─ Auth/
 │   ├─ Storage/
 │   └─ Export/
 ├─ Models/
 └─ Mappers/
```

責務:

- Service定義
- Request / Response DTO
- Validation
- Repository Interface
- Export Interface
- Storage Interface
- Transaction境界制御
- DTO変換

## 5. FlowDesigner.Domain

SSOTの意味とルールを表す。

配置例:

```text
FlowDesigner.Domain/
 ├─ Entities/
 ├─ ValueObjects/
 ├─ Enums/
 ├─ Rules/
 └─ Exceptions/
```

責務:

- Domain Entity
- Value Object
- Domain Rule
- Domain Exception
- Enum定義

禁止事項:

- EF Coreへ依存しない。
- ASP.NET Coreへ依存しない。
- API DTOへ依存しない。

## 6. FlowDesigner.Infrastructure

外部技術依存を担当する。

配置例:

```text
FlowDesigner.Infrastructure/
 ├─ Persistence/
 │   ├─ AppDbContext.cs
 │   ├─ Configurations/
 │   └─ Migrations/
 ├─ Repositories/
 ├─ Auth/
 ├─ Storage/
 ├─ Export/
 └─ Extensions/
```

責務:

- EF Core DbContext
- Repository実装
- SQLite / SQL Server接続
- GitHub OAuth連携
- JWT生成
- File Storage実装
- Export Service実装

## 7. FlowDesigner.Tests

テストを管理する。

配置例:

```text
FlowDesigner.Tests/
 ├─ Unit/
 ├─ Application/
 ├─ Infrastructure/
 ├─ Api/
 └─ TestData/
```

責務:

- Domain Rule Test
- Service Test
- Repository Test
- API Test
- Export Test

## 8. 依存方向

依存方向は以下とする。

```text
Api -> Application -> Domain
Infrastructure -> Application
Infrastructure -> Domain
Tests -> All
```

ApplicationはInfrastructure実装へ依存しない。
ApplicationはRepository Interfaceを定義し、Infrastructureが実装する。

## 9. 参照関係

| プロジェクト | 参照先 |
| --- | --- |
| Api | Application, Infrastructure |
| Application | Domain |
| Domain | なし |
| Infrastructure | Application, Domain |
| Tests | Api, Application, Domain, Infrastructure |

## 10. DI登録

DI登録はExtension Methodで分割する。

例:

```text
services.AddApplication();
services.AddInfrastructure(configuration);
services.AddApi();
```

## 11. 設定ファイル

設定はappsettingsに分ける。

```text
appsettings.json
appsettings.Development.json
appsettings.Production.json
```

主な設定:

- ConnectionStrings
- Jwt
- GitHubOAuth
- Storage
- Logging
- Export

Secretはappsettingsへ直接書かない。

## 12. 命名規則

| 対象 | 例 |
| --- | --- |
| Controller | FlowController |
| Service Interface | IFlowService |
| Service Impl | FlowService |
| Repository Interface | IFlowRepository |
| Repository Impl | FlowRepository |
| DTO | FlowDetailDto |
| Request | SaveFlowRequest |
| Response | SaveFlowResponse |
| Validator | SaveFlowRequestValidator |

## 13. 完了条件

- Solution構成が明確である。
- 依存方向が明確である。
- Domainが外部技術に依存しない。
- ApplicationがInfrastructure実装に依存しない。
- AIが本書を読んでBackendプロジェクトを作成できる。
