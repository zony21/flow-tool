# 12_07_例外処理設計

## 1. 本書の目的

本書は、AI Flow Designer のバックエンドにおける例外処理設計を定義する。

対象は ASP.NET Core .NET 8 で実装する Web API であり、Controller / Service / Repository / EF Core / 認証処理 / Export 処理 / 画像管理処理で発生する例外を、API利用者およびフロントエンドが一貫して扱える形へ変換することを目的とする。

本システムでは、図面ではなく構造化データを正とするため、例外処理は単なるエラー返却ではなく、構造化データの整合性を守るための重要機能として扱う。

## 2. 基本方針

例外処理は以下の方針で統一する。

- Controller で try-catch を多用しない
- 例外は原則として Service 層で意味付けする
- Repository 層では業務例外を生成しない
- 共通例外ハンドリング Middleware で API レスポンス形式へ変換する
- Entity Framework Core の例外は外部へ直接返さない
- 内部例外の詳細はログへ記録する
- API レスポンスには必要最小限の情報のみ返却する
- フロントエンドで判定しやすい errorCode を必ず返却する
- 入力検証エラーと業務ルール違反は区別する
- 想定外例外は InternalServerError として扱う

## 3. 例外分類

本システムの例外は以下に分類する。

|分類|HTTP|概要|
|---|---:|---|
|ValidationException|400|DTO入力値が不正|
|BusinessRuleException|409|業務ルール違反|
|NotFoundException|404|対象データなし|
|UnauthorizedException|401|未認証|
|ForbiddenException|403|権限不足|
|ConflictException|409|Revision競合・編集ロック競合|
|FileStorageException|500|画像・ファイル保存失敗|
|ExportException|500|Mermaid/PDF/JSON出力失敗|
|DatabaseException|500|DB永続化失敗|
|UnexpectedException|500|想定外例外|

## 4. 共通エラーレスポンス形式

APIのエラーレスポンスは以下の形式で統一する。

```json
{
  "success": false,
  "errorCode": "FLOW_REVISION_CONFLICT",
  "message": "Flow has been updated by another user.",
  "details": [
    {
      "field": "revision",
      "message": "Requested revision is older than current revision."
    }
  ],
  "traceId": "00-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx-xxxxxxxxxxxxxxxx-00"
}
```

## 5. フィールド定義

|項目|必須|説明|
|---|---|---|
|success|必須|失敗時は false|
|errorCode|必須|フロントエンド判定用コード|
|message|必須|利用者または開発者向け概要|
|details|任意|項目別エラー|
|traceId|必須|ログ追跡用ID|

## 6. errorCode 命名規則

errorCode は以下の規則とする。

```text
{DOMAIN}_{REASON}
```

例:

```text
PROJECT_NOT_FOUND
FLOW_NOT_FOUND
FLOW_REVISION_CONFLICT
FLOW_LOCKED_BY_OTHER_USER
NODE_INVALID_POSITION
LINK_INVALID_ENDPOINT
TEMPLATE_NAME_DUPLICATED
IMAGE_UPLOAD_FAILED
EXPORT_MERMAID_FAILED
```

## 8. ValidationException

DTOの入力値が不正な場合に使用する。

対象例:

- 必須項目なし
- 最大文字数超過
- GUID形式不正
- enum範囲外
- 座標値不正
- サイズが0以下
- LinkのFROM/TO未指定
- NodeType未指定
- FlowId未指定

ValidationException は FluentValidation または独自Validatorで検出する。

## 9. Validation エラー例

```json
{
  "success": false,
  "errorCode": "VALIDATION_FAILED",
  "message": "Request validation failed.",
  "details": [
    {
      "field": "nodes[0].displayName",
      "message": "Display name is required."
    },
    {
      "field": "links[0].toNodeId",
      "message": "To node id is required."
    }
  ],
  "traceId": "00-..."
}
```

## 10. BusinessRuleException

業務ルール違反の場合に使用する。

対象例:

- Viewer が編集しようとした
- Flow が編集ロック中
- 削除対象のLaneにNodeがあるが削除方法が未指定
- Stage削除時にNode移動先が未指定
- Template名が重複
- Flow Version復元時に対象Versionが削除済
- NodeTypeが無効
- Linkの始点または終点Nodeが削除済

## 11. NotFoundException

対象データが存在しない場合に使用する。

対象例:

- Projectが存在しない
- Flowが存在しない
- Nodeが存在しない
- Linkが存在しない
- Templateが存在しない
- FlowVersionが存在しない

論理削除済データも、通常APIでは NotFound として扱う。

## 12. ConflictException

同時編集やRevision不一致を扱う。

対象例:

- Flow保存時の revision が最新でない
- 編集ロックを他ユーザーが保持している
- Version復元中にFlowが更新された
- Template複写時に同名が作成された

ConflictException はフロントエンドで再読込・差分確認を促す。

## 13. DatabaseException

DBアクセス失敗時に使用する。

ただし、DbUpdateException をそのままControllerへ返してはならない。

Service層または共通Middlewareで DatabaseException へ変換する。

記録する情報:

- DbContext操作名
- 対象テーブル
- 対象ID
- ユーザーID
- traceId
- InnerException

APIレスポンスへSQL文や接続文字列を返してはならない。

## 14. FileStorageException

画像・ファイル保存で失敗した場合に使用する。

対象例:

- 画像保存先ディレクトリが存在しない
- 書き込み権限がない
- Hash計算失敗
- 一時ファイル移動失敗
- SVG検証失敗
- ファイルサイズ超過

SQLiteにはメタデータのみ保持し、実体ファイルはストレージ保存するため、DB更新とファイル保存の整合性を考慮する。

## 15. ExportException

成果物出力で失敗した場合に使用する。

対象例:

- Mermaid Flowchart変換失敗
- Mermaid Sequence変換失敗
- PDF生成失敗
- JSON出力失敗
- AI DSL生成失敗
- 構造化データ不整合による変換不可

ExportExceptionでは、対象FlowId、ExportType、VersionIdをログに記録する。

## 16. 共通 Middleware

例外は `ApiExceptionHandlingMiddleware` で捕捉する。

責務:

- 例外種別判定
- HTTPステータス決定
- errorCode決定
- レスポンス生成
- ログ出力
- traceId付与

Controllerは通常 try-catch しない。

## 17. Middleware 擬似コード

```csharp
public sealed class ApiExceptionHandlingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<ApiExceptionHandlingMiddleware> _logger;

    public async Task InvokeAsync(HttpContext context)
    {
        try
        {
            await _next(context);
        }
        catch (AppException ex)
        {
            await HandleAppExceptionAsync(context, ex);
        }
        catch (Exception ex)
        {
            await HandleUnexpectedExceptionAsync(context, ex);
        }
    }
}
```

## 18. AppException 基底

```csharp
public abstract class AppException : Exception
{
    public string ErrorCode { get; }
    public int StatusCode { get; }
    public IReadOnlyList<ApiErrorDetail> Details { get; }

    protected AppException(
        string errorCode,
        string message,
        int statusCode,
        IReadOnlyList<ApiErrorDetail>? details = null,
        Exception? innerException = null)
        : base(message, innerException)
    {
        ErrorCode = errorCode;
        StatusCode = statusCode;
        Details = details ?? Array.Empty<ApiErrorDetail>();
    }
}
```

## 20. Service層での例外利用

例:

```csharp
var flow = await _flowRepository.FindByIdAsync(flowId);

if (flow == null)
{
    throw new NotFoundAppException(
        "FLOW_NOT_FOUND",
        "Flow was not found.");
}
```

## 21. Revision競合の例

```csharp
if (request.Revision != flow.Revision)
{
    throw new ConflictAppException(
        "FLOW_REVISION_CONFLICT",
        "Flow has been updated by another user.");
}
```

## 22. 編集ロック競合の例

```csharp
if (lockInfo.UserId != currentUser.UserId)
{
    throw new ConflictAppException(
        "FLOW_LOCKED_BY_OTHER_USER",
        "This flow is locked by another user.");
}
```

## 23. フロントエンドでの扱い

フロントエンドは errorCode を見て制御する。

例:

|errorCode|画面挙動|
|---|---|
|FLOW_REVISION_CONFLICT|再読込確認ダイアログ|
|FLOW_LOCKED_BY_OTHER_USER|編集不可表示|
|VALIDATION_FAILED|項目エラー表示|
|PROJECT_NOT_FOUND|一覧へ戻す|
|EXPORT_PDF_FAILED|再実行案内|
|IMAGE_UPLOAD_FAILED|アップロード失敗通知|

## 24. ログ出力方針

例外ログには以下を含める。

- traceId
- userId
- projectId
- flowId
- requestPath
- httpMethod
- errorCode
- exceptionType
- message
- stackTrace
- innerException

個人情報やGitHub OAuthトークンは出力しない。

## 25. セキュリティ注意点

APIレスポンスへ以下を含めてはならない。

- SQL文
- 接続文字列
- StackTrace
- OAuth token
- JWT
- ファイル物理パス
- サーバ内部ディレクトリ
- 環境変数
- InnerException全文

## 26. テスト観点

単体テスト:

- 各AppExceptionのStatusCode確認
- errorCode確認
- details確認

結合テスト:

- Validation失敗時に400
- NotFound時に404
- Revision競合時に409
- 権限不足時に403
- 未認証時に401
- 想定外例外時に500

E2E:

- 編集競合時にフロントエンドが再読込確認を表示する
- Flow保存失敗時にデータがロールバックされる
- Export失敗時にエラー通知される

## 27. 完了条件

本設計の完了条件は以下とする。

- 共通例外形式が全APIで統一されている
- Controllerに不要なtry-catchが存在しない
- AppException系が整備されている
- フロントエンドがerrorCodeで制御可能
- ログからtraceIdで追跡可能
- 想定外例外でもレスポンス形式が崩れない
