# 12_09_認証・認可設計

## 1. 本書の目的

本書は、AI Flow Designer のバックエンドにおける認証・認可設計を定義する。

本システムはGitHub OAuthによるログインを採用し、APIアクセスにはJWTを利用する。複数人利用を前提とし、Project単位で権限を管理する。

## 2. 基本方針

- 認証はGitHub OAuthで行う
- API認可はJWTで行う
- Project単位で権限を判定する
- EntityをAPIへ直接返さない
- 認可判定はControllerまたはActionFilterで実施する
- Service層でも重要な認可は再確認する
- Viewerは参照のみ可能
- Flow単位の編集ロックと権限を組み合わせる
- 管理者は編集ロック解除可能

## 3. ロール

本システムのロールは以下とする。

|ロール|概要|
|---|---|
|Admin|システム全体管理者|
|ProjectAdmin|Project管理者|
|Editor|編集者|
|Viewer|閲覧者|

## 5. USER_ACCOUNT

GitHub OAuthログイン成功時、USER_ACCOUNTを作成または更新する。

保持項目:

- USER_ID
- GITHUB_USER_ID
- LOGIN_NAME
- DISPLAY_NAME
- AVATAR_URL
- EMAIL
- LAST_LOGIN_AT
- IS_DELETED
- SYSTEM_CREATE_USER_ID
- SYSTEM_CREATE_DATETIME
- SYSTEM_UPDATE_USER_ID
- SYSTEM_UPDATE_DATETIME

## 6. JWT Claim

JWTには以下を含める。

|Claim|説明|
|---|---|
|sub|USER_ID|
|github_user_id|GitHub User ID|
|name|表示名|
|iat|発行日時|
|exp|有効期限|
|jti|JWT ID|

ProjectRoleはJWTへ大量に含めない。Project単位の権限はDBで確認する。

## 7. JWT有効期限

初期方針:

- Access Token: 60分
- Refresh Token: 初期実装では持たない
- 再ログインはGitHub OAuthで行う

将来的にRefresh Tokenを追加する場合、DBへHash化して保持する。

## 8. GitHub OAuth フロー

1. フロントエンドがログインボタン押下
2. Backendの `/api/auth/github/login` へ遷移
3. BackendがGitHub認可URLへリダイレクト
4. GitHub認証
5. `/api/auth/github/callback` へ戻る
6. Authorization CodeをAccess Tokenへ交換
7. GitHub User APIでユーザー情報取得
8. USER_ACCOUNT作成または更新
9. JWT発行
10. フロントエンドへリダイレクト

## 9. 認証API

```text
GET  /api/auth/github/login
GET  /api/auth/github/callback
POST /api/auth/logout
GET  /api/auth/me
```

## 11. Projectメンバー

Project単位のメンバー管理テーブルを用意する。

- SYSTEM_UPDATE_USER_ID
- SYSTEM_UPDATE_DATETIME

## 12. 認可判定の基本

認可は以下の順に判定する。

1. JWTが有効か
2. USER_ACCOUNTが存在するか
3. USER_ACCOUNTが削除済でないか
4. 対象Projectが存在するか
5. ProjectMemberに登録されているか
6. 必要Roleを満たすか
7. Flow編集の場合は編集ロックを満たすか

## 13. 認可失敗時

未認証:

- HTTP 401
- errorCode: AUTH_UNAUTHORIZED

権限不足:

- HTTP 403
- errorCode: AUTH_FORBIDDEN

Project権限なし:

- HTTP 403
- errorCode: PROJECT_ACCESS_DENIED

## 14. Controller実装方針

Controllerでは `[Authorize]` を付与する。

Project単位権限が必要なAPIでは、独自AttributeまたはServiceを利用する。

例:

```csharp
[Authorize]
[HttpPost("{projectId}/flows")]
public async Task<ActionResult<ApiResponse<FlowDto>>> CreateFlow(
    Guid projectId,
    CreateFlowRequest request)
{
    await _authorizationService.RequireProjectRoleAsync(
        projectId,
        ProjectRole.Editor);

    var result = await _flowService.CreateAsync(projectId, request);
    return Ok(ApiResponse.Success(result));
}
```

## 15. 認可Service

```text
IProjectAuthorizationService
```

責務:

- ProjectRole取得
- RequiredRole判定
- Admin判定
- ProjectAdmin判定
- Editor判定
- Viewer判定
- 権限不足時の例外送出

## 16. ロール階層

ロール階層は以下とする。

```text
Admin > ProjectAdmin > Editor > Viewer
```

Adminは全Projectへアクセス可能とする。ただし監査ログ上はAdmin操作として記録する。

## 17. 編集ロックとの関係

Flow編集時は、権限だけでなく編集ロックも確認する。

編集可能条件:

- ProjectRoleがEditor以上
- Flowが他ユーザーによりロックされていない
- ロック期限切れの場合は再取得可能
- 自分がロック保持者の場合は編集可能

## 18. Viewer制御

Viewerは以下のみ可能。

- Project閲覧
- Flow閲覧
- Version閲覧
- Export
- Template閲覧

Viewerは禁止:

- Flow保存
- Node追加
- Link変更
- Comment追加
- Version復元
- Template作成
- Project設定変更

## 19. Export権限

ExportはViewer以上で可能とする。

理由:

- Mermaid/PDF/JSONは閲覧成果物
- 設計書ドラフトは閲覧者にも必要な場合がある

ただし、将来的に機密度設定を追加する場合はProject設定で制御する。

## 20. Template権限

ユーザーテンプレート:

- Editor以上が作成可能
- 作成者本人またはProjectAdminが更新・削除可能

標準テンプレート:

- Adminのみ作成・更新・削除可能

## 21. Setting権限

個人設定:

- 本人のみ更新可能

Project設定:

- ProjectAdmin以上のみ更新可能

## 22. セキュリティ禁止事項

- JWTをログ出力しない
- GitHub AccessTokenをDBに保存しない
- Authorization Codeをログ出力しない
- ClientSecretをリポジトリへ含めない
- ProjectRoleをフロントエンドだけで判定しない
- Controllerのみの認可に依存しない
- 削除済ユーザーを認証済として扱わない

## 23. CORS方針

初期開発ではフロントエンドURLのみ許可する。

許可例:

- http://localhost:5173
- 開発環境URL
- 本番フロントURL

全許可 `AllowAnyOrigin` は本番禁止。

## 24. Cookie / Header

JWTは初期実装では Authorization Header で送信する。

```text
Authorization: Bearer {token}
```

将来的にCookie化する場合は以下を検討する。

- HttpOnly
- Secure
- SameSite
- CSRF対策

## 25. テスト観点

単体テスト:

- Role階層判定
- Editor以上判定
- Viewer禁止判定
- Admin全許可判定

結合テスト:

- 未認証で401
- ViewerがFlow保存で403
- EditorがFlow保存で200
- Project未参加で403
- 削除済ユーザーで401または403

E2E:

- GitHubログイン後にユーザー情報表示
- Viewerでは編集ボタン非活性
- Editorでは編集可能
- 他ユーザーロック中は編集不可

## 26. 完了条件

- GitHub OAuthログインが動作する
- JWT認証が全APIに適用されている
- Project単位認可が実装されている
- Viewer/Editor/ProjectAdmin/Adminの権限差が反映されている
- 権限エラーが共通エラーレスポンスで返る
- 認証情報がログへ漏れない
