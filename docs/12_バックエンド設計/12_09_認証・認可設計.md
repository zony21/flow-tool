# 12_09_認証・認可設計

## 1. 目的

本書は、AI Flow Designer Backendにおける認証・認可設計を定義する。

本システムはGitHub OAuthによるログインを採用し、APIアクセスにはJWTを利用する。複数人利用を前提とし、Project単位で権限を管理する。

## 2. 基本方針

- 認証はGitHub OAuthで行う。
- API認証はJWTで行う。
- Project単位で権限を判定する。
- Project RoleはJWTへ大量に含めずDBで確認する。
- EntityをAPIへ直接返さない。
- ControllerまたはFilterで認証を確認する。
- Service層でも重要な認可を再確認する。
- Viewerは参照のみ可能とする。
- Flow単位の編集ロックと権限を組み合わせる。
- 管理者は編集ロック解除可能とする。

## 5. USER_ACCOUNT

GitHub OAuthログイン成功時、USER_ACCOUNTを作成または更新する。

保持項目:

- USER_ID
- GITHUB_USER_ID
- LOGIN_NAME
- DISPLAY_NAME
- AVATAR_URL
- EMAIL
- LAST_LOGIN_AT
- IS_DELETED
- SYSTEM_CREATE_USER_ID
- SYSTEM_CREATE_DATETIME
- SYSTEM_UPDATE_USER_ID
- SYSTEM_UPDATE_DATETIME

## 6. JWT Claim

JWTには以下を含める。

| Claim | 説明 |
| --- | --- |
| sub | USER_ID |
| github_user_id | GitHub User ID |
| name | 表示名 |
| iat | 発行日時 |
| exp | 有効期限 |
| jti | JWT ID |

ProjectRoleはJWTへ大量に含めない。Project単位の権限はDBで確認する。

## 7. JWT有効期限

初期方針:

- Access Token: 60分
- Refresh Token: 初期実装では持たない
- 再ログインはGitHub OAuthで行う

将来的にRefresh Tokenを追加する場合、DBへHash化して保持する。

## 8. GitHub OAuthフロー

1. Frontendがログインボタン押下。
2. Backendの `/api/auth/github/login` へ遷移。
3. BackendがGitHub認可URLへリダイレクト。
4. GitHubで認証。
5. `/api/auth/github/callback` へ戻る。
6. Authorization CodeをAccess Tokenへ交換。
7. GitHub User APIでユーザー情報取得。
8. USER_ACCOUNT作成または更新。
9. JWT発行。
10. Frontendへリダイレクト。

## 9. 認証API

```text
GET  /api/auth/github/login
GET  /api/auth/github/callback
POST /api/auth/logout
GET  /api/auth/me
```

## 11. PROJECT_MEMBER

Project単位のメンバー管理テーブルを用意する。

保持項目:

- PROJECT_MEMBER_ID

- USER_ID
- ROLE
- IS_DELETED

- SYSTEM_CREATE_DATETIME
- SYSTEM_UPDATE_USER_ID
- SYSTEM_UPDATE_DATETIME

## 12. 認可判定の基本

認可は以下の順に判定する。

1. JWTが有効か。
2. USER_ACCOUNTが存在するか。
3. USER_ACCOUNTが削除済でないか。
4. 対象Projectが存在するか。
5. ProjectMemberに登録されているか。
6. 必要Roleを満たすか。
7. Flow編集の場合は編集ロックを満たすか。

## 13. 認可失敗時

未認証:

- HTTP 401
- errorCode: AUTH_UNAUTHORIZED

権限不足:

- HTTP 403
- errorCode: AUTH_FORBIDDEN

Project権限なし:

- HTTP 403
- errorCode: PROJECT_ACCESS_DENIED

## 14. Policy設計

ASP.NET CoreのPolicyを利用する。

例:

```text
ProjectViewerPolicy
ProjectEditorPolicy
ProjectAdminPolicy
SystemAdminPolicy
```

ProjectIdをRouteまたはRequestから取得し、ProjectAuthorizationServiceで判定する。

## 15. Service層での再確認

重要操作ではService層でも認可を確認する。

対象:

- Flow保存
- Flow削除
- Version復元
- Template適用
- メンバー変更
- 編集ロック強制解除

理由:

- Controller以外からServiceが呼ばれる将来拡張に備えるため。
- テスト時に認可漏れを検出しやすくするため。

## 16. 編集ロックとの関係

Editor以上でも、他ユーザーがFlow編集ロックを保持している場合は保存できない。

判定順:

1. Project権限確認
2. Editor以上確認
3. 編集ロック確認
4. Revision確認

## 17. セキュリティ注意点

- OAuth stateを検証する。
- GitHub Access Tokenをログ出力しない。
- JWTをログ出力しない。
- JWT Secretは設定ファイルへ直書きしない。
- ProjectRoleをFrontendだけで判定しない。
- Viewerが編集APIを呼んでもBackendで拒否する。

## 18. テスト観点

- 未ログインでAPIを呼ぶと401になる。
- ViewerがFlow保存を呼ぶと403になる。
- EditorがExportを実行できる。
- ViewerがExportを実行できる。
- ProjectMemberでないユーザーはProjectへアクセスできない。
- 編集ロック中のFlow保存が拒否される。
- OAuth tokenやJWTがログに出ない。

## 19. 完了条件

- GitHub OAuthフローが定義されている。
- JWT Claimと有効期限が定義されている。
- Project単位の権限管理が定義されている。
- PolicyとService層再確認方針が定義されている。
- AIが本書を読んで認証・認可を実装できる。
