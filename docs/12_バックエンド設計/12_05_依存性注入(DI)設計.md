# 12_05_依存性注入(DI)設計

## 1. 目的

本書は、AI Flow Designer Backendにおける依存性注入（DI）設計を定義する。

ASP.NET Core標準DIコンテナを利用し、Controller、Service、Repository、DbContext、Export、Storage、Auth関連処理を疎結合に保つ。

## 2. 基本方針

- Constructor Injectionを採用する。
- ControllerはService Interfaceへ依存する。
- ServiceはRepository Interfaceへ依存する。
- Application層はInfrastructure実装へ依存しない。
- DbContextはScopedとする。
- RepositoryはScopedとする。
- ServiceはScopedとする。
- 状態を持たないUtilityのみSingletonを許可する。
- SingletonにScoped依存を持たせない。

## 3. DI登録構成

DI登録は各プロジェクトのExtension Methodへ分離する。

```csharp
builder.Services.AddApplication();
builder.Services.AddInfrastructure(builder.Configuration);
builder.Services.AddApi();
```

配置例:

```text
FlowDesigner.Application/DependencyInjection.cs
FlowDesigner.Infrastructure/DependencyInjection.cs
FlowDesigner.Api/DependencyInjection.cs
```

## 4. Application登録

Application層ではService、Validator、Mapperを登録する。

```csharp
public static IServiceCollection AddApplication(this IServiceCollection services)
{
    services.AddScoped<IProjectService, ProjectService>();
    services.AddScoped<IFlowService, FlowService>();
    services.AddScoped<IFlowQueryService, FlowQueryService>();
    services.AddScoped<IFlowSaveService, FlowSaveService>();
    services.AddScoped<IFlowVersionService, FlowVersionService>();
    services.AddScoped<ITemplateService, TemplateService>();
    services.AddScoped<IExportService, ExportService>();
    services.AddScoped<IImageFileService, ImageFileService>();
    services.AddScoped<IProjectAuthorizationService, ProjectAuthorizationService>();

    services.AddValidatorsFromAssembly(typeof(DependencyInjection).Assembly);

    return services;
}
```

## 5. Infrastructure登録

Infrastructure層ではDbContext、Repository、Auth、Storage、Export実装を登録する。

```csharp
public static IServiceCollection AddInfrastructure(this IServiceCollection services, IConfiguration configuration)
{
    services.AddDbContext<AppDbContext>(options =>
        options.UseSqlite(configuration.GetConnectionString("DefaultConnection")));

    services.AddScoped<IProjectRepository, ProjectRepository>();
    services.AddScoped<IFlowRepository, FlowRepository>();
    services.AddScoped<ILaneRepository, LaneRepository>();
    services.AddScoped<IStageRepository, StageRepository>();
    services.AddScoped<IFlowNodeRepository, FlowNodeRepository>();
    services.AddScoped<IFlowLinkRepository, FlowLinkRepository>();
    services.AddScoped<IFlowCommentRepository, FlowCommentRepository>();
    services.AddScoped<IFlowVersionRepository, FlowVersionRepository>();
    services.AddScoped<ITemplateRepository, TemplateRepository>();
    services.AddScoped<IImageFileRepository, ImageFileRepository>();
    services.AddScoped<IUnitOfWork, EfCoreUnitOfWork>();

    services.AddScoped<IJwtTokenService, JwtTokenService>();
    services.AddScoped<IGitHubOAuthClient, GitHubOAuthClient>();
    services.AddScoped<IFileStorageService, LocalFileStorageService>();

    services.AddScoped<IMermaidExportRenderer, MermaidExportRenderer>();
    services.AddScoped<IPdfExportRenderer, PdfExportRenderer>();
    services.AddScoped<IJsonExportRenderer, JsonExportRenderer>();
    services.AddScoped<IAiDslExportRenderer, AiDslExportRenderer>();

    return services;
}
```

## 6. Api登録

Api層ではMiddleware、認証、認可、Swagger、CORSを登録する。

```csharp
public static IServiceCollection AddApi(this IServiceCollection services)
{
    services.AddControllers();
    services.AddEndpointsApiExplorer();
    services.AddSwaggerGen();
    services.AddHttpContextAccessor();

    return services;
}
```

## 7. Options Pattern

設定値はOptions Patternで注入する。

対象:

- JwtOptions
- GitHubOAuthOptions
- StorageOptions
- ExportOptions
- CorsOptions

```csharp
services.Configure<JwtOptions>(configuration.GetSection("Jwt"));
services.Configure<GitHubOAuthOptions>(configuration.GetSection("GitHubOAuth"));
services.Configure<StorageOptions>(configuration.GetSection("Storage"));
```

Secretはappsettingsへ直接書かず、環境変数またはSecret管理を利用する。

## 8. ライフタイム設計

| 対象 | Lifetime | 理由 |
| --- | --- | --- |
| DbContext | Scoped | Request単位で扱う |
| Repository | Scoped | DbContextと同じ寿命にする |
| Service | Scoped | Request単位の業務処理 |
| UnitOfWork | Scoped | DbContextと同じ寿命 |
| OAuth Client | Scoped | Request文脈を扱う可能性がある |
| Storage Service | Scoped | 設定とRequest文脈を扱う |
| Stateless Utility | Singleton | 状態を持たない場合のみ |

## 9. 禁止事項

- SingletonからDbContextを参照する。
- ControllerからRepositoryを直接注入する。
- Application層にInfrastructure実装を注入する。
- Service内でnewしてRepositoryを生成する。
- Static Serviceで状態を保持する。
- DbContextを複数Service間で手動生成する。

## 10. テスト方針

DI設計のテストでは以下を確認する。

- ServiceProviderを構築できる。
- 主要ServiceをResolveできる。
- RepositoryがScopedで登録されている。
- DbContextがScopedで登録されている。
- SingletonがScoped依存を持たない。
- Test用Repository差し替えが可能である。

## 11. 完了条件

- Application / Infrastructure / ApiごとにDI登録が分離されている。
- Service / Repository / DbContextのLifetimeが明確である。
- Options Patternの利用方針が定義されている。
- ControllerからRepositoryを直接呼ばない構成になっている。
- AIが本書を読んでDI登録を実装できる。
