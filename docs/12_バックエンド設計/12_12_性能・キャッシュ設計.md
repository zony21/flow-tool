# 12_12_性能・キャッシュ設計

## 1. 本書の目的

本書は、AI Flow Designer のバックエンドにおける性能・キャッシュ設計を定義する。

本システムは、1000ノード、10000リンク、Undo1000回規模を性能試験対象とする。初期DBはSQLiteだが、将来的にSQL Serverへ移行するため、DB非依存で性能劣化しにくい設計とする。

## 2. 性能目標

初期目標は以下とする。

|処理|目標|
|---|---:|
|Project一覧取得|1秒以内|
|Flow一覧取得|1秒以内|
|Flow読込 1000ノード/10000リンク|3秒以内|
|Flow保存 1000ノード/10000リンク|5秒以内|
|Mermaid出力|3秒以内|
|JSON出力|3秒以内|
|PDF出力|10秒以内|
|Version作成|5秒以内|
|Version比較|5秒以内|

## 3. 基本方針

- 読込処理はAsNoTrackingを基本とする
- 一覧APIでは詳細データを取得しない
- Flow詳細取得は必要な関連データのみ取得する
- N+1を禁止する
- 大量データは一括取得する
- 更新は差分保存を基本とする
- 不要なSaveChangesを行わない
- キャッシュは整合性を壊さない範囲に限定する
- SQLite前提の過剰最適化をしない
- SQL Server移行を妨げない

## 4. Flow読込最適化

Flow読込では以下を一括取得する。

- FLOW
- LANE
- STAGE
- FLOW_NODE
- FLOW_LINK
- FLOW_COMMENT
- NODE_TYPE

Repositoryで個別NodeごとにLinkを取得してはならない。

禁止例:

```text
foreach node:
  GetLinksByNodeId(node.Id)
```

推奨:

```text
Get all links by flowId
```

## 5. AsNoTracking

参照専用APIでは `AsNoTracking()` を使用する。

対象:

- Project一覧
- Flow一覧
- Flow読込
- Version一覧
- Template一覧
- Export用読込
- Setting読込

更新予定EntityではAsNoTrackingを使わない。

## 6. Projection

APIレスポンスに不要な列を取得しない。

一覧DTOでは以下のみ取得する。

Project一覧:

- ProjectId
- ProjectName
- UpdatedAt
- Role
- FlowCount

Flow一覧:

- FlowId
- FlowName
- UpdatedAt
- Revision
- NodeCount
- LinkCount

## 7. Include方針

無制限Includeは禁止する。

禁止:

```csharp
_context.Flows
    .Include(x => x.Nodes)
    .ThenInclude(x => x.Links)
```

推奨:

- 必要テーブルを個別に取得
- FlowIdで絞る
- DTOへ組み立てる

## 8. インデックス方針

主要検索条件にはインデックスを設定する。

対象:

- PROJECT.USER_ID
- FLOW.PROJECT_ID
- LANE.FLOW_ID
- STAGE.FLOW_ID
- FLOW_NODE.FLOW_ID
- FLOW_LINK.FLOW_ID
- FLOW_COMMENT.FLOW_ID
- FLOW_VERSION.FLOW_ID
- TEMPLATE.PROJECT_ID
- PROJECT_MEMBER.PROJECT_ID
- PROJECT_MEMBER.USER_ID

論理削除条件も考慮する。

## 9. 論理削除と性能

すべての検索では原則 `IS_DELETED = 0` を条件に含める。

EF CoreのGlobal Query Filterを採用する場合、性能と明示性を確認する。

初期方針:

- Repositoryで明示的にIS_DELETED条件を付与
- 共通拡張メソッド化を検討

## 10. Flow保存最適化

Flow保存は差分反映を基本とする。

入力DTOを以下に分類する。

- Added
- Updated
- Deleted
- Unchanged

全削除・全再作成は禁止する。

理由:

- GUIDが変わる
- Version比較が困難になる
- Undo/Redo履歴と不整合になる
- Link参照が壊れやすい
- DB負荷が大きい

## 11. SaveChanges方針

Flow一括保存では、Transaction内でSaveChangesを最小回数にする。

原則:

- 1回の保存処理でSaveChangesは1回
- 採番都合で必要な場合のみ複数回を許可
- RepositoryごとのSaveChangesは禁止

## 12. Bulk処理

初期実装ではEF Core標準で実装する。

将来的に性能問題が出た場合のみ以下を検討する。

- ExecuteUpdate
- ExecuteDelete
- BulkExtensions
- SQL Server TVP
- Raw SQL

ただし、実装複雑度が増すため初期から導入しない。

## 13. キャッシュ対象

キャッシュしてよいもの:

- NodeType一覧
- 標準テンプレート一覧
- アプリ設定
- Mermaid変換設定
- Export設定
- 権限定義
- 静的マスタ

キャッシュしないもの:

- Flow本体
- Node一覧
- Link一覧
- 編集ロック
- ProjectMember権限
- 最新Revision
- 自動保存データ

## 14. NodeTypeキャッシュ

NODE_TYPEは頻繁に参照されるためキャッシュ対象とする。

更新頻度が低く、整合性リスクが小さい。

キャッシュキー:

```text
node_types:all
```

無効化タイミング:

- NodeType追加
- NodeType更新
- NodeType削除

## 15. Templateキャッシュ

標準テンプレートはキャッシュ可能。

ユーザーテンプレートはProject依存のため慎重に扱う。

初期方針:

- 標準テンプレートのみキャッシュ
- ユーザーテンプレートはDB参照

## 16. MemoryCache

初期実装では `IMemoryCache` を利用する。

理由:

- 単一Webアプリ構成
- 実装が簡単
- 外部Redis不要

将来複数サーバ化する場合はRedis等を検討する。

## 17. キャッシュ有効期限

|対象|期限|
|---|---:|
|NodeType|30分|
|標準テンプレート|10分|
|設定|10分|
|権限定義|60分|

重要な更新時は期限を待たず明示削除する。

## 18. Version Snapshot性能

FlowVersionはSnapshot JSONを保持するため、サイズが大きくなる。

対策:

- Snapshot生成は必要時のみ
- 自動保存ごとにVersionを作らない
- 圧縮は将来検討
- Version一覧ではSnapshot本文を取得しない

## 19. Export性能

ExportではFlow構造を一括取得し、変換処理はメモリ上で行う。

PDF出力は重いため、将来的に非同期化を検討する。

初期実装では同期APIでよいが、10秒超過が多い場合は非同期Exportへ移行する。

## 20. 大量リンク対策

10000リンクではLink検索が重くなる可能性がある。

対策:

- FLOW_IDインデックス
- FROM_NODE_IDインデックス
- TO_NODE_IDインデックス
- DTO変換時にDictionaryを利用
- NodeごとにLINQ Whereを繰り返さない

禁止:

```csharp
foreach (var node in nodes)
{
    var links = allLinks.Where(x => x.FromNodeId == node.Id).ToList();
}
```

推奨:

```csharp
var linksByFrom = allLinks
    .GroupBy(x => x.FromNodeId)
    .ToDictionary(x => x.Key, x => x.ToList());
```

## 21. DTO組立性能

DTO組立ではDictionaryを活用する。

- LaneId -> LaneDto
- StageId -> StageDto
- NodeId -> NodeDto
- NodeTypeId -> NodeTypeDto

大量データで線形探索を繰り返してはならない。

## 22. Pagination

一覧APIはPagination必須。

対象:

- Project一覧
- Flow一覧
- Template一覧
- Version一覧
- User一覧
- 監査ログ一覧

Flow内部のNode/Linkはページングしない。キャンバス表示に必要なため一括取得する。

## 23. 検索条件

検索APIは以下を受け取る。

- page
- pageSize
- keyword
- sortKey
- sortDirection

pageSize上限を設定する。

初期上限:

```text
100
```

## 24. SQLite注意点

SQLiteでは書き込み同時実行に制約がある。

対策:

- Flow保存Transactionを長時間保持しない
- Export中にDB Transactionを保持しない
- ファイル処理をTransaction内で長時間行わない
- 不要な排他を避ける

## 25. SQL Server移行考慮

SQL Server移行を見据え、以下を避ける。

- SQLite固有SQL依存
- 型の曖昧な扱い
- GUID文字列比較の乱用
- 日時のローカル保存
- boolのDB依存表現への過度な依存

日時はUTC保存を基本とする。
