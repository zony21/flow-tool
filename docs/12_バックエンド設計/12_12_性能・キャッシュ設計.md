# 12_12_性能・キャッシュ設計

## 1. 目的

本書は、AI Flow Designer Backendにおける性能・キャッシュ設計を定義する。

本システムは、将来的に1000Node、10000Link規模のFlowを扱う可能性がある。初期DBはSQLiteだが、将来的にSQL Serverへ移行するため、DB非依存で性能劣化しにくい設計とする。

## 2. 性能目標

初期目標:

| 処理 | 目標 |
| --- | ---: |
| Project一覧取得 | 1秒以内 |
| Flow一覧取得 | 1秒以内 |
| Flow読込 1000Node / 10000Link | 3秒以内 |
| Flow保存 1000Node / 10000Link | 5秒以内 |
| Mermaid出力 | 3秒以内 |
| JSON出力 | 3秒以内 |
| PDF出力 | 10秒以内 |
| Version作成 | 5秒以内 |
| Version比較 | 5秒以内 |

## 3. 基本方針

- 読込処理はAsNoTrackingを基本とする。
- 一覧APIでは詳細データを取得しない。
- Flow詳細取得は必要な関連データのみ取得する。
- N+1を禁止する。
- 大量データは一括取得する。
- 更新は差分保存を基本とする。
- 不要なSaveChangesを行わない。
- キャッシュは整合性を壊さない範囲に限定する。
- SQLite前提の過剰最適化をしない。
- SQL Server移行を妨げない。

## 4. Flow読込最適化

Flow読込では以下をFlowIdで一括取得する。

- FLOW
- LANE
- STAGE
- FLOW_NODE
- FLOW_LINK
- FLOW_COMMENT
- NODE_TYPE

禁止例:

```text
foreach node:
  GetLinksByNodeId(node.Id)
```

推奨:

```text
Get all nodes by flowId
Get all links by flowId
Build DTO in memory
```

## 5. AsNoTracking

参照専用APIでは `AsNoTracking()` を使用する。

対象:

- Project一覧
- Flow一覧
- Flow読込
- Version一覧
- Template一覧
- Export用読込
- Setting読込

更新予定EntityではAsNoTrackingを使わない。

## 6. Projection

APIレスポンスに不要な列を取得しない。

Project一覧:

- ProjectId
- ProjectName
- UpdatedAt
- Role
- FlowCount

Flow一覧:

- FlowId
- FlowName
- UpdatedAt
- Revision
- NodeCount
- LinkCount

Flow詳細:

- Flow本体
- Lane一覧
- Stage一覧
- Node一覧
- Link一覧
- Comment一覧

## 7. Include方針

無制限Includeは禁止する。

禁止:

```csharp
_context.Flows
    .Include(x => x.Nodes)
    .ThenInclude(x => x.Links)
```

推奨:

- 必要テーブルを個別に取得する。
- FlowIdで絞る。
- DTOへ組み立てる。
- AsNoTrackingを利用する。

## 8. インデックス方針

主要検索条件にはIndexを設定する。

対象:

- PROJECT.USER_ID
- FLOW.PROJECT_ID
- LANE.FLOW_ID
- STAGE.FLOW_ID
- FLOW_NODE.FLOW_ID
- FLOW_LINK.FLOW_ID
- FLOW_COMMENT.FLOW_ID
- FLOW_VERSION.FLOW_ID
- TEMPLATE.PROJECT_ID
- PROJECT_MEMBER.PROJECT_ID
- PROJECT_MEMBER.USER_ID
- IMAGE_FILE.PROJECT_ID
- IMAGE_FILE.FLOW_ID

論理削除条件も考慮する。

## 9. 論理削除と性能

すべての検索では原則 `IS_DELETED = 0` を条件に含める。

EF CoreのGlobal Query Filterを採用する場合、性能と明示性を確認する。

初期方針:

- Repositoryで明示的にIS_DELETED条件を付与する。
- 共通拡張メソッド化を検討する。

## 10. Flow保存最適化

Flow保存は差分反映を基本とする。

入力DTOを以下に分類する。

- Added
- Updated
- Deleted
- Unchanged

全削除・全再作成は禁止する。

理由:

- Revision管理が難しくなる。
- Link参照が不安定になる。
- 大規模Flowで性能劣化する。
- 監査ログが分かりにくくなる。

## 11. SaveChanges方針

1 UseCaseにつきSaveChangesは原則1回にする。

Flow保存でNodeごとにSaveChangesしてはならない。

禁止:

```text
foreach node:
  Add node
  SaveChanges
```

推奨:

```text
AddRange nodes
UpdateRange links
SaveChanges once
```

## 12. キャッシュ方針

キャッシュ対象は整合性への影響が少ないものに限定する。

初期キャッシュ候補:

- NodeType定義
- Export設定
- System Setting
- Standard Template一覧

キャッシュしないもの:

- Flow詳細
- ProjectMember権限
- 編集ロック
- Flow Revision

## 13. Compiled Query方針

将来SQL Server移行後、頻繁に使うQueryはCompiled Queryを検討する。

候補:

- Flow一覧
- Project一覧
- ProjectMember権限確認
- FlowId単位Node一覧
- FlowId単位Link一覧

初期SQLite段階では必須ではない。

## 14. Export性能

ExportはBackendで実行する。

方針:

- Export入力Modelを一度構築する。
- Mermaid / JSON / AI DSLは文字列生成を軽量にする。
- PDFは重いため将来非同期Job化を検討する。
- Export処理中にDB Transactionを保持しない。

## 15. API Paging

一覧APIはPagingを前提とする。

対象:

- Project一覧
- Flow一覧
- Version一覧
- Template一覧
- 画像一覧

Flow詳細は構造単位で取得するためPagingしない。

## 16. ログと性能計測

重要APIではElapsedMsをログに出力する。

対象:

- Flow読込
- Flow保存
- Export
- Version作成
- Version復元
- 画像アップロード

## 17. テスト観点

- Flow読込でN+1が発生しない。
- Project一覧でFlow詳細を取得しない。
- 1000Node / 10000LinkのFlow読込が目標内に収まる。
- Flow保存でSaveChangesが1回に集約される。
- Export中にDB Transactionを保持しない。
- Cache対象にProjectMember権限が含まれない。

## 18. 完了条件

- Backend性能目標が定義されている。
- AsNoTracking、Projection、Index、差分保存方針が定義されている。
- Cache対象と非対象が定義されている。
- AIが本書を読んで性能劣化しにくいRepository / Serviceを実装できる。
