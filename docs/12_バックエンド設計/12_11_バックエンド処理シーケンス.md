# 12_11_バックエンド処理シーケンス

## 1. 本書の目的

本書は、AI Flow Designer の主要バックエンド処理シーケンスを定義する。

対象処理は以下である。

- GitHub OAuthログイン
- Project作成
- Flow作成
- Flow読込
- Flow一括保存
- 自動保存
- FlowVersion作成
- FlowVersion復元
- Export
- Template作成
- 画像アップロード
- 編集ロック取得・解除

## 2. 共通前提

すべての認証必須APIは以下を通る。

1. JWT検証
2. USER_ACCOUNT確認
3. 論理削除確認
4. Project権限確認
5. Service呼び出し
6. DTO返却

EntityをControllerから直接返却してはならない。

## 3. GitHub OAuthログイン

```text
User
 -> Frontend
 -> Backend AuthController
 -> GitHub OAuth
 -> Backend AuthService
 -> UserRepository
 -> JWT Service
 -> Frontend
```

詳細:

1. ユーザーがログインボタン押下
2. FrontendがBackendのログインURLへ遷移
3. BackendがGitHub認可URLへリダイレクト
4. GitHubで認証
5. GitHubがcallbackへcodeを返却
6. BackendがcodeをAccessTokenへ交換
7. GitHub User APIからユーザー情報取得
8. USER_ACCOUNTを作成または更新
9. JWT発行
10. FrontendへJWTを返却またはリダイレクト


6. Revision=1

8. FlowDto返却

## 6. Flow読込

```text
FlowController
 -> AuthorizationService
 -> FlowQueryService
 -> FlowRepository
 -> LaneRepository
 -> StageRepository
 -> NodeRepository
 -> LinkRepository
 -> CommentRepository
```

処理:

1. JWT認証
2. Project権限確認
3. Flow存在確認
4. Lane一覧取得
5. Stage一覧取得
6. Node一覧取得
7. Link一覧取得
8. Comment一覧取得
9. DTOへ組み立て
10. 返却

取得時はAsNoTrackingを基本とする。

## 7. Flow一括保存

Flow保存は本システムで最重要の処理である。

```text
FlowController
 -> AuthorizationService
 -> EditLockService
 -> FlowSaveService
 -> Validator
 -> Repositories
 -> DbContext Transaction
 -> VersionService(optional)
```

処理:

1. JWT認証
2. Project権限確認
3. Editor以上確認
4. 編集ロック確認
5. Revision確認
6. DTO入力検証
7. 構造整合性検証
8. Transaction開始
9. Lane差分反映
10. Stage差分反映
11. Node差分反映
12. Link差分反映
13. Comment差分反映
14. Flow Revision更新
15. 必要に応じてSnapshot保存
16. Commit
17. 保存結果DTO返却

## 8. 構造整合性検証

Flow保存前に以下を検証する。

- Nodeが存在しないLinkは禁止
- Link FROM/TOが同一でも許可するかはルールに従う
- ループは許可
- LaneIdが存在しないNodeは禁止
- StageIdが存在しないNodeは禁止
- NodeTypeが存在しないNodeは禁止
- 論理削除済Laneへの所属は禁止
- 論理削除済Stageへの所属は禁止
- 座標が数値であること
- サイズが正数であること

## 9. 自動保存

自動保存は30秒ごとに差分のみ送信する。

```text
Frontend AutoSave
 -> FlowController
 -> FlowSaveService
 -> Transaction
```

通常保存との差:

- isAutoSave=true
- 復元ポイントを保持
- ユーザー通知は控えめ
- 競合時は自動上書きしない
- Validationエラーは画面上に保持

## 10. FlowVersion作成

```text
VersionController
 -> AuthorizationService
 -> FlowVersionService
 -> FlowQueryService
 -> FlowVersionRepository
```

処理:

1. Editor以上確認
2. Flow全体構造取得
3. Snapshot JSON生成
4. VersionNo採番
5. FLOW_VERSION登録
6. Commit
7. VersionDto返却

SnapshotにはLane/Stage/Node/Link/Comment/設定を含める。

## 11. FlowVersion復元

```text
VersionController
 -> AuthorizationService
 -> EditLockService
 -> FlowVersionService
 -> FlowSaveService
```

処理:

1. Editor以上確認
2. 編集ロック確認
3. Version存在確認
4. Snapshot読込
5. 現在Flowの復元前Version作成
6. Snapshot内容でFlow上書き
7. Revision更新
8. Commit
9. 復元結果返却

復元前に現在状態の退避Versionを作る。

## 15. Template作成

## 18. 編集ロック取得

```text
EditorController
 -> AuthorizationService
 -> EditLockService
 -> EditLockRepository
```

処理:

1. Editor以上確認
2. 既存ロック取得
3. ロックなしなら新規作成
4. 自分のロックなら期限延長
5. 期限切れなら上書き
6. 他ユーザー有効ロックなら409
7. LockDto返却

## 19. 編集ロック解除

処理:

1. Editor以上確認
2. ロック取得
3. 自分のロックなら解除
4. ProjectAdmin以上なら管理解除可能
5. 解除ログ出力
6. 結果返却

## 20. 削除処理共通

論理削除を基本とする。

処理:

1. 対象存在確認
2. 権限確認
3. 参照関係確認
4. IS_DELETED=true
5. 監査カラム更新
6. Commit

## 21. Lane削除

Lane削除時は選択式。

- ノード移動
- 同時削除

ノード移動の場合:

1. 移動先LaneId確認
2. 対象NodeのLaneId更新
3. Lane論理削除

同時削除の場合:

1. Lane配下Node論理削除
2. Nodeに紐づくLink論理削除
3. Node紐付けComment処理
4. Lane論理削除

## 22. Stage削除

Stage削除時も選択式。

- ノード移動
- 同時削除

処理方針はLane削除に準ずる。

## 23. エラー処理共通

すべてのシーケンスで例外は共通Middlewareへ渡す。

Service層では必要に応じてAppExceptionを送出する。

Controllerで例外を握りつぶしてはならない。

## 24. ログ共通

主要シーケンスは開始・終了・失敗をログ出力する。

特に以下は必須。

- Flow保存
- Version作成
- Version復元
- Export
- 画像アップロード
- 編集ロック競合

## 25. 完了条件

- 主要処理シーケンスが実装者に伝わる
- Controller/Service/Repositoryの責務が明確
- Transaction境界が明確
- 認可確認位置が明確
- 例外発生時の流れが明確
