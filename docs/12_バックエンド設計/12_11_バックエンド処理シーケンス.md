# 12_11_バックエンド処理シーケンス

## 1. 目的

本書は、AI Flow Designer Backendの主要処理シーケンスを定義する。

対象処理は、GitHub OAuth、Project作成、Flow作成、Flow読込、Flow一括保存、自動保存、Version Snapshot、Export、Template、画像アップロード、編集ロックである。

## 2. 共通前提

すべての認証必須APIは以下を通る。

1. JWT検証。
2. USER_ACCOUNT確認。
3. 論理削除確認。
4. Project権限確認。
5. Service呼び出し。
6. DTO返却。

EntityをControllerから直接返却してはならない。

## 3. GitHub OAuthログイン

```text
User
 -> Frontend
 -> AuthController
 -> GitHub OAuth
 -> AuthService
 -> UserRepository
 -> JwtTokenService
 -> Frontend
```

処理:

1. ユーザーがログインボタン押下。
2. FrontendがBackendのログインURLへ遷移。
3. BackendがGitHub認可URLへリダイレクト。
4. GitHubで認証。
5. GitHubがcallbackへcodeを返却。
6. BackendがcodeをAccessTokenへ交換。
7. GitHub User APIからユーザー情報取得。
8. USER_ACCOUNTを作成または更新。
9. JWT発行。
10. Frontendへリダイレクト。

異常系:

- state不一致: 401。
- GitHub API失敗: 500。
- USER_ACCOUNT作成失敗: 500。

## 4. Project作成

```text
ProjectController
 -> ProjectAuthorizationService
 -> ProjectService
 -> ProjectRepository
 -> ProjectMemberRepository
 -> UnitOfWork / Transaction
```

処理:

1. JWT認証。
2. CreateProjectRequest検証。
3. Project名重複確認。
4. Transaction開始。
5. PROJECT作成。
6. PROJECT_MEMBER作成。
7. 作成者をProjectAdminに設定。
8. SaveChanges。
9. Commit。
10. ProjectDto返却。

異常系:

- Project名重複: 409。
- Validation失敗: 400。
- DB保存失敗: 500。

## 5. Flow作成

```text
FlowController
 -> ProjectAuthorizationService
 -> FlowService
 -> FlowRepository
 -> LaneRepository
 -> StageRepository
 -> UnitOfWork / Transaction
```

処理:

1. Project権限確認。
2. Editor以上確認。
3. Flow名検証。
4. Transaction開始。
5. FLOW作成。
6. 初期Lane / Stageが指定されていれば作成。
7. Revision=1。
8. SaveChanges。
9. Commit。
10. FlowDto返却。

## 6. Flow読込

```text
FlowController
 -> ProjectAuthorizationService
 -> FlowQueryService
 -> FlowRepository
 -> LaneRepository
 -> StageRepository
 -> FlowNodeRepository
 -> FlowLinkRepository
 -> FlowCommentRepository
```

処理:

1. JWT認証。
2. Project権限確認。
3. Flow存在確認。
4. Lane一覧取得。
5. Stage一覧取得。
6. Node一覧取得。
7. Link一覧取得。
8. Comment一覧取得。
9. DTOへ組み立て。
10. 返却。

取得時はAsNoTrackingを基本とする。
N+1は禁止する。

## 7. Flow一括保存

Flow保存は本システムで最重要の処理である。

```text
FlowController
 -> ProjectAuthorizationService
 -> EditorLockService
 -> FlowSaveService
 -> SaveFlowRequestValidator
 -> FlowStructureValidator
 -> Repositories
 -> UnitOfWork / Transaction
 -> FlowVersionService(optional)
```

処理:

1. JWT認証。
2. Project権限確認。
3. Editor以上確認。
4. 編集ロック確認。
5. Revision確認。
6. DTO入力検証。
7. 構造整合性検証。
8. Transaction開始。
9. Lane差分反映。
10. Stage差分反映。
11. Node差分反映。
12. Link差分反映。
13. Comment差分反映。
14. Flow Revision更新。
15. 必要に応じてSnapshot保存。
16. SaveChanges。
17. Commit。
18. 保存結果DTO返却。

異常系:

- Revision不一致: 409。
- 編集ロック不一致: 409。
- Link参照不正: 400。
- Transaction失敗: 500。

## 8. 構造整合性検証

Flow保存前に以下を検証する。

- Nodeが存在しないLinkは禁止。
- Link Source / Targetが同一でもLoop方針に従い許可する。
- ループは許可。
- LaneIdが存在しないNodeは禁止。
- StageIdが存在しないNodeは禁止。
- NodeTypeが存在しないNodeは禁止。
- 論理削除済Laneへの所属は禁止。
- decision Nodeの条件Link不足を検出する。
- image NodeのImageFile存在を確認する。

## 9. 自動保存

自動保存は通常保存と同じ整合性を守る。

違い:

- isAutoSave=true。
- Frontendへ保存状態を返す。
- 競合時は上書きしない。
- Snapshotは原則作成しない。

## 10. Version Snapshot作成

```text
VersionController
 -> ProjectAuthorizationService
 -> FlowVersionService
 -> FlowQueryService
 -> FlowVersionRepository
 -> UnitOfWork / Transaction
```

処理:

1. JWT認証。
2. Project権限確認。
3. Editor以上確認。
4. 保存済みFlow取得。
5. VersionNo採番。
6. SnapshotJson生成。
7. FLOW_VERSION追加。
8. SaveChanges。
9. Commit。
10. VersionDto返却。

未保存Frontend状態からSnapshotを作成しない。

## 11. Version復元

```text
VersionController
 -> ProjectAuthorizationService
 -> FlowVersionService
 -> FlowSaveService
 -> UnitOfWork / Transaction
```

処理:

1. JWT認証。
2. Project権限確認。
3. Editor以上確認。
4. 対象Version取得。
5. 現在Flowを復元前Snapshotとして保存。
6. SnapshotJsonを現在Flowへ反映。
7. Revision更新。
8. SaveChanges。
9. Commit。
10. 復元結果DTO返却。

失敗時は現在Flowを変更しない。

## 12. Export

```text
ExportController
 -> ProjectAuthorizationService
 -> ExportService
 -> FlowQueryService / FlowVersionService
 -> ExportInputBuilder
 -> FormatRenderer
```

処理:

1. JWT認証。
2. Project権限確認。
3. Export対象確認。
4. FlowまたはSnapshot取得。
5. ExportInputModel作成。
6. 形式別Renderer呼び出し。
7. 結果返却。

対象形式:

- Mermaid
- PDF
- JSON
- AI DSL

FrontendのCanvas DOMを入力にしない。

```text
TemplateController
```

処理:

1. JWT認証。

## 14. 画像アップロード

```text
ImageController
 -> ProjectAuthorizationService
 -> ImageFileService
 -> FileValidator
 -> FileStorageService
 -> ImageFileRepository
 -> UnitOfWork
```

処理:

1. JWT認証。
2. Project権限確認。
3. ファイル検証。
4. Hash計算。
5. StorageKey生成。
6. ファイル保存。
7. IMAGE_FILE登録。
8. SaveChanges。
9. Response返却。

## 15. 編集ロック取得・解除

```text
EditorController
 -> ProjectAuthorizationService
 -> EditorLockService
 -> EditorLockRepository
```

取得処理:

1. JWT認証。
2. Project権限確認。
3. Editor以上確認。
4. 既存ロック確認。
5. ロック期限切れなら再取得。
6. ロック作成または更新。
7. Response返却。

解除処理:

- 本人は解除可能。
- ProjectAdmin以上は強制解除可能。
- Viewerは不可。

## 16. 完了条件

- 主要Backend処理の正常系・異常系が定義されている。
- Flow保存、Snapshot、Export、Template、画像、編集ロックの流れが明確である。
- Controller / Service / Repository / UnitOfWorkの関係が明確である。
- AIが本書を読んで主要UseCaseを実装できる。
