# 12_10_ファイル・画像管理設計

## 1. 目的

本書は、AI Flow Designer Backendにおけるファイル・画像管理設計を定義する。

本システムでは、Node種別として画像Nodeを扱う。また、将来的にPDF、SVG、設計書ドラフト、添付資料、Export成果物などのファイル管理が発生する可能性がある。

DBにはファイル実体を保存せず、メタデータのみ保持する。ファイル実体はStorageへ保存する。

## 2. 基本方針

- DBにはメタデータのみ保存する。
- ファイル実体はStorageへ保存する。
- 物理ファイル名には元ファイル名を使わない。
- ファイル名はStorageKeyへ変換する。
- Hashを保持する。
- 同一ファイル検出にHashを利用できるようにする。
- アップロード前に拡張子、MIME、サイズを検証する。
- 画像はProjectまたはFlowに紐付ける。
- 物理パスをAPIレスポンスへ返さない。
- 削除は論理削除を基本とする。
- 物理削除は管理処理または将来バッチで行う。

## 3. 対象ファイル

初期対象:

- PNG
- JPEG
- WebP
- SVG

将来対象:

- PDF
- Mermaid出力ファイル
- JSON出力ファイル
- AI DSL出力ファイル
- 設計書ドラフト
- 添付資料

## 4. 保存方式

初期はローカルストレージを利用する。
将来、S3互換ストレージやクラウドストレージへ移行可能なInterfaceを用意する。

```text
StorageRoot/
  projects/
    {projectId}/
      images/
        {yyyy}/
          {MM}/
            {imageFileId}.{ext}
      exports/
        {flowId}/
          {exportId}.{ext}
```

物理ファイル名には元ファイル名を使わない。

## 5. StorageKey

StorageKeyは物理保存先を表す論理キーとする。

例:

```text
projects/{projectId}/images/2026/07/{imageFileId}.png
```

APIレスポンスでは必要に応じて画像取得APIのURLを返す。

```text
/api/images/{imageFileId}
```

## 6. IMAGE_FILE

画像用メタデータテーブルとして `IMAGE_FILE` を用意する。

保持項目:

- IMAGE_FILE_ID
- PROJECT_ID
- FLOW_ID
- ORIGINAL_FILE_NAME
- CONTENT_TYPE
- EXTENSION
- FILE_SIZE
- HASH_SHA256
- STORAGE_KEY
- WIDTH
- HEIGHT
- IS_SVG
- DESCRIPTION
- IS_DELETED
- SYSTEM_CREATE_USER_ID
- SYSTEM_CREATE_DATETIME
- SYSTEM_UPDATE_USER_ID
- SYSTEM_UPDATE_DATETIME

## 7. Interface設計

Application層にInterfaceを定義する。

```csharp
public interface IFileStorageService
{
    Task<StoredFileResult> SaveAsync(Stream stream, string storageKey, CancellationToken ct);
    Task<Stream> OpenReadAsync(string storageKey, CancellationToken ct);
    Task<bool> ExistsAsync(string storageKey, CancellationToken ct);
    Task DeleteAsync(string storageKey, CancellationToken ct);
}
```

Infrastructure層に初期実装を配置する。

```text
LocalFileStorageService
```

## 8. アップロードAPI

```text
POST /api/projects/{projectId}/images
```

Content-Type:

```text
multipart/form-data
```

Request:

- file
- flowId 任意
- description 任意

Response:

```json
{
  "imageFileId": "guid",
  "contentType": "image/png",
  "fileSize": 123456,
  "width": 800,
  "height": 600,
  "url": "/api/images/guid"
}
```

## 9. アップロード処理フロー

```text
ImageController
  -> ProjectAuthorizationService
  -> ImageFileService
  -> FileValidator
  -> IFileStorageService
  -> ImageFileRepository
  -> UnitOfWork
```

処理:

1. JWT認証。
2. Project権限確認。
3. multipart file取得。
4. 拡張子検証。
5. Content-Type検証。
6. サイズ検証。
7. Hash計算。
8. 画像サイズ取得。
9. StorageKey生成。
10. ファイル保存。
11. IMAGE_FILE登録。
12. SaveChanges。
13. Response返却。

## 10. 画像取得API

```text
GET /api/images/{imageFileId}
```

処理:

1. JWT認証。
2. IMAGE_FILE取得。
3. Project権限確認。
4. 論理削除確認。
5. STORAGE_KEYから物理ファイル取得。
6. Content-Typeを設定して返却。

物理パスは返さない。

## 11. 削除API

```text
DELETE /api/images/{imageFileId}
```

処理:

1. JWT認証。
2. Project権限確認。
3. IMAGE_FILE取得。
4. 使用中Node確認。
5. 論理削除。
6. SaveChanges。

使用中画像を削除する場合は、Backendで拒否するか、Node側参照解除を要求する。
初期方針では使用中画像削除を拒否する。

## 12. FileValidator

検証項目:

- fileが存在する。
- file sizeが上限以下。
- 拡張子が許可対象。
- MIMEが許可対象。
- 拡張子とMIMEが矛盾しない。
- SVGの場合は危険なscriptを含まない。
- 画像として読み取れる。

## 13. SVG方針

SVGは便利だが、script混入リスクがある。

初期方針:

- SVGを許可する場合はSanitize必須。
- script、foreignObject、外部参照を禁止する。
- 危険なSVGはアップロード拒否する。

Sanitize実装が間に合わない場合、初期リリースではSVGを無効化できる設定を持つ。

## 14. Export成果物との関係

PDF、JSON、AI DSLなどのExport結果をStorageへ保存する場合、IMAGE_FILEとは別にEXPORT_FILEテーブルを検討する。

初期ではExport結果はレスポンスとして返却し、永続保存は将来機能とする。

## 15. Transactionと補償処理

ファイル保存とDB保存は完全な同一Transactionにできない。

初期方針:

1. ファイル検証。
2. 一時StorageKeyへ保存。
3. DB Transaction開始。
4. IMAGE_FILE登録。
5. SaveChanges。
6. Commit。
7. 正式StorageKeyへ移動。

正式移動失敗時は補償処理でメタデータを無効化する。

## 16. セキュリティ注意点

- 物理パスを返さない。
- 元ファイル名を物理保存名に使わない。
- Content-Typeを信用しすぎない。
- 拡張子だけで判定しない。
- JWTや権限確認なしでファイルを返さない。
- ファイル内容をログに出さない。

## 17. テスト観点

- PNGをアップロードできる。
- サイズ超過ファイルが拒否される。
- 不正MIMEが拒否される。
- SVGの危険要素が拒否される。
- 権限なしユーザーが画像を取得できない。
- 物理パスがAPIレスポンスに含まれない。
- 使用中画像削除が拒否される。

## 18. 完了条件

- StorageとDBメタデータの責務が分離されている。
- IMAGE_FILEの項目が定義されている。
- Upload / Get / Delete APIが定義されている。
- SVGとセキュリティ方針が定義されている。
- AIが本書を読んで画像管理Backendを実装できる。
