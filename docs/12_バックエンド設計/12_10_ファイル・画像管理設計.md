# 12_10_ファイル・画像管理設計

## 1. 本書の目的

本書は、AI Flow Designer のバックエンドにおけるファイル・画像管理設計を定義する。

本システムでは、Node種別として画像ノードを扱う。また、将来的に設計書ドラフト、PDF、SVG、添付資料などのファイル管理が発生する可能性がある。

SQLiteにはファイル実体を保存せず、メタデータのみ保持する。ファイル実体はストレージへ保存する。

## 2. 基本方針

- SQLiteにはメタデータのみ保存
- ファイル実体はストレージ保存
- ファイル名は保存キーへ変換する
- Hashを保持する
- 同一ファイル検出にHashを利用可能
- SVG対応予定
- アップロード前に拡張子・MIME・サイズを検証
- 画像はProjectまたはFlowに紐付ける
- 物理パスをAPIレスポンスへ返さない
- ファイル削除は論理削除を基本とする
- 物理削除はバッチまたは管理処理で行う

## 3. 対象ファイル

初期対象:

- PNG
- JPEG
- WebP
- SVG

将来対象:

- PDF
- Mermaid出力ファイル
- JSON出力ファイル
- AI DSL出力ファイル
- 設計書ドラフト
- 添付資料

## 4. 保存方式

保存方式は以下とする。

```text
StorageRoot/
  projects/
    {projectId}/
      images/
        {yyyy}/
          {MM}/
            {imageId}.{ext}
      exports/
        {flowId}/
          {exportId}.{ext}
```

物理ファイル名には元ファイル名を使わない。

## 5. メタデータテーブル

画像用テーブル例:

```text
IMAGE_FILE
```

保持項目:

- IMAGE_FILE_ID
- PROJECT_ID
- FLOW_ID
- ORIGINAL_FILE_NAME
- CONTENT_TYPE
- EXTENSION
- FILE_SIZE
- HASH_SHA256
- STORAGE_KEY
- WIDTH
- HEIGHT
- IS_SVG
- IS_DELETED
- SYSTEM_CREATE_USER_ID
- SYSTEM_CREATE_DATETIME
- SYSTEM_UPDATE_USER_ID
- SYSTEM_UPDATE_DATETIME

## 6. STORAGE_KEY

STORAGE_KEYは物理保存先を表す論理キーとする。

例:

```text
projects/{projectId}/images/2026/07/{imageFileId}.png
```

APIレスポンスでは必要に応じて画像取得APIのURLを返す。

```text
/api/images/{imageFileId}
```

## 7. アップロードAPI

```text
POST /api/projects/{projectId}/images
```

Content-Type:

```text
multipart/form-data
```

Request:

- file
- flowId 任意
- description 任意

Response:

```json
{
  "imageFileId": "guid",
  "contentType": "image/png",
  "fileSize": 123456,
  "width": 800,
  "height": 600,
  "url": "/api/images/guid"
}
```

## 8. 画像取得API

```text
GET /api/images/{imageFileId}
```

処理:

1. JWT認証
2. IMAGE_FILE取得
3. Project権限確認
4. STORAGE_KEYから物理ファイル取得
5. Content-Typeを設定して返却

## 9. 削除API

```text
DELETE /api/images/{imageFileId}
```

処理:

1. JWT認証
2. Project権限確認
3. 使用中Node有無確認
4. IMAGE_FILEを論理削除
5. 物理削除は即時しない

使用中の場合は削除不可または参照解除確認を行う。

## 10. 検証ルール

拡張子:

- .png
- .jpg
- .jpeg
- .webp
- .svg

MIME:

- image/png
- image/jpeg
- image/webp
- image/svg+xml

サイズ:

- 初期上限 10MB
- Project設定で将来変更可能

## 11. SVG検証

SVGはスクリプト混入リスクがあるため、初期実装では以下を禁止する。

- scriptタグ
- 外部参照
- onload属性
- onclick属性
- foreignObject
- javascript: URL

SVG対応が難しい場合、初期リリースではSVGアップロードを禁止し、設計のみ残す。

## 12. Hash管理

アップロード時にSHA-256を計算する。

用途:

- 同一ファイル検出
- 改ざん検知
- 将来の重複排除
- ログ追跡

同一Hashでも別Projectの場合は別メタデータとして登録する。

## 13. Transaction方針

ファイル保存とDB保存は完全な分散Transactionにはしない。

推奨手順:

1. 一時ディレクトリへ保存
2. Hash計算
3. 画像情報取得
4. DB Transaction開始
5. IMAGE_FILE登録
6. 正式ディレクトリへ移動
7. DB Commit

失敗時:

- DB失敗時は一時ファイル削除
- ファイル移動失敗時はRollback
- Commit後の削除失敗はWarningログ

## 14. 画像ノードとの関係

画像Nodeは `FLOW_NODE.EXTENSION_JSON` または専用項目で imageFileId を保持する。

例:

```json
{
  "imageFileId": "guid",
  "fitMode": "contain",
  "opacity": 1.0
}
```

画像Node削除時は画像ファイル自体は削除しない。

理由:

- 他Nodeで使用している可能性がある
- Undo/Redoで復元される可能性がある
- Version復元で参照される可能性がある

## 15. Versionとの関係

FlowVersion Snapshotには imageFileId を含める。

ただし、画像実体はSnapshotへ含めない。

Version復元時に画像が論理削除済でも、参照表示できるようにするかは別途方針を持つ。

初期方針:

- Versionから参照される画像は物理削除しない
- 論理削除済でもVersion参照時は取得可能

## 16. Exportとの関係

PDF出力時に画像Nodeが含まれる場合、画像を読み込んでPDFへ埋め込む。

Mermaid出力では画像Nodeは以下のいずれかで表現する。

- 画像ノード名のみ
- `[[Image: 表示名]]`
- コメントとして画像参照を出力

初期は表示名のみとする。

## 17. FileStorageService

実装するService:

```text
IFileStorageService
IImageFileService
```

責務分担:

`IFileStorageService`

- 物理保存
- 物理取得
- 物理削除
- Hash計算
- パス生成

`IImageFileService`

- 認可
- メタデータ登録
- 画像検証
- IMAGE_FILE管理
- Node参照確認

## 19. 例外

|状況|例外|
|---|---|
|サイズ超過|ValidationAppException|
|MIME不正|ValidationAppException|
|SVG不正|ValidationAppException|
|保存失敗|FileStorageException|
|画像なし|NotFoundAppException|
|権限なし|ForbiddenAppException|
|使用中削除|BusinessRuleException|

## 20. ログ

アップロード成功:

- ImageFileId
- ProjectId
- FlowId
- ContentType
- FileSize
- Hash
- ElapsedMs

アップロード失敗:

- ProjectId
- FlowId
- ErrorCode
- ExceptionType
- ElapsedMs

元ファイル名はログに出さない。

## 21. セキュリティ

禁止事項:

- 物理パスをAPIへ返す
- 元ファイル名を保存ファイル名に使う
- SVGを無検証で配信する
- MIMEだけを信用する
- 拡張子だけを信用する
- Authorizationなしで画像取得可能にする
- privateなProject画像をpublic URL化する

## 22. テスト観点

単体テスト:

- 拡張子検証
- MIME検証
- サイズ検証
- StorageKey生成
- Hash計算
- SVG危険要素検出

結合テスト:

- 画像アップロード成功
- 権限なしアップロード失敗
- 画像取得成功
- 他Project画像取得不可
- 削除済画像取得制御
- 使用中画像削除制御

E2E:

- 画像Node追加
- 画像表示
- Flow保存
- Flow再読込
- Version復元
- PDF出力

## 23. 完了条件

- 画像アップロードAPIが実装されている
- SQLiteにはメタデータのみ保存されている
- 物理パスがAPIに漏れない
- Project権限が適用されている
- 画像Nodeから参照できる
- Version/Exportで破綻しない
