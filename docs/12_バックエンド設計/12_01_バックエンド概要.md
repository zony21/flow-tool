# 12_01_バックエンド全体方針

## 1. 目的

本書は、AI Flow DesignerのBackend全体方針を定義する。

Backendは、SSOTであるProject / Flow / Lane / Stage / Node / Link / Commentを永続化し、検証し、Version Snapshot、Export、認証・認可、画像管理、Template適用、将来のAIレビュー連携を支える中核である。

Frontendは編集体験を提供するが、保存の正当性を最終的に保証するのはBackendである。

## 2. 採用技術

| 区分 | 技術 |
| --- | --- |
| Framework | ASP.NET Core .NET 8 |
| ORM | Entity Framework Core |
| 初期DB | SQLite |
| 将来DB | SQL Server |
| Auth | GitHub OAuth / JWT |
| Logging | ILogger / 構造化ログ |

## 3. 設計思想

- Controllerは薄くする。
- 業務処理はService層へ集約する。
- Repositoryは永続化処理のみ担当する。
- DomainはSSOTの意味と整合性を表す。
- EntityをAPIへ直接返さない。
- SQLite固有処理を上位層へ漏らさない。
- Exportは保存済みFlowまたはSnapshotから生成する。
- Version SnapshotはUndo / Redoとは別責務とする。

## 4. レイヤ構成

```text
Backend/
```

## 5. Backendの主要責務

- GitHub OAuth認証
- JWT発行・検証
- Project権限確認
- Flow構造保存
- Flow構造読込
- Lane / Stage / Node / Link / Comment整合性検証
- Version Snapshot作成・復元
- Template作成・適用
- Mermaid / PDF / JSON / AI DSL Export
- 画像アップロード・取得
- 例外処理
- ログ出力
- テスト容易性の確保

## 6. Controller方針

ControllerはHTTP境界を担当する。

担当すること:

- Request受信
- 認証属性付与
- DTO受け渡し
- Service呼び出し
- Response返却

担当しないこと:

- 業務判断
- Transaction開始
- Repository直接呼び出し
- Entity直接返却
- DB操作

## 7. Service方針

ServiceはApplication層の中心である。

担当すること:

- 業務ルール判断
- 認可確認
- DTO Validation呼び出し
- Domain整合性検証
- Transaction境界制御
- Repository呼び出し
- 例外送出
- DTO変換

特にFlowSaveServiceは、Lane / Stage / Node / Link / Commentを一括保存する最重要Serviceである。

## 8. Repository方針

RepositoryはEF Coreを利用した永続化処理を担当する。

担当すること:

- Entity取得
- Entity追加
- Entity更新準備
- Entity論理削除準備
- Exists確認
- Count取得

担当しないこと:

- 認可判定
- 業務ルール判定
- Transaction開始
- SaveChanges
- API DTO作成

SaveChangesはServiceまたはUnitOfWorkで集約する。

## 9. Domain方針

DomainはSSOTの意味と構造ルールを表す。

主な対象:

- Project
- Flow
- Lane
- Stage
- FlowNode
- FlowLink
- FlowComment
- FlowVersionSnapshot
- Template
- ImageFile

Domainでは以下を守る。

- Linkは存在するNodeのみ参照する。
- Nodeは存在するLaneへ所属する。
- Stage参照が必要なNodeは存在するStageを参照する。
- Loopは許可する。
- NodeTypeごとの必須Propertyを検証する。

## 10. Transaction方針

TransactionはService層で制御する。

対象:

- Flow一括保存
- Project作成とProjectMember作成
- Version Snapshot作成
- Version復元
- Template適用
- 画像メタデータ登録

Repository内でTransactionを開始しない。
Repository内でSaveChangesを呼ばない。

## 11. 認証・認可方針

認証はGitHub OAuthを採用する。
API認証にはJWTを利用する。

認可はProject単位を基本とする。

- Admin
- ProjectAdmin
- Editor
- Viewer

Project権限はJWTへ大量に含めず、DBで確認する。

## 12. Export方針

ExportはBackendで実行する。

対象:

- Mermaid
- PDF
- JSON
- AI DSL
- 将来のAPI仕様、DB更新一覧、PLC一覧、通信一覧、設計書ドラフト

Export入力は保存済みFlowまたはVersion Snapshotとする。
FrontendのCanvas DOMを入力にしない。

## 13. Version Snapshot方針

Version Snapshotは保存済みFlowの正式履歴である。

Undo / Redoとは異なる。
SnapshotはBackendで作成し、Exportや将来復元の入力にできる形式で保持する。

## 14. 例外・ログ方針

例外は共通Middlewareで標準レスポンスへ変換する。

ログは構造化ログを前提とし、traceId、userId、projectId、flowId、action、elapsedMsを可能な限り含める。

DB接続文字列、OAuth token、JWT、個人情報はログへ出力しない。

## 15. SQLite / SQL Server方針

初期はSQLiteを利用する。
将来SQL Serverへ移行する。

そのため、以下を守る。

- DB固有SQLをApplication層へ書かない。
- EF Coreで表現できる範囲を優先する。
- DB固有最適化はInfrastructureへ閉じ込める。
- IDはGUIDを基本とし、DB自動採番依存を避ける。

## 16. テスト観点

- ControllerがServiceのみ呼び出す。
- ServiceがTransaction境界を持つ。
- RepositoryがSaveChangesしない。
- Flow保存でLane / Stage / Node / Link / Commentが一括保存される。
- 失敗時にRollbackされる。
- 認可失敗時に403が返る。
- Validation失敗時に標準エラー形式が返る。
- Exportが保存済みFlowまたはSnapshotから生成される。

## 17. 完了条件

- Backend全体の責務境界が明確である。
- Controller / Service / Repository / Domain / Infrastructureの役割が定義されている。
- Transaction、認証、Export、Snapshot、例外、ログ方針が明確である。
- AIが本書を読んでBackend実装の全体構成を判断できる。
