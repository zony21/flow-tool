# 12_08_ログ設計

## 1. 目的

本書は、AI Flow Designer Backendにおけるログ設計を定義する。

本システムは、Flow保存、Version作成、Export、画像管理、編集ロック、権限確認など、後から原因調査が必要になる処理が多い。ログは単なる障害記録ではなく、運用・監査・性能調査・AI実装時のデバッグ容易化を支える基盤として扱う。

## 2. 基本方針

- ASP.NET Core標準のILoggerを利用する。
- 構造化ログを前提とする。
- 文字列連結ログは禁止する。
- traceIdを必ず出力する。
- userIdを可能な限り出力する。
- projectId / flowIdを可能な限り出力する。
- OAuth token / JWT / 接続文字列 / 個人情報は出力しない。
- 通常操作はInformation。
- 業務上の想定内エラーはWarning。
- システム障害はError。
- 処理時間を記録する。
- 将来の外部ログ基盤連携を妨げない。

## 3. ログレベル

| Level | 用途 |
| --- | --- |
| Trace | 原則使用しない。局所調査用 |
| Debug | 開発環境の詳細確認 |
| Information | 正常な重要処理 |
| Warning | 想定内だが注意が必要な事象 |
| Error | 処理失敗・復旧可能な障害 |
| Critical | システム継続困難な障害 |

## 4. Information対象

- ユーザーログイン成功
- Project作成
- Flow作成
- Flow保存成功
- FlowVersion作成
- FlowVersion復元
- Template作成
- Export成功
- 画像アップロード成功
- 編集ロック取得
- 編集ロック解除
- 自動保存成功

## 5. Warning対象

- Validation失敗
- 権限不足
- 編集ロック競合
- Revision競合
- NotFound
- Template名重複
- 削除済データへのアクセス
- 想定範囲内のExport不可
- 画像サイズ超過
- SVG検証失敗

## 6. Error対象

- DB保存失敗
- Transaction失敗
- Export処理失敗
- PDF生成失敗
- 画像保存失敗
- ファイル移動失敗
- 予期しない例外
- 外部連携失敗
- GitHub OAuth検証失敗

## 7. Critical対象

Criticalは以下に限定する。

- DB接続不能が継続
- ストレージルートが消失
- 設定ファイル読込不能
- 起動時Migration失敗
- アプリケーション継続不能

## 8. 必須ログ項目

| 項目 | 説明 |
| --- | --- |
| TraceId | HTTPリクエスト単位の追跡ID |
| UserId | ログインユーザーID |
| ProjectId | 対象Project ID |
| FlowId | 対象Flow ID |
| Action | 処理名 |
| Result | Success / Failed |
| ElapsedMs | 処理時間 |
| ErrorCode | エラー時のコード |
| RequestPath | APIパス |
| HttpMethod | HTTPメソッド |

## 9. ログスコープ

ASP.NET CoreのBeginScopeを利用し、リクエスト単位で共通情報を付与する。

```csharp
using (_logger.BeginScope(new Dictionary<string, object>
{
    ["TraceId"] = traceId,
    ["UserId"] = currentUser.UserId,
    ["ProjectId"] = projectId,
    ["FlowId"] = flowId
}))
{
    await _flowService.SaveAsync(request, currentUser, ct);
}
```

## 10. APIリクエストログ

すべてのAPIリクエストについて、最低限以下を記録する。

- Method
- Path
- StatusCode
- ElapsedMs
- UserId
- TraceId

正常終了はInformation、4xxはWarning、5xxはErrorとする。

## 11. Request Bodyログ

Request Bodyは原則ログ出力しない。

理由:

- Flow保存時に大量データとなる。
- 構造化フローには業務情報が含まれる可能性がある。
- 画像や拡張JSONを含む可能性がある。
- 個人情報混入リスクがある。

必要な場合は件数やIDのみ記録する。

例:

```text
NodeCount=120
LinkCount=300
CommentCount=40
```

## 12. Flow保存ログ

Flow保存成功時は以下を記録する。

- FlowId
- ProjectId
- RevisionBefore
- RevisionAfter
- LaneCount
- StageCount
- NodeCount
- LinkCount
- CommentCount
- IsAutoSave
- ElapsedMs

Flow保存失敗時は以下を記録する。

- FlowId
- ProjectId
- RequestedRevision
- CurrentRevision
- ErrorCode
- ExceptionType
- TransactionState
- ElapsedMs

## 13. Versionログ

FlowVersion作成時:

- VersionId
- FlowId
- ProjectId
- VersionNo
- SnapshotSize
- CreatedBy

FlowVersion復元時:

- VersionId
- FlowId
- ProjectId
- RestoreFromVersionNo
- NewRevision

## 14. Exportログ

Export開始時:

- ExportType
- FlowId
- SnapshotId
- ProjectId
- UserId

Export成功時:

- ExportType
- OutputSize
- ElapsedMs

Export失敗時:

- ExportType
- ErrorCode
- ExceptionType
- traceId

## 15. 監査ログ

監査ログとして扱う操作:

- Project作成・更新・削除
- ProjectMember追加・変更・削除
- Flow削除
- Version復元
- Template削除
- 管理者による編集ロック解除

監査ログは通常ログと同じ基盤でもよいが、Actionと対象IDを必ず記録する。

## 16. 出力禁止情報

ログへ出力してはいけない情報:

- GitHub OAuth Access Token
- JWT
- Password相当情報
- DB接続文字列
- Secret Key
- 個人情報本文
- アップロードファイル実体
- Flow全体JSON

## 17. テスト観点

- 主要APIでtraceIdがログに出る。
- Flow保存成功ログが出る。
- Flow保存失敗ログが出る。
- Export失敗時にExportTypeが出る。
- OAuth tokenやJWTがログに出ない。
- 4xxはWarning、5xxはErrorになる。

## 18. 完了条件

- ログレベルと出力対象が定義されている。
- 必須ログ項目が定義されている。
- Flow保存、Version、Export、監査ログが定義されている。
- 出力禁止情報が明確である。
- AIが本書を読んでログ実装に着手できる。
