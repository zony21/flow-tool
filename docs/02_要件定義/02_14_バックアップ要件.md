# 02_14_バックアップ要件

## 1. 本書の目的

本書は、AI Flow Designer のバックアップ要件を定義する。

本システムでは、Project、Flow、Version、Template、画像メタデータ、設定情報が重要データであるため、障害時に復旧できるようバックアップ方針を定義する。

## 2. 基本方針

- DBと画像ストレージをバックアップ対象とする
- Version SnapshotもDBバックアップ対象に含める
- 画像ファイル実体と画像メタデータの整合性を保つ
- 初期開発では手動バックアップでもよい
- 将来的に定期自動バックアップを導入する

## 3. バックアップ対象

対象:

- SQLite DBファイル
- 画像ファイルストレージ
- アプリケーション設定ファイル
- 環境設定サンプル
- Migration定義

対象外:

- node_modules
- bin / obj
- 一時ファイル
- ログの全量
- JWT
- GitHubアクセストークン

## 4. DBバックアップ

SQLiteの場合:

- DBファイルをバックアップする
- バックアップ中の書き込み競合に注意する
- 可能であればSQLite Backup API相当を利用する

SQL Server移行後:

- SQL Server標準バックアップを利用する
- フルバックアップと差分バックアップを検討する

## 5. 画像ストレージバックアップ

画像ファイルはDBではなくストレージに保存する。

要件:

- DBの画像メタデータと同じタイミングでバックアップする
- ファイル名、Hash、サイズを保持する
- 欠落ファイルを検出できるようにする

## 6. バックアップ頻度

初期方針:

- 開発環境: 任意手動
- 個人利用本番: 1日1回
- チーム利用本番: 1日1回以上

将来的にProject重要度に応じて頻度を変更可能にする。

## 7. 世代管理

初期目安:

- 日次バックアップ: 7世代
- 週次バックアップ: 4世代
- 月次バックアップ: 3世代

個人開発段階では必須ではないが、運用設計として定義する。

## 8. バックアップ保存先

候補:

- ローカル別ディレクトリ
- 外部ストレージ
- クラウドストレージ

本番ではアプリケーション稼働ディスクと別領域へ保存することを推奨する。

## 9. バックアップ整合性

バックアップ後に以下を確認する。

- DBファイルが存在する
- 画像ストレージが存在する
- 画像メタデータ件数とファイル件数に大きな差異がない
- バックアップファイルサイズが0でない

## 10. 復元手順

復元手順:

1. アプリケーションを停止する
2. 現在のDBと画像ストレージを退避する
3. バックアップDBを配置する
4. バックアップ画像ストレージを配置する
5. アプリケーションを起動する
6. 起動確認を行う
7. 主要Project / Flowを確認する

## 11. バックアップ検証

バックアップは取得するだけでなく、復元可能性を定期確認する。

確認内容:

- 別環境でDBを開ける
- Flow一覧が表示できる
- Flowエディタが開ける
- 画像Nodeが表示できる
- Version復元ができる

## 12. 機密情報

バックアップには機密情報が含まれる可能性がある。

要件:

- バックアップ保存先のアクセス制御を行う
- 公開リポジトリへ配置しない
- .env実ファイルは慎重に扱う
- Token類を含めない設計にする

## 13. 障害時利用

障害発生時は、障害復旧要件に従いバックアップから復元する。

復元前には、可能であれば現在状態も退避する。

## 14. テスト観点

- DBバックアップを取得できる
- 画像ストレージをバックアップできる
- バックアップから復元できる
- 復元後にFlowを開ける
- 画像Nodeが欠落していない

## 15. 完了条件

- バックアップ対象が定義されている
- DBと画像の整合性方針がある
- 復元手順が定義されている
- SQL Server移行後の方針へ拡張可能である
