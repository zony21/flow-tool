# 02_14_バックアップ要件

## 1. 本書の目的

本書は、AI Flow Designer のバックアップ要件を定義する。

本システムでは、Project、Flow、Version、Template、画像メタデータ、設定情報が重要データであるため、障害時に復旧できるようバックアップ方針を定義する。

## 2. 基本方針

- DBと画像ストレージをバックアップ対象とする
- Version SnapshotもDBバックアップ対象に含める
- 画像ファイル実体と画像メタデータの整合性を保つ
- 初期開発では手動バックアップでもよい
- 将来的に定期自動バックアップを導入する
- バックアップは復元確認まで含めて完了とみなす

## 3. バックアップ対象

対象:

- SQLite DBファイル
- 画像ファイルストレージ
- アプリケーション設定ファイル
- 環境設定サンプル
- EF Core Migration定義
- Template定義
- Flow Version Snapshot

対象外:

- node_modules
- bin / obj
- 一時ファイル
- フロントエンドビルド成果物
- JWT
- GitHubアクセストークン
- OSキャッシュ

## 4. SQLiteバックアップ要件

SQLite初期運用では、DBファイルをバックアップする。

要件:

- バックアップ中の書き込み競合に注意する
- アプリ停止中のバックアップを基本とする
- 稼働中バックアップが必要な場合はSQLite Backup API相当の方式を利用する
- DBファイルサイズが0でないことを確認する
- バックアップ後に整合性確認を行う

## 5. SQL Server移行後の要件

将来的にSQL Serverへ移行した場合は、SQL Server標準バックアップを利用する。

対象:

- フルバックアップ
- 差分バックアップ
- トランザクションログバックアップ

SQL Server移行後は、RPO / RTO要件に応じてバックアップ頻度を見直す。

## 6. 画像ストレージバックアップ

画像ファイルはDBではなくストレージに保存する。

要件:

- DBの画像メタデータと同じ時点のファイルを保存する
- ファイル名、Hash、サイズを保持する
- 画像メタデータが存在するのにファイルがない状態を検出できること
- ファイルだけ存在しDBにメタデータがない状態を検出できること

## 7. 設定ファイルバックアップ

対象:

- appsettingsの環境別設定
- ストレージパス設定
- CORS設定
- JWT設定のサンプル

注意:

- 本番secret値をGitHubへ置かない
- `.env` 実ファイルは公開リポジトリへ含めない
- secretは安全な保管先で管理する

## 8. バックアップ頻度

初期方針:

|環境|頻度|
|---|---|
|開発環境|任意手動|
|個人利用本番|1日1回|
|チーム利用本番|1日1回以上|
|リリース前|必ず取得|
|Migration前|必ず取得|

## 9. 世代管理

初期目安:

- 日次バックアップ: 7世代
- 週次バックアップ: 4世代
- 月次バックアップ: 3世代

個人開発段階では厳密な自動削除は必須ではないが、運用設計として定義する。

## 10. バックアップ保存先

候補:

- ローカル別ディレクトリ
- 外部ストレージ
- クラウドストレージ
- NAS

本番ではアプリケーション稼働ディスクと別領域へ保存することを推奨する。

## 11. バックアップ命名規則

命名例:

```text
flow-tool-db_yyyyMMddHHmmss.sqlite
flow-tool-images_yyyyMMddHHmmss.zip
flow-tool-config_yyyyMMddHHmmss.zip
```

バックアップ時点をファイル名で確認できること。

## 12. バックアップ整合性確認

バックアップ後に以下を確認する。

- DBファイルが存在する
- DBファイルサイズが0でない
- 画像ストレージが存在する
- 画像メタデータ件数とファイル件数に大きな差異がない
- バックアップファイルが読み取り可能である
- バックアップ取得日時が記録されている

## 13. 復元手順

復元手順:

1. アプリケーションを停止する
2. 現在のDBと画像ストレージを退避する
3. バックアップDBを配置する
4. バックアップ画像ストレージを配置する
5. 設定ファイルを確認する
6. アプリケーションを起動する
7. 起動確認を行う
8. Project一覧を確認する
9. Flowエディタを開く
10. 画像Nodeを確認する

## 14. 復元確認項目

復元後に確認する内容:

- ログインできる
- Project一覧が表示できる
- Flow一覧が表示できる
- Flowエディタが開ける
- Node / Link / Commentが表示される
- 画像Nodeが表示される
- Version一覧が表示される
- Exportが実行できる

## 15. バックアップ検証

バックアップは取得するだけでなく、定期的に復元可能性を確認する。

検証内容:

- 別環境でDBを開ける
- Migration状態に矛盾がない
- Flow一覧が表示できる
- Flowエディタが開ける
- 画像Nodeが表示できる
- Version復元ができる

## 16. セキュリティ要件

バックアップには設計情報が含まれるため、アクセス制御を行う。

要件:

- バックアップ保存先を公開しない
- GitHub公開リポジトリへ配置しない
- secretやTokenを含めない
- 保存先のアクセス権限を限定する
- 不要になったバックアップは安全に削除する

## 17. 障害時利用

障害発生時は、障害復旧要件に従いバックアップから復元する。

復元前には、可能であれば現在状態も退避する。

## 18. テスト観点

- DBバックアップを取得できる
- 画像ストレージをバックアップできる
- 設定ファイルをバックアップできる
- バックアップから復元できる
- 復元後にFlowを開ける
- 画像Nodeが欠落していない
- バックアップにTokenが含まれていない

## 19. 完了条件

- バックアップ対象が定義されている
- DBと画像の整合性方針がある
- 復元手順が定義されている
- 復元確認手順が定義されている
- SQL Server移行後の方針へ拡張可能である
