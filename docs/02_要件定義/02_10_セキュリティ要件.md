# 02_10_セキュリティ要件

## 1. 本書の目的

本書は、AI Flow Designer のセキュリティ要件を定義する。

本システムはProject、Flow、設計情報、画像、テンプレートなどを扱うため、認証、認可、入力検証、ファイル管理、監査、通信保護を必須要件とする。

## 2. 基本方針

- 認証はGitHub OAuthを利用する
- APIはJWT認証を必須とする
- Project配下データはProject権限で保護する
- フロントエンド表示制御だけに依存しない
- Backend側で必ず認可チェックを行う
- 画像ファイルも認可対象とする
- 入力値はBackendでValidationする
- EntityをAPIで直接返却しない

## 3. 通信保護

本番環境ではHTTPSを必須とする。

要件:

- HTTP通信は禁止する
- Cookieを利用する場合はSecure属性を付与する
- 外部URLへのredirectは許可しない
- OAuth callback URLは許可リストで管理する

## 4. 認証保護

- GitHub OAuthで本人確認を行う
- 認証成功後にJWTを発行する
- JWTには必要最小限のClaimsのみ含める
- GitHubアクセストークンを不要に長期保存しない
- トークンをログへ出力しない

## 5. 認可保護

Backend APIでは以下を必ず確認する。

- ユーザーが認証済みであること
- Projectに参加していること
- 要求操作に必要なRoleを持つこと
- Flow編集時は編集ロックを取得していること

## 6. XSS対策

要件:

- ユーザー入力文字列をHTMLとして直接描画しない
- MarkdownやHTML出力を導入する場合はサニタイズする
- Comment本文、Node説明、Flow説明を安全に表示する
- Vueのテンプレートエスケープを前提とする

## 7. CSRF対策

JWTをAuthorization Headerで送信する場合、CSRFリスクは低い。

Cookie認証を採用する場合はCSRF Tokenを必須とする。

初期方針:

- Authorization: Bearer によるJWT送信

## 8. SQL Injection対策

要件:

- EF Coreのパラメータ化クエリを利用する
- SQL文字列結合を禁止する
- 動的ソート項目は許可リスト方式にする
- 検索条件は型付きRequest DTOで受け取る

## 9. ファイルアップロード対策

対象:

- 画像Node用画像

要件:

- 許可拡張子を制限する
- MIME Typeを検証する
- ファイルサイズ上限を設ける
- 保存ファイル名に元ファイル名を直接使わない
- 保存先パスを外部入力から組み立てない
- 画像メタデータのみDB保存する

## 10. 画像アクセス制御

画像参照時も認可チェックを行う。

要件:

- 画像が所属するProjectを確認する
- Project権限がないユーザーへ返却しない
- 直接URL推測で参照できないようにする
- 将来的に署名付きURLを検討する

## 11. セキュリティヘッダー

本番環境では以下を設定する。

- Content-Security-Policy
- X-Content-Type-Options
- X-Frame-Options または frame-ancestors
- Referrer-Policy
- Permissions-Policy

## 12. ログ出力制限

ログへ出力してはいけない情報:

- JWT
- GitHubアクセストークン
- パスワード相当情報
- 個人情報の詳細
- 画像実体パス

ログへ出力してよい情報:

- traceId
- userId
- projectId
- flowId
- errorCode
- API path

## 13. 入力Validation

Backendで必須Validationを行う。

対象:

- Project名
- Flow名
- Node表示名
- Link条件
- Template名
- Comment本文
- 画像ファイル

## 14. エラー表示

セキュリティ上、内部詳細をユーザーへ表示しない。

例:

- StackTraceを表示しない
- SQL詳細を表示しない
- ファイルパスを表示しない

ユーザーには共通メッセージとtraceIdを表示する。

## 15. 脆弱性対策

運用要件:

- npm packageを定期更新する
- NuGet packageを定期更新する
- GitHub Dependabotを将来導入する
- 依存ライブラリの脆弱性を確認する

## 16. 将来要件

将来検討:

- 多要素認証
- SSO
- IP制限
- 監査ログ閲覧画面
- 署名付きURL
- 権限テンプレート

## 17. テスト観点

- 未認証APIが401になる
- 権限なしAPIが403になる
- Viewerが編集APIを実行できない
- 他ProjectのFlowを参照できない
- 不正ファイルをアップロードできない
- XSS文字列がHTML実行されない

## 18. 完了条件

- 認証、認可、入力検証、ファイル保護が定義されている
- Backend側認可チェックが必須化されている
- 画像アクセスも保護対象になっている
- セキュリティテストへ展開できる
