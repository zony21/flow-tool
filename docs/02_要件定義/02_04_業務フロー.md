# 02_04_業務フロー

## 1. 本書の目的

本書は、AI Flow Designer の主要業務フローを定義する。

業務フローは画面設計、API設計、DB設計、フロントエンド実装、バックエンド実装、E2Eテストの共通基準とする。

## 2. 業務フロー共通方針

すべての業務フローで以下を共通前提とする。

- 認証が必要な操作ではJWTを検証する
- Project配下操作ではProject権限を検証する
- 編集操作では編集ロックを確認する
- 更新操作ではRevisionを確認する
- 更新成功後は監査カラムを更新する
- エラー時は共通エラー形式で返却する

## 3. システム全体フロー

```text
ログイン
  ↓
Project一覧
  ↓
Project選択
  ↓
Flow選択
  ↓
Flowエディタ
  ├─ 編集
  ├─ AutoSave
  ├─ Version作成
  ├─ Template適用
  └─ Export
```

## 4. ログインフロー

開始条件:

- 未認証ユーザーがログイン画面を開く

正常フロー:

1. ユーザーがGitHubログインを押下する
2. BackendがGitHub OAuth認証URLへリダイレクトする
3. GitHub認証を行う
4. Callbackで認証コードを受け取る
5. Backendがユーザー情報を取得する
6. USER_ACCOUNTを作成または更新する
7. JWTを発行する
8. Project一覧へ遷移する

異常フロー:

- GitHub認証失敗
- Callbackパラメータ不正
- ユーザー情報取得失敗
- JWT発行失敗

終了条件:

- 認証済み状態になる

## 5. Project作成フロー

開始条件:

- 認証済みユーザーがProject一覧で作成操作を行う

正常フロー:

1. Project名と説明を入力する
2. 入力Validationを行う
3. PROJECTを作成する
4. 作成者をProjectAdminとして登録する
5. 必要に応じて初期Flowを作成する
6. Project詳細へ遷移する

異常フロー:

- Project名未入力
- Project名重複
- DB保存失敗

終了条件:

- 新しいProjectが作成される

## 6. Flow作成フロー

開始条件:

- Editor以上がProject内でFlow作成を行う

正常フロー:

1. Flow名を入力する
2. 初期テンプレートを選択する
3. FLOWを作成する
4. 初期Lane / Stageを作成する
5. テンプレート選択時は構造を適用する
6. Flowエディタへ遷移する

異常フロー:

- 権限不足
- Flow名未入力
- テンプレート不正
- DB保存失敗

## 7. Flow編集フロー

開始条件:

- Editor以上がFlowエディタを開く

正常フロー:

1. Flow情報を取得する
2. Project権限を確認する
3. 編集ロック取得を試みる
4. Flow構造をStoreへ展開する
5. ユーザーが編集操作を行う
6. Commandを生成する
7. Storeを更新する
8. dirty状態にする
9. AutoSaveまたは手動保存を行う

異常フロー:

- 編集ロック取得失敗
- Revision競合
- 保存失敗
- Validationエラー

## 8. Node編集フロー

正常フロー:

1. Node追加または選択を行う
2. プロパティを編集する
3. AddNodeCommandまたはUpdateNodeCommandを生成する
4. Node情報をStoreへ反映する
5. 関連Validationを実行する
6. dirtyNodesへ追加する

業務ルール:

- NodeはLane / Stageへ所属する
- Node削除時は関連Linkを削除対象にする
- 画像Nodeは画像メタデータを参照する

## 9. Link編集フロー

正常フロー:

1. FROM NodeのHandleをドラッグする
2. TO Nodeへ接続する
3. 接続可能性を検証する
4. AddLinkCommandを生成する
5. Link情報をStoreへ反映する
6. dirtyLinksへ追加する

業務ルール:

- LinkはfromNodeId / toNodeIdを必須とする
- ループ接続は許可する
- Linkは条件、通信種別、データ名を保持できる

## 10. Template適用フロー

開始条件:

- Editor以上がテンプレート適用を行う
- 編集ロック取得済み

正常フロー:

1. Templateを選択する
2. Template JSONを取得する
3. JSON構造をValidationする
4. ID再採番Mapを作成する
5. Lane / Stage / Node / Link / Commentを変換する
6. ApplyTemplateCommandを生成する
7. Storeへ反映する
8. AutoSave対象にする

異常フロー:

- 権限不足
- 編集ロックなし
- JSON不正
- 参照不整合

## 11. AutoSaveフロー

開始条件:

- Flowがdirty状態である
- AutoSaveが有効である
- 最終保存から30秒経過している

正常フロー:

1. dirty対象を収集する
2. Revisionを付与する
3. AutoSave APIを呼び出す
4. 保存成功時にdirtyを解除する
5. Revisionを更新する

異常フロー:

- 通信失敗
- Revision競合
- Validationエラー

## 12. Version作成フロー

正常フロー:

1. 現在Flowの保存状態を確認する
2. 未保存変更がある場合は保存確認を行う
3. Versionコメントを入力する
4. Flow Snapshotを作成する
5. FLOW_VERSIONへ保存する
6. Version一覧を更新する

## 13. Version復元フロー

正常フロー:

1. 復元対象Versionを選択する
2. 編集権限とロックを確認する
3. 現在状態を復元前Versionとして保存する
4. 対象SnapshotをFlowへ反映する
5. Undo / Redo履歴をクリアする
6. Flowエディタへ戻る

## 14. Exportフロー

正常フロー:

1. Export形式を選択する
2. 出力オプションを設定する
3. Flow構造Validationを行う
4. Export APIを呼び出す
5. プレビューを表示する
6. ファイルをダウンロードする

## 15. 編集ロックフロー

取得:

1. Flowエディタを開く
2. 編集権限を確認する
3. Lock取得APIを呼び出す
4. 成功時は編集可能にする
5. 失敗時は読み取り専用にする

解除:

1. 画面を閉じる、または遷移する
2. 未保存変更を確認する
3. Lock解除APIを呼び出す

## 16. 共通エラー条件

- 未認証
- 権限不足
- 対象なし
- Revision競合
- 編集ロック競合
- Validationエラー
- 通信失敗
- DB保存失敗

## 17. 完了条件

- 主要業務フローが定義されている
- 正常系と異常系が分離されている
- 画面、API、DB、テストへ展開できる
