# 02_08_非機能要件

## 1. 本書の目的

本書は、AI Flow Designer の非機能要件を定義する。

非機能要件は、機能が動作するだけでなく、安全に、保守しやすく、拡張しやすく、AIが理解しやすい品質で提供するための基準である。

## 2. 非機能要件分類

- 可用性
- 保守性
- 拡張性
- 性能
- セキュリティ
- 運用性
- 監査性
- 復旧性
- AI理解性
- ユーザビリティ
- アクセシビリティ

## 3. 可用性

初期開発では単一Webアプリ構成とする。

要件:

- 通常利用時に主要画面へアクセスできること
- APIエラー時にユーザーへ分かりやすく通知すること
- 保存失敗時に編集内容を即時破棄しないこと
- AutoSave失敗時も手動保存できること

## 4. 保守性

要件:

- フロントエンドはVue3 / TypeScriptで型安全に実装する
- バックエンドはASP.NET Core .NET8で層分離する
- EntityをAPIへ直接返却しない
- DTOを利用する
- 章ごとの設計書と実装フォルダを対応させる

## 5. 拡張性

拡張対象:

- NodeType
- LinkType
- Export形式
- Template形式
- AI DSL
- DB移行
- 画像形式

要件:

- 初期仕様に固定せず追加可能な設計にする
- extensionJsonによりNode拡張を許可する
- Exportは形式追加できる構造にする

## 6. AI理解性

本システム固有の重要要件である。

要件:

- Node / Link / Lane / Stageが意味を持つこと
- 表示名だけに依存しないこと
- LinkにdataName、communicationType、conditionを保持すること
- Laneに担当情報を保持すること
- Stageに工程情報を保持すること
- JSON / AI DSLで意味情報を欠落させないこと

## 7. セキュリティ

要件:

- GitHub OAuthを利用する
- APIはJWT認証を必須とする
- Project権限をAPI側で必ず検証する
- Viewerに編集APIを許可しない
- 画像ファイルも権限チェック対象とする
- XSS対策を行う

## 8. 運用性

要件:

- エラー時にtraceIdを返却する
- APIエラーを共通形式にする
- ログ出力方針を統一する
- DB Migrationを管理する
- 設定値は環境ごとに切り替えられること

## 9. 監査性

要件:

- 主要テーブルに監査カラムを持つ
- 作成者、作成日時、更新者、更新日時を保持する
- Projectメンバー変更は監査対象とする
- Version復元は監査対象とする

## 10. 復旧性

要件:

- Flow Versionにより過去状態へ復元できる
- AutoSave失敗時も手動保存できる
- Version復元前に現在状態を保存する
- Undo / Redoで編集ミスを戻せる

## 11. ユーザビリティ

要件:

- FlowエディタはExcelライクな操作感を目指す
- ショートカットを提供する
- Drag & DropでNode追加できる
- エラーは対象箇所が分かるように表示する
- Viewerでは編集操作を非表示または非活性にする

## 12. アクセシビリティ

初期要件:

- キーボード操作に対応する
- 主要ボタンにラベルを付与する
- 色だけで状態を判断させない
- Validationメッセージをテキスト表示する

## 13. 国際化

初期UIは日本語を優先する。

将来的に英語対応できるよう、画面文言を定数化する。

## 14. データ整合性

要件:

- GUIDを主キーとする
- 論理削除を基本とする
- Revisionで競合を検出する
- Flow保存はTransactionで行う
- Linkが存在しないNodeを参照しないよう検証する

## 15. ログ要件

ログ対象:

- 認証失敗
- APIエラー
- 保存失敗
- Version復元
- 権限エラー
- 編集ロック競合

機密情報はログ出力しない。

## 16. 完了条件

- 非機能品質が分類されている
- AI理解性が明示要件として定義されている
- セキュリティ、運用、復旧、監査の基準がある
- 実装・テストへ展開できる
