# 02_11_認証要件

## 1. 本書の目的

本書は、AI Flow Designer の認証要件を定義する。

本システムでは、ユーザー認証にGitHub OAuthを利用し、Backendが認証済みユーザーに対してJWTを発行する。

## 2. 基本方針

- GitHub OAuthでログインする
- 独自パスワード認証は実装しない
- 認証成功後、USER_ACCOUNTを作成または更新する
- APIアクセスはJWTで認証する
- 未認証ユーザーは保護画面へアクセスできない

## 3. 認証対象

認証対象:

- Project一覧
- Project詳細
- Flow一覧
- Flowエディタ
- Template管理
- Version管理
- Export
- 設定
- メンバー管理

認証不要:

- ログイン画面
- GitHub OAuth Callback
- 静的アセット

## 4. GitHub OAuthフロー

正常フロー:

1. ユーザーがログイン画面でGitHubログインを押下する
2. BackendがGitHub OAuth認可URLを生成する
3. GitHub認可画面へリダイレクトする
4. ユーザーが認可する
5. GitHubからCallbackへcodeが返る
6. Backendがcodeをアクセストークンへ交換する
7. BackendがGitHubユーザー情報を取得する
8. USER_ACCOUNTを登録または更新する
9. JWTを発行する
10. Frontendへ認証結果を返す

## 5. Callback要件

Callbackでは以下を検証する。

- stateが一致すること
- codeが存在すること
- errorがないこと
- redirect先が内部URLであること

不正な場合はログイン画面へ戻す。

## 6. USER_ACCOUNT登録要件

GitHub認証成功時、GitHubユーザー情報からUSER_ACCOUNTを作成または更新する。

保持候補:

- userId
- githubId
- githubLogin
- displayName
- avatarUrl
- email
- lastLoginAt

## 7. JWT要件

JWTに含めるClaim:

- userId
- githubId
- githubLogin
- issuedAt
- expiresAt

JWTに含めない情報:

- GitHubアクセストークン
- Project権限一覧の全量
- 機密情報

## 8. JWT有効期限

初期方針:

- Access Token: 1時間
- Refresh Token: 将来対応

初期実装では、期限切れ時に再ログインを要求してよい。

## 9. 自動ログイン

初期実装では長期自動ログインを必須としない。

将来、Refresh Token方式を導入する場合は以下を検討する。

- Refresh TokenのDB保存
- Token Rotation
- 失効管理
- 端末別ログアウト

## 10. ログアウト

ログアウト時:

- FrontendのJWTを破棄する
- authStoreを初期化する
- ログイン画面へ遷移する

サーバー側Token失効管理はRefresh Token導入時に実装する。

## 11. 認証状態確認

Frontendは初期表示時に以下APIで認証状態を確認する。

```text
GET /api/auth/me
```

認証成功:

- currentUserをStoreへ保存する

認証失敗:

- ログイン画面へ遷移する

## 12. 認証エラー

|状況|HTTP|表示|
|---|---:|---|
|JWTなし|401|ログインしてください|
|JWT期限切れ|401|セッションが切れました|
|JWT不正|401|ログインしてください|
|GitHub OAuth失敗|401|認証に失敗しました|

## 13. Redirect要件

未認証で保護画面へアクセスした場合、ログイン後に元URLへ戻す。

ただし、外部URLへのredirectは禁止する。

## 14. API認証要件

保護APIはAuthorization Headerを必須とする。

```text
Authorization: Bearer {token}
```

JWT検証失敗時は401を返す。

## 15. Frontend要件

- authStoreで認証状態を管理する
- Router Guardで保護画面を制御する
- 401受信時はログイン画面へ遷移する
- Tokenをログへ出力しない

## 16. Backend要件

- JWT検証Middlewareを設定する
- 認証済みユーザーIDをContextから取得できる
- Controllerで現在ユーザーを参照できる
- 認証例外は共通エラー形式で返す

## 17. テスト観点

- GitHubログイン開始URLへ遷移できる
- CallbackでUSER_ACCOUNTが作成される
- JWTでAPIアクセスできる
- JWTなしで401になる
- 期限切れJWTで401になる
- 不正redirectが拒否される
- ログアウトで認証状態が消える

## 18. 完了条件

- GitHub OAuthログインフローが定義されている
- JWT発行と検証要件が定義されている
- Frontend / Backend双方の認証責務が明確である
