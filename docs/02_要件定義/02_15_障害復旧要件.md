# 02_15_障害復旧要件

## 1. 本書の目的

本書は、AI Flow Designer の障害復旧要件を定義する。

障害復旧要件は、DB障害、ストレージ障害、保存失敗、Version復元失敗、認証障害などが発生した場合に、データ損失と業務停止を最小限にするための基準である。

## 2. 基本方針

- 編集中データを可能な限り失わない
- AutoSave失敗時も手動保存を試せるようにする
- Versionにより過去状態へ戻せるようにする
- バックアップから復元できるようにする
- 障害時はユーザーへ分かりやすく通知する

## 3. 障害レベル

|レベル|概要|例|
|---|---|---|
|Level 1|軽微|一時的なAPI失敗、AutoSave失敗|
|Level 2|中程度|Flow保存失敗、Export失敗|
|Level 3|重大|DB破損、画像ストレージ欠落|
|Level 4|致命的|アプリ起動不可、全DB喪失|

## 4. RTO / RPO

初期目標:

|指標|目標|
|---|---:|
|RTO|24時間以内|
|RPO|24時間以内|

チーム利用や本番運用を強化する場合は、RTO / RPOを短縮する。

## 5. AutoSave失敗時

AutoSave失敗時の要件:

- 編集内容を即時破棄しない
- 失敗をToastまたはStatusBarに表示する
- dirty状態を維持する
- 次回AutoSaveで再試行する
- 手動保存を許可する

## 6. 手動保存失敗時

手動保存失敗時:

- エラーメッセージを表示する
- dirty状態を維持する
- ユーザーに再試行を促す
- Revision競合の場合は再読込または差分確認を促す

## 7. Revision競合時

Revision競合が発生した場合:

- 自動上書きしない
- ユーザーへ競合を通知する
- 再読込するか確認する
- 将来的に差分比較を提供する

## 8. DB障害時

DB障害時の対応:

1. アプリケーションを停止する
2. 現在DBを退避する
3. 直近バックアップを確認する
4. バックアップDBを復元する
5. 起動確認を行う
6. 主要Project / Flowを確認する

## 9. 画像ストレージ障害時

画像欠落時:

- 画像Nodeに欠落表示を行う
- Flow自体は開けるようにする
- 欠落画像一覧を確認できるようにする
- バックアップから画像を復元する

## 10. Version復元失敗時

Version復元失敗時:

- TransactionをRollbackする
- 現在Flowを破壊しない
- エラーを表示する
- traceIdを表示する

## 11. Export失敗時

Export失敗時:

- Flowデータは変更しない
- エラーを表示する
- 再実行可能にする
- PDFなど重い処理ではProgressを解除する

## 12. 認証障害時

GitHub OAuth障害時:

- ログイン不可を表示する
- 既存ログインユーザーのJWTが有効なら利用継続可能とする
- 新規ログインは復旧後に再試行する

## 13. 復旧後確認

復旧後に確認する内容:

- ログインできる
- Project一覧が表示できる
- Flowエディタが開ける
- 保存できる
- Version一覧が表示できる
- 画像Nodeが表示できる
- Exportできる

## 14. 通知要件

障害時はユーザーへ以下を通知する。

- 保存失敗
- AutoSave失敗
- 認証切れ
- 権限エラー
- 復旧不能なシステムエラー

内部詳細は表示せず、traceIdを表示する。

## 15. テスト観点

- AutoSave失敗時にdirtyが残る
- 手動保存失敗時に再試行できる
- Version復元失敗時にRollbackされる
- 画像欠落時もFlowを開ける
- バックアップから復元後に主要機能が利用できる

## 16. 完了条件

- 障害レベルが定義されている
- AutoSave / 保存 / Version / 画像障害時の挙動が定義されている
- バックアップ復元後の確認項目が定義されている
- データ損失を最小化する方針が明確である
