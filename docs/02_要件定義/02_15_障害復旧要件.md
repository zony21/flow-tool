# 02_15_障害復旧要件

## 1. 本書の目的

本書は、AI Flow Designer の障害復旧要件を定義する。

障害復旧要件は、DB障害、画像ストレージ障害、API障害、AutoSave失敗、Version復元失敗、編集ロック異常などが発生した場合に、データ損失と業務停止を最小化するための基準である。

## 2. 基本方針

- 障害時にユーザーの編集内容を可能な限り保護する
- 障害発生時は共通エラー形式で通知する
- 復旧前に現在状態を可能な限り退避する
- DBと画像ストレージの整合性を重視する
- Version機能とBackup機能を復旧手段として利用する
- 編集ロック異常時に管理者解除できるようにする

## 3. 障害レベル

|レベル|概要|例|
|---|---|---|
|L1|一時的な画面・通信エラー|API一時失敗、AutoSave失敗|
|L2|一部機能停止|Export失敗、画像表示失敗|
|L3|Project / Flow単位障害|Flow保存不可、Version復元不可|
|L4|システム全体障害|DB破損、アプリ起動不可|
|L5|データ損失障害|DB消失、画像ストレージ消失|

## 4. RTO / RPO

初期目標:

|対象|RTO|RPO|
|---|---:|---:|
|一時的なAPI障害|30分以内|直近保存時点|
|Flow保存障害|1時間以内|直近AutoSaveまたは手動保存|
|DB障害|当日中|直近バックアップ|
|画像ストレージ障害|当日中|直近バックアップ|
|全体障害|1営業日以内|直近バックアップ|

## 5. 障害検知

検知方法:

- APIエラーログ
- Frontend Toast / ErrorPanel
- Backend例外ログ
- DB接続失敗ログ
- 画像ファイル参照失敗ログ
- AutoSave失敗回数
- Export失敗ログ

将来的に監視サービス連携を検討する。

## 6. DB障害時の復旧

DB障害例:

- DBファイル破損
- Migration失敗
- データ不整合
- DB接続不可

復旧手順:

1. アプリケーションを停止する
2. 現在DBを退避する
3. バックアップDBを配置する
4. Migration状態を確認する
5. アプリケーションを起動する
6. Project / Flow表示を確認する
7. 監査ログに復旧作業を記録する

## 7. 画像ストレージ障害時の復旧

障害例:

- 画像ファイル欠落
- 画像ディレクトリ破損
- ファイルアクセス権限不正
- DBメタデータと実体不一致

復旧手順:

1. 画像ストレージを退避する
2. バックアップ画像ストレージを配置する
3. DB画像メタデータと突合する
4. 欠落ファイルを検出する
5. Flowエディタで画像Nodeを確認する

## 8. AutoSave障害時の対応

AutoSave失敗時:

- ユーザーへ警告表示する
- 編集内容はStore上に保持する
- 次回AutoSaveで再試行する
- 手動保存を促す

AutoSave失敗が連続する場合:

- ステータスバーで警告表示
- 保存API再試行
- 必要に応じて画面遷移時に警告

## 9. Flow保存障害時の対応

保存失敗時:

- dirty状態を維持する
- 保存失敗Toastを表示する
- エラー詳細をErrorPanelへ表示する
- 再保存できるようにする

Revision競合時:

- 競合メッセージを表示する
- 最新状態再取得を案内する
- 上書き保存は初期実装では禁止する

## 10. Version復元失敗時の対応

Version復元では、復元前に現在状態をVersion保存する。

復元失敗時:

- 復元処理をRollbackする
- 現在Flowを破損させない
- エラーを表示する
- 必要に応じて復元前Versionから戻せるようにする

## 11. 編集ロック異常時の対応

障害例:

- ロックが解除されない
- ロック保持者がブラウザを閉じた
- 通信断でロック更新できない

対応:

- ロックにはタイムアウトを設ける
- タイムアウト後は再取得可能とする
- ProjectAdminは管理者解除できる
- 解除操作は監査対象とする

## 12. 通信障害時の対応

通信障害時:

- API失敗をToast表示する
- 編集内容を即時破棄しない
- 再試行可能なUIにする
- ログイン切れの場合は再ログインへ誘導する

## 13. Export障害時の対応

Export失敗時:

- Flow構造は変更しない
- Exportエラーを表示する
- 再実行できるようにする
- PDF生成など長時間処理ではProgress表示を行う

## 14. 復旧時の優先順位

優先順位:

1. DB復旧
2. 画像ストレージ復旧
3. 認証復旧
4. Flow閲覧復旧
5. Flow編集復旧
6. Export復旧
7. Template / Version補助機能復旧

## 15. 復旧後確認

復旧後に確認する。

- ログインできる
- Project一覧が表示される
- Flow一覧が表示される
- Flowエディタが開く
- Node / Linkが表示される
- 画像Nodeが表示される
- 保存できる
- AutoSaveできる
- Version一覧が表示される
- Exportできる

## 16. 障害通知

初期実装では画面上のToast / ErrorPanelで通知する。

将来的には以下を検討する。

- 管理者メール通知
- Slack通知
- GitHub Issue自動作成
- 監視サービス連携

## 17. 運用記録

障害復旧作業では以下を記録する。

- 発生日時
- 復旧日時
- 障害レベル
- 原因
- 影響範囲
- 復旧手順
- 復旧担当
- 再発防止策

## 18. テスト観点

- AutoSave失敗時にdirtyが残る
- 保存失敗時に再保存できる
- Version復元失敗時にFlowが破損しない
- 編集ロックタイムアウト後に再取得できる
- バックアップからDB復元できる
- 画像欠落を検出できる
- 通信失敗時に編集内容が即時消えない

## 19. 完了条件

- 障害レベルが定義されている
- RTO / RPOの初期目標がある
- DB、画像、AutoSave、Version、編集ロックの復旧方針がある
- 復旧後確認手順が定義されている
- テストへ展開できる
