# 11_14_パフォーマンス設計

## 1. 目的

本書は、AI Flow DesignerのFrontendパフォーマンス設計を定義する。

AI Flow Designerは、小規模なフローだけでなく、将来的に数百〜数千Node規模のFlowを扱う可能性がある。
Canvas、Property Panel、Store、Adapter、Validation、Export UIが重くなると、設計ツールとして利用できない。

そのため、初期段階から再描画、状態更新、データ変換、API通信の負荷を抑える設計にする。

## 2. 基本方針

- Canvas再描画を最小化する。
- Store更新の粒度を適切にする。
- Vue Flow表示モデルの再生成を抑制する。
- 大きなFlowでは遅延処理・分割処理を行う。
- ExportはBackend実行とし、Frontendで重い生成処理をしない。
- 初期実装から将来の大規模Flow対応を妨げない構成にする。

## 3. 想定規模

| 段階 | Node数 | Link数 | 方針 |
| --- | --- | --- | --- |
| 初期 | 50〜200 | 50〜300 | 通常描画で対応 |
| 中規模 | 200〜1000 | 300〜1500 | 再描画抑制が必要 |
| 大規模 | 1000以上 | 1500以上 | Lazy / Virtual / 分割対応を検討 |

## 4. Store更新最適化

### 4.1 方針

- Flow全体を毎回Deep Copyしない。
- Node移動など頻発操作は対象Nodeのみ更新する。
- Drag中は一時状態をeditorStoreに持ち、Drag終了時にflowStoreへ確定する。
- 保存Request作成は保存時に行う。

### 4.2 悪い例

```text
Nodeを1px動かすたびにFlow全体を再生成する。
```

### 4.3 良い例

```text
Drag中はCanvas内部状態で動かし、dragStopでNode positionを更新する。
```

## 5. Adapter最適化

### 5.1 対象

- flowToVueFlowAdapter
- vueFlowToCommandAdapter
- flowToSaveRequestAdapter
- apiResponseToFlowAdapter

### 5.2 方針

- Flow全体変換は初期表示、保存成功後、Import後などに限定する。
- Node単体変更では対象Nodeの表示Modelだけ更新する。
- Link単体変更では対象Edgeだけ更新する。
- ComputedやMemoizationを活用する。

## 6. Vue Flow描画最適化

### 6.1 方針

- Node Componentを小さく保つ。
- Node内に重い処理を書かない。
- Node表示ComponentでAPI通信しない。
- unnecessary propsを渡しすぎない。
- selected / error / labelなど必要な情報だけdataに渡す。

### 6.2 Node Component

Node Componentは表示に集中する。
Property編集や保存処理は持たない。

### 6.3 Edge描画

初期はBezierを採用する。
大量Edgeで重い場合は、表示簡略化モードを検討する。

## 7. Property Panel最適化

Property Panelは選択中要素のみ表示する。

方針:

- 選択対象が変わった時だけFormを切り替える。
- Flow全体をpropsで渡さない。
- 必要なNode / Link / Commentのみ渡す。
- Validation Errorも対象要素分だけ抽出する。

## 8. Validation最適化

### 8.1 Frontend Validation

Frontendでは即時検証を行うが、全件検証を常時走らせない。

方針:

- 入力項目単位の検証は即時。
- Flow全体検証は保存前、Export前、AIレビュー前に実行。
- Canvas上のError BadgeはValidation結果をキャッシュして利用する。

### 8.2 Backend Validation

最終検証はBackendで実行する。
FrontendはUX向上のための事前検証に留める。

## 9. API通信最適化

- Project一覧とFlow詳細を分離取得する。
- Flow一覧ではFlow全体構造を取得しない。
- FlowEditorPage表示時に対象Flow詳細を取得する。
- 保存APIの二重実行を防止する。
- ExportはBackend実行とする。

## 10. Lazy Load方針

将来、以下をLazy Load対象にする。

- ExportDialog
- AIレビューUI
- SettingsPage
- 大型Template一覧
- 画像Preview

## 11. 仮想化方針

Canvas自体の仮想化はVue Flowの特性を踏まえて検討する。

初期は以下を対象にする。

- Project一覧
- Flow一覧
- Snapshot一覧
- Template一覧
- Validation Error一覧

## 12. 大規模Flow対応

大規模Flowでは以下を検討する。

- MiniMap簡略化
- Edge表示簡略化
- Node詳細表示の省略
- 検索結果中心表示
- Lane / Stage単位の折りたたみ
- 表示範囲外Nodeの軽量化

## 13. 禁止事項

- Node移動中に保存Requestを作成する。
- ComponentでFlow全体をDeep Watchする。
- Canvas上の全Nodeへ巨大なpropsを渡す。
- Export PDFをFrontend DOMから生成する。
- 入力1文字ごとにFlow全体Validationを実行する。

## 14. テスト観点

- 200Node規模でNode移動が操作可能である。
- Node単体更新で全Nodeが不要に再描画されない。
- 保存中に二重保存されない。
- Property Panelが選択対象だけを更新する。
- Flow一覧取得でFlow詳細を取得しない。
- Export実行中に画面が固まらない。

## 15. 完了条件

- Store更新、Adapter変換、Vue Flow描画の最適化方針が定義されている。
- 大規模Flowへの将来対応方針がある。
- ExportがFrontend負荷にならない設計である。
- AIが本書を読んで性能劣化を避けた実装に着手できる。
