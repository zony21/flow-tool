# 11_02_ディレクトリ構成

## 1. 目的

本書は、AI Flow DesignerのFrontendディレクトリ構成と命名規則を定義する。

Frontendは、Flow Canvas、Property Panel、Node Palette、API通信、Pinia Store、Vue Flow Adapterなど多くの責務を持つ。
責務が混在すると、Vue Componentが巨大化し、SSOTと表示モデルの境界が崩れる。

そのため、src配下の構成を明確化する。

## 2. 基本構成

```text
frontend/
 ├─ package.json
 ├─ vite.config.ts
 ├─ tsconfig.json
 └─ src/
     ├─ app/
     ├─ pages/
     ├─ layouts/
     ├─ features/
     ├─ components/
     ├─ stores/
     ├─ api/
     ├─ adapters/
     ├─ types/
     ├─ utils/
     ├─ router/
     ├─ assets/
     └─ styles/
```

## 3. app

アプリケーション起動設定を配置する。

```text
src/app/
 ├─ App.vue
 ├─ main.ts
 ├─ primevue.ts
 └─ plugins.ts
```

責務:

- Vue app生成
- PrimeVue登録
- Pinia登録
- Router登録
- 共通Plugin登録

禁止事項:

- 業務ロジックを置かない。
- Flow編集処理を置かない。

## 4. pages

ルーティング単位の画面を配置する。

```text
src/pages/
 ├─ ProjectListPage.vue
 ├─ ProjectDetailPage.vue
 ├─ FlowEditorPage.vue
 ├─ SettingsPage.vue
 └─ LoginPage.vue
```

責務:

- URL単位の画面構成
- Layout選択
- 初期データ取得指示
- Page単位の権限制御

禁止事項:

- Node描画処理を直接書かない。
- API通信詳細を直接書かない。
- Storeの内部構造に依存しすぎない。

## 5. layouts

画面枠を配置する。

```text
src/layouts/
 ├─ MainLayout.vue
 ├─ EditorLayout.vue
 └─ AuthLayout.vue
```

EditorLayoutの構成:

```text
Header
Toolbar
EditorMain
 ├─ NodePalette
 ├─ FlowCanvas
 └─ PropertyPanel
StatusBar
```

## 6. features

機能単位の実装を配置する。

```text
src/features/
 ├─ project/
 ├─ flow/
 ├─ editor/
 ├─ node/
 ├─ link/
 ├─ comment/
 ├─ export/
 ├─ template/
 ├─ auth/
 └─ aiReview/
```

### 6.1 project

Project一覧、Project作成、Project選択を担当する。

### 6.2 flow

Flow一覧、Flow作成、Flow保存、Flow取得を担当する。

### 6.3 editor

Canvas全体、Toolbar、Selection、Undo / Redoなど編集機能を担当する。

### 6.4 node

Node表示、Node種別、Node Property Formを担当する。

### 6.5 link

Link表示、条件、ラベル、接続Propertyを担当する。

### 6.6 comment

独立Comment、Node紐付けComment、AI専用メモを担当する。

### 6.7 export

Export UI、Export Request生成、Export結果表示を担当する。

### 6.8 template

Template選択、適用、複製を担当する。

### 6.9 auth

ログイン、ログアウト、ログイン状態表示を担当する。

### 6.10 aiReview

AIレビュー要求、結果表示を担当する。
初期は将来拡張領域として空構成でもよい。

## 7. components

再利用可能な共通UI部品を配置する。

```text
src/components/
 ├─ common/
 ├─ form/
 ├─ dialog/
 ├─ icon/
 └─ feedback/
```

例:

- AppButton
- AppDialog
- ConfirmDialog
- FormTextInput
- FormTextarea
- StatusBadge
- ValidationMessage

## 8. stores

Pinia Storeを配置する。

```text
src/stores/
 ├─ projectStore.ts
 ├─ flowStore.ts
 ├─ editorStore.ts
 ├─ selectionStore.ts
 ├─ undoRedoStore.ts
 ├─ exportStore.ts
 └─ authStore.ts
```

Storeは一時状態を保持する。
DBの正ではない。

## 9. api

Backend API Clientを配置する。

```text
src/api/
 ├─ httpClient.ts
 ├─ projectApi.ts
 ├─ flowApi.ts
 ├─ snapshotApi.ts
 ├─ exportApi.ts
 ├─ templateApi.ts
 └─ authApi.ts
```

原則:

- Componentからfetchを直接呼ばない。
- API URLをComponentに書かない。
- Request / Response型を明示する。

## 10. adapters

変換処理を配置する。

```text
src/adapters/
 ├─ flowToVueFlowAdapter.ts
 ├─ vueFlowToCommandAdapter.ts
 ├─ flowToSaveRequestAdapter.ts
 ├─ apiResponseToFlowAdapter.ts
 ├─ nodePropertyFormAdapter.ts
 └─ exportRequestAdapter.ts
```

Adapterの目的:

- SSOT構造とVue Flow表示モデルを分離する。
- API DTOとFrontend Store構造を分離する。
- Form ModelとDomain-like Modelを分離する。

## 11. types

TypeScript型を配置する。

```text
src/types/
 ├─ project.ts
 ├─ flow.ts
 ├─ lane.ts
 ├─ stage.ts
 ├─ node.ts
 ├─ link.ts
 ├─ comment.ts
 ├─ snapshot.ts
 ├─ export.ts
 └─ api.ts
```

型分類:

- API Request型
- API Response型
- Frontend Store型
- Vue Flow表示型
- Form型

これらを混同しない。

## 12. utils

純粋関数を配置する。

```text
src/utils/
 ├─ id.ts
 ├─ date.ts
 ├─ validation.ts
 ├─ array.ts
 └─ file.ts
```

禁止事項:

- Storeへ依存しない。
- APIへ依存しない。
- Componentへ依存しない。

## 13. 命名規則

| 対象 | 規則 | 例 |
| --- | --- | --- |
| Vue Component | PascalCase | FlowCanvas.vue |
| Store | camelCase + Store | flowStore.ts |
| API Client | camelCase + Api | flowApi.ts |
| Adapter | 変換元To変換先 | flowToVueFlowAdapter.ts |
| Type | PascalCase | FlowNode |
| Composable | use + Name | useFlowEditor.ts |

## 14. 完了条件

- src配下の責務分割が明確である。
- Vue Flow依存がAdapterとEditor機能周辺に閉じている。
- ComponentにAPI通信が散らばらない。
- Store、API、Adapter、Typeの配置場所が明確である。
- AIが本書を読んで初期ディレクトリを作成できる。
