# 11_15_テスト設計

## 1. 目的

本書は、AI Flow DesignerのFrontendテスト設計を定義する。

Frontendは、Vue3 Component、Pinia Store、Vue Flow Adapter、API Client、Dialog、Property Panel、Node Paletteなど多くの責務を持つ。
SSOTを正しく編集できることを担保するため、表示確認だけではなく、状態管理、変換、APIエラー、Undo / Redo、Export UIまでテスト対象とする。

## 2. 基本方針

- Store、Adapter、API ClientはUnit Testで検証する。
- Componentは表示とemitを中心に検証する。
- Flow Editorの主要操作はE2E Testで検証する。
- Backendはモックし、Frontendの責務を明確に検証する。
- Vue Flow内部挙動そのものではなく、アプリ側AdapterとStore更新を検証する。

## 3. テスト分類

| 分類 | 対象 | 目的 |
| --- | --- | --- |
| Unit Test | utils / adapters / stores | 純粋処理・状態更新検証 |
| Component Test | Vue Component | 表示・入力・emit検証 |
| API Mock Test | api client | Request / Error処理検証 |
| E2E Test | 画面操作 | 主要ユーザー操作検証 |
| Regression Test | 既知不具合 | 再発防止 |

## 4. Unit Test対象

### 4.1 Adapter

対象:

- flowToVueFlowAdapter
- vueFlowToCommandAdapter
- flowToSaveRequestAdapter
- apiResponseToFlowAdapter
- exportRequestAdapter

確認観点:

- Flow構造からVue Flow Nodes / Edgesへ変換できる。
- Vue Flow Nodeのselectedなど一時状態がSaveRequestへ混入しない。
- Link条件、通信方式、dataNameが欠落しない。
- 保存ResponseでeditingFlowを再同期できる。

### 4.2 Store

対象:

- flowStore
- editorStore
- selectionStore
- undoRedoStore
- exportStore
- authStore

確認観点:

- Flow取得後にeditingFlowが生成される。
- 保存成功後にisDirtyがfalseになる。
- 保存失敗時にeditingFlowが保持される。
- selection変更でProperty Panel対象が変わる。
- Undo / Redoで操作が戻る。

### 4.3 Utils

対象:

- id生成
- date整形
- validation補助
- array操作
- file補助

## 5. Component Test対象

### 5.1 EditorHeader

確認観点:

- Project / Flow名が表示される。
- 保存状態が表示される。
- 保存ボタン押下でsave emitが発火する。
- 未保存状態が表示される。

### 5.2 NodePalette

確認観点:

- 初期Nodeが表示される。
- Nodeカテゴリが表示される。
- Drag開始時にNode定義が渡る。

### 5.3 PropertyPanel

確認観点:

- selectedTypeに応じてFormが切り替わる。
- Node名変更でupdateが発火する。
- Validation Errorが項目直下に表示される。
- AI専用メモ欄が表示される。

### 5.4 Dialog

確認観点:

- ConfirmDialogで実行・キャンセルが動作する。
- UnsavedChangesDialogで3選択肢が表示される。
- ExportDialogでExport Optionが入力できる。
- SnapshotCreateDialogで未保存警告を表示できる。

## 6. API Mock Test

対象:

- projectApi
- flowApi
- snapshotApi
- exportApi
- authApi

確認観点:

- 正常Responseを受け取れる。
- 400 Validation Errorを標準形式で扱える。
- 401でログイン誘導状態になる。
- 403で権限不足表示へつながる。
- 409で競合Dialogへつながる。
- 500でtraceIdを取得できる。

## 7. E2E Test

代表シナリオ:

1. LoginPageを表示する。
2. Project一覧を表示する。
3. Projectを選択する。
4. Flowを作成する。
5. FlowEditorPageを表示する。
6. Node PaletteからNodeを追加する。
7. Nodeを選択しProperty Panelで名前を編集する。
8. Linkを接続する。
9. Link条件を入力する。
10. 保存する。
11. 保存成功状態が表示される。
12. Snapshot作成Dialogを開く。
13. ExportDialogを開く。

## 8. 未保存変更テスト

確認観点:

- Flow編集後にisDirtyがtrueになる。
- Flow切替時にUnsavedChangesDialogが表示される。
- 保存して移動で保存成功後に遷移する。
- 保存失敗時に遷移しない。
- 保存せず移動でeditingFlowが破棄される。

## 9. Error表示テスト

確認観点:

- Node名未入力がProperty Panelに表示される。
- Link参照不正がStatus Barに表示される。
- 保存失敗でHeaderに失敗状態が表示される。
- Export失敗でExportDialogにエラーが表示される。
- 500 ErrorでtraceIdが表示される。

## 10. Performance Test方針

初期段階では厳密な自動Performance Testは必須としない。

ただし、以下の簡易検証を行う。

- 200Node規模のFlowを表示できる。
- Node移動時に大きな遅延がない。
- Property Panel切替が遅くならない。
- 保存中にUIが固まらない。

将来、数千Node規模のBenchmarkを追加する。

## 11. テストデータ

テスト用Flowを用意する。

- 最小Flow
- 分岐ありFlow
- LoopありFlow
- CommentありFlow
- AI専用メモありFlow
- 画像NodeありFlow
- Validation ErrorありFlow

## 12. 禁止事項

- Component TestでBackend実APIへ接続する。
- Vue Flow内部実装そのものを過剰にテストする。
- Storeの状態更新をE2Eだけに依存する。
- Adapter変換テストを省略する。
- 未保存変更の確認テストを省略する。

## 13. 完了条件

- Store、Adapter、Component、API、E2Eのテスト対象が定義されている。
- SSOTとVue Flow表示モデルの分離をテストできる。
- 未保存変更、保存、Export、Error表示をテストできる。
- AIが本書を読んでFrontendテスト実装に着手できる。
