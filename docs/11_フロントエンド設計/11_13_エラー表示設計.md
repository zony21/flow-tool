# 11_13_エラー表示設計

## 1. 目的

本書は、AI Flow DesignerのFrontendエラー表示設計を定義する。

AI Flow Designerでは、Flow構造、Node参照、Link条件、Lane責務、Snapshot、Export、API通信など多くのエラーが発生し得る。
エラー表示が曖昧だと、ユーザーは何を修正すべきか分からず、SSOT品質が低下する。

そのため、エラー種別ごとの表示場所、文言、復旧導線を定義する。

## 2. 基本方針

- ユーザーが修正できるエラーは具体的に表示する。
- System Errorは詳細を出しすぎず、traceIdを表示する。
- Validation Errorは項目単位で表示する。
- Flow全体のエラーはStatus BarやValidation Summaryで表示する。
- API ErrorはHTTPステータスに応じて表示を分ける。
- Error、Warning、Infoを混同しない。

## 3. エラー分類

| 分類 | 例 | 表示場所 |
| --- | --- | --- |
| 入力Validation | Node名未入力 | Property Panel項目下 |
| 構造Validation | Link参照不正 | Status Bar / Canvas / Panel |
| API Error | 401 / 403 / 404 / 409 / 500 | Toast / Dialog / Page |
| 保存Error | 保存失敗 | Header / Status Bar / Toast |
| Export Error | PDF生成失敗 | ExportDialog / Toast |
| 認証Error | 未ログイン | LoginPage誘導 |
| 権限Error | Viewerが編集 | Toast / Disabled表示 |
| System Error | 予期しない障害 | ErrorDialog + traceId |

## 4. 表示場所

### 4.1 Property Panel

項目単位エラーを表示する。

例:

- Node名は必須です。
- 判定Nodeの分岐条件を入力してください。
- Link先Nodeが存在しません。
- Laneを選択してください。

### 4.2 Canvas

要素単位の視覚エラーを表示する。

例:

- Node右上にError Badge
- LinkをエラーStyleで表示
- Commentに警告Icon

### 4.3 Status Bar

Flow全体の状態を表示する。

例:

```text
未保存の変更あり / エラー3件 / 最終保存 2026/07/03 19:30
```

### 4.4 Toast

一時通知を表示する。

用途:

- 保存成功
- Export開始
- Export完了
- 一時的な通信失敗

### 4.5 Dialog

ユーザー判断が必要なエラーを表示する。

例:

- 保存競合
- Export失敗詳細
- System Error詳細

## 5. Backendエラー形式との対応

Backend標準エラー形式:

```json
{
  "errorCode": "FLOW_VALIDATION_FAILED",
  "message": "Flowの構造に不整合があります。",
  "traceId": "xxxxxxxx",
  "details": [
    {
      "field": "nodes[0].title",
      "message": "Node名は必須です。"
    }
  ]
}
```

Frontendはdetails.fieldをProperty Panelの項目へ紐付ける。
紐付けできない場合はValidation Summaryへ表示する。

## 6. HTTPステータス別表示

| HTTP | 表示方針 |
| --- | --- |
| 400 | Validation Errorとして表示 |
| 401 | LoginPageへ誘導 |
| 403 | 権限不足メッセージ表示 |
| 404 | 対象が存在しない表示 |
| 409 | 競合Dialog表示 |
| 422 | 未対応形式・未対応Nodeとして表示 |
| 500 | traceId付きSystem Error表示 |

## 7. 保存エラー

保存失敗時は以下を行う。

- editingFlowを失わない。
- isDirtyを維持する。
- Headerに保存失敗状態を表示する。
- ToastまたはStatus Barにエラーを表示する。
- Validation Errorがある場合は対象要素へ紐付ける。

## 8. Exportエラー

Export失敗時は以下を行う。

- ExportDialogを閉じない。
- exportStore.exportErrorへ保持する。
- traceIdがある場合は表示する。
- 再実行ボタンを表示する。

## 9. 認証・権限エラー

### 9.1 401

- LoginPageへ誘導する。
- 現在の遷移先を保持できる場合は保持する。

### 9.2 403

- 権限不足メッセージを表示する。
- 編集操作ボタンを非活性にする。
- Backendで拒否された操作は再試行しても同じ結果であることを示す。

## 10. 競合エラー

409 Conflictでは、以下の選択肢を提示する。

- 最新を再読み込み
- 自分の変更を破棄
- キャンセル

将来、差分比較UIを追加できる設計にする。

## 11. エラー文言方針

- 何が問題かを明確にする。
- ユーザーが次に何をすればよいか示す。
- 技術詳細を出しすぎない。
- traceIdは問い合わせ用として表示する。

悪い例:

```text
Error occurred.
```

良い例:

```text
Node名が未入力です。Property PanelでNode名を入力してください。
```

## 12. 禁止事項

- すべてのエラーをToastだけで表示する。
- Validation Errorを保存失敗の一言に潰す。
- 500エラーのStack Traceを画面に表示する。
- 保存失敗時に編集中内容を破棄する。
- 401と403を同じ扱いにする。

## 13. テスト観点

- Node名未入力が項目下に表示される。
- Link参照不正がCanvasとStatus Barに表示される。
- 401でLoginPageへ遷移する。
- 403で権限不足表示になる。
- 409で競合Dialogが表示される。
- 500でtraceIdが表示される。
- 保存失敗時にeditingFlowが保持される。

## 14. 完了条件

- エラー種別ごとの表示場所が定義されている。
- Backend標準エラーとの対応が定義されている。
- Validation Errorを項目単位で表示できる。
- 保存・Export・認証・権限・競合エラーの復旧導線がある。
- AIが本書を読んでエラー表示実装に着手できる。
