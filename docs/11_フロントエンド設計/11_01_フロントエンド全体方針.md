# 11_01_フロントエンド全体方針

## 1. 目的

本書は、AI Flow DesignerのFrontend全体方針を定義する。

Frontendは、ユーザーがシステムフローを視覚的に編集するための入口である。
ただし、本システムの正は画面上の図ではなく、Project / Flow / Lane / Stage / Node / Link / Commentから構成されるSSOTである。

そのため、Frontendは「見た目を直接保存する画面」ではなく、「SSOTを直感的に編集するためのUI」として設計する。

## 2. 採用技術

| 区分 | 技術 | 採用理由 |
| --- | --- | --- |
| Framework | Vue3 | Composition APIにより機能分割しやすい |
| Language | TypeScript | SSOT、DTO、UI Modelの型安全性を担保する |
| UI Component | PrimeVue | 業務画面に必要な入力・Dialog・Tableが揃う |
| Flow Canvas | Vue Flow | Node / Edge編集、Zoom、Pan、接続表現に適する |
| State | Pinia | Vue3標準に近く、Storeを分割しやすい |
| Build | Vite | 開発体験とビルド速度が高い |

## 3. UIコンセプト

採用するUI:

- Figmaのように直感的に配置できる。
- Miroのように広いCanvasで自由に編集できる。
- Notionのように情報整理しやすい。
- Linearのように操作が軽く、状態が分かりやすい。

採用しないUI:

- Excel風セル画面
- PLCを横一列に並べるだけの画面
- ページという概念を中心にした画面
- 手書き線を結ぶだけの画面
- Visio風の汎用作図ツール

## 4. Frontendの責務

Frontendは以下を担当する。

- Project選択
- Flow選択
- Flow Canvas表示
- Lane / Stage / Node / Link / Comment編集
- Node Palette表示
- Property Panel編集
- Undo / Redo
- Zoom / Pan / Grid / Snap
- 保存要求
- Export要求
- AIレビュー要求
- Validation Error表示
- 保存状態表示

Frontendが担当しないもの:

- 保存データの最終検証
- 権限判定の最終決定
- DB永続化
- Version Snapshot作成の正処理
- PDF / Mermaid / AI DSLの正式生成

## 5. SSOTとの関係

FrontendはSSOTを直接編集しているように見せるが、実際には以下の状態を扱う。

```text
Backend Saved Flow
  ↓ API Response
Frontend Editing Flow Store
  ↓ Adapter
Vue Flow Display Model
  ↓ User Operation
Editing Command
  ↓ Store Update
Save Request DTO
  ↓ API
Backend Validation / Save
```

重要な原則:

- Vue Flow Node / EdgeはSSOTではない。
- Pinia Storeは編集中の一時状態である。
- 保存成功後はBackend Responseで再同期する。
- Exportは保存済みFlowまたはSnapshotを入力とする。

## 6. レイヤ方針

Frontendは以下のレイヤで構成する。

```text
pages
layouts
features
components
stores
api
adapters
types
utils
```

### 6.1 pages

ルーティング単位の画面を配置する。

### 6.2 layouts

Header、Toolbar、左右Panel、StatusBarなどの画面枠を配置する。

### 6.3 features

Project、Flow、Editor、Export、Authなどの機能単位実装を配置する。

### 6.4 components

再利用可能なUI部品を配置する。

### 6.5 stores

Pinia Storeを配置する。

### 6.6 api

Backend API Clientを配置する。

### 6.7 adapters

SSOT構造、DTO、Vue Flow表示モデル、フォームモデルの変換を配置する。

### 6.8 types

TypeScript型定義を配置する。

### 6.9 utils

純粋関数や共通処理を配置する。

## 7. 実装原則

- Componentは大きくしすぎない。
- API通信をComponentへ直接書かない。
- Vue Flow依存を全体へ広げない。
- SSOT型と表示型を混同しない。
- Property Panelは選択対象ごとに責務分割する。
- Undo / RedoはVersion Snapshotと分離する。
- 保存前と保存後の状態差分を明確に扱う。
- Errorは画面全体通知と項目単位表示を分ける。

## 8. 完了条件

- Frontendの責務とBackendの責務が分離されている。
- Vue Flow表示モデルをSSOTとして扱わない方針が明確である。
- Pinia Storeが一時編集状態であることが明確である。
- Component、Store、API、Adapter、Typeの配置方針が明確である。
- AIが本書を読んでFrontend実装の大枠を判断できる。
