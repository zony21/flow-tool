# 11_08_API通信設計

## 1. 目的

本書は、AI Flow DesignerのFrontend API通信設計を定義する。

FrontendはBackend APIを通じてProject、Flow、Snapshot、Export、Authなどを操作する。
ただし、Componentから直接fetchやaxiosを呼び出すと、通信処理、エラー処理、DTO変換が散在する。

そのため、API通信はapi層へ集約し、StoreやFeatureから利用する。

## 2. 基本方針

- API Clientはsrc/api配下に集約する。
- Componentから直接HTTP通信しない。
- Request / Response型をTypeScriptで定義する。
- BackendのDomain EntityをFrontendへそのまま露出しない。
- API Responseは必要に応じてAdapterでStore Modelへ変換する。
- エラー形式はBackend標準エラーレスポンスに合わせる。

## 3. APIディレクトリ構成

```text
src/api/
 ├─ httpClient.ts
 ├─ apiError.ts
 ├─ projectApi.ts
 ├─ flowApi.ts
 ├─ snapshotApi.ts
 ├─ exportApi.ts
 ├─ templateApi.ts
 └─ authApi.ts
```

## 4. httpClient

### 4.1 目的

共通HTTP処理を担当する。

### 4.2 責務

- baseUrl設定
- JSON Header設定
- 認証Cookie / Token送信
- Timeout制御
- 共通エラー変換
- traceId取得

### 4.3 方針

初期実装ではfetchまたはaxiosのどちらでもよい。
ただし、呼び出し元はhttpClient経由に統一する。

## 5. API Error型

Backend標準エラーに合わせる。

```ts
export interface ApiErrorResponse {
  errorCode: string;
  message: string;
  traceId?: string;
  details?: ApiValidationError[];
}

export interface ApiValidationError {
  field: string;
  message: string;
}
```

Frontendでは、ApiErrorResponseを以下へ振り分ける。

- Global Error
- Toast
- Property Panel項目エラー
- Status Barエラー件数
- Dialog内エラー

## 7. flowApi

### 7.1 API例

- fetchFlows
- fetchFlow
- createFlow
- saveFlow
- deleteFlow

### 7.2 saveFlow方針

保存時は、flowStoreのeditingFlowを直接送るのではなく、flowToSaveRequestAdapterでSaveFlowRequestへ変換する。

```text
editingFlow
  ↓ Adapter
SaveFlowRequest
  ↓ flowApi.saveFlow
Backend
  ↓
SaveFlowResponse
  ↓ Adapter
Store Re-Sync
```

### 7.3 注意点

- 保存成功後はBackend Responseを正としてStoreを更新する。
- save中は二重保存を防ぐ。
- 409 Conflict時は再読み込みまたは上書き確認へ誘導する。

## 8. snapshotApi

### 8.1 API例

- fetchSnapshots
- createSnapshot
- fetchSnapshot
- restoreSnapshot

### 8.2 方針

Snapshot作成は保存済みFlowを対象とする。
未保存変更がある場合、Frontendで保存確認を表示する。

## 9. exportApi

### 9.1 API例

- exportMermaid
- exportPdf
- exportJson
- exportAiDsl

### 9.2 Request例

```ts
export interface ExportRequest {
  projectId: string;
  flowId: string;
  snapshotId?: string;
  exportType: 'mermaid' | 'pdf' | 'json' | 'aiDsl';
  includeComments: boolean;
  includeAiNotes: boolean;
  includeMetadata: boolean;
}
```

### 9.3 注意点

Export対象は保存済みFlowまたはSnapshotである。
未保存Frontend状態を直接Exportしない。

## 10. authApi

### 10.1 API例

- fetchCurrentUser
- logout
- getLoginUrl

GitHub OAuthのCallback処理はBackend側で実施する。
Frontendはログイン開始とログイン状態表示を担当する。

## 11. 認証・権限エラー対応

| HTTP | Frontend対応 |
| --- | --- |
| 401 | LoginPageへ誘導 |
| 403 | 権限不足メッセージ表示 |
| 404 | 対象なし表示 |
| 409 | 競合Dialog表示 |
| 500 | traceId付き汎用エラー表示 |

## 12. Timeout / Retry方針

初期方針:

- 通常APIは一定時間でTimeoutする。
- 保存APIの自動Retryは行わない。
- GET系APIは将来Retry可能にする。
- Exportは重い処理のため、将来非同期Job化に対応できる設計にする。

## 13. 禁止事項

- Componentからfetch / axiosを直接呼ぶ。
- API URLをComponentに直書きする。
- Response DTOをStoreやComponentで無秩序に加工する。
- Backend EntityをFrontend型としてそのまま使う。
- 401 / 403 / 409をすべて同じエラー表示にする。

## 14. テスト観点

- flowApi.saveFlowがSaveFlowRequestを送信できる。
- API Errorが標準形式に変換される。
- 401でログイン誘導できる。
- 409で競合Dialogへつながる。
- 保存成功ResponseでStoreを再同期できる。
- Export要求に未保存状態が混入しない。

## 15. 完了条件

- API通信がsrc/apiへ集約されている。
- Request / Response型が明確である。
- Error処理方針が定義されている。
- Componentから直接HTTP通信しない設計である。
- AIが本書を読んでAPI Client実装に着手できる。
