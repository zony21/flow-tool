# 11_12_テーマUI設計

## 1. 目的

本書は、AI Flow Designerのテーマ・UI設計を定義する。

AI Flow Designerは、Figma / Miro / Notion / Linear のようなモダンUIを目指す。
Excel風セル画面や古い業務画面ではなく、広いCanvas上で軽快に編集でき、同時に設計書として信頼できる情報密度を持つUIにする。

## 2. 基本方針

- モダンで軽量なUIにする。
- Canvasを主役にする。
- 情報の階層を明確にする。
- 状態色を統一する。
- PrimeVueを利用しつつ、アプリ独自のDesign Tokenで統制する。
- ダークテーマ対応を将来可能にする。

## 3. Design Token

Design Tokenは以下を定義する。

```text
styles/tokens/
 ├─ color.ts
 ├─ typography.ts
 ├─ spacing.ts
 ├─ radius.ts
 ├─ shadow.ts
 └─ zIndex.ts
```

## 4. Color Token

### 4.1 基本色

| Token | 用途 |
| --- | --- |
| color.background.base | 画面背景 |
| color.background.panel | Panel背景 |
| color.background.canvas | Canvas背景 |
| color.border.default | 通常境界線 |
| color.border.active | 選択中境界線 |
| color.text.primary | 主テキスト |
| color.text.secondary | 補助テキスト |
| color.text.muted | 非活性テキスト |
| color.primary | 主操作 |
| color.danger | 削除・破壊的操作 |
| color.warning | 警告 |
| color.success | 成功 |
| color.info | 情報 |

### 4.2 状態色

| 状態 | 用途 |
| --- | --- |
| 保存済み | success |
| 未保存 | warning |
| 保存中 | info |
| エラー | danger |
| 選択中 | primary |
| 非活性 | muted |

## 5. Typography

### 5.1 方針

- 画面密度を高めすぎない。
- HeaderやPanel名は短く明確にする。
- Node内テキストは視認性を優先する。
- 補足説明はsecondary textにする。

### 5.2 Token例

| Token | 用途 |
| --- | --- |
| font.size.xs | 補助情報 |
| font.size.sm | Panel内項目 |
| font.size.md | 通常本文 |
| font.size.lg | Header |
| font.weight.normal | 通常 |
| font.weight.medium | ラベル |
| font.weight.bold | 強調 |

## 6. Spacing

### 6.1 方針

- Panel内は8px基準にする。
- Button間隔は8pxまたは12pxにする。
- Header / Toolbarは詰めすぎない。
- Property Panelは入力しやすさを優先する。

### 6.2 Token例

- spacing.1 = 4px
- spacing.2 = 8px
- spacing.3 = 12px
- spacing.4 = 16px
- spacing.6 = 24px
- spacing.8 = 32px

## 7. Radius / Shadow

### 7.1 Radius

- 小さな入力部品: 6px
- Panel: 8px
- Dialog: 12px
- Node: Node種別に応じる

### 7.2 Shadow

Shadowは使いすぎない。

用途:

- Dialog
- Floating Panel
- Drag Preview
- Context Menu

## 8. Layout UI

### 8.1 Header

- 高さは固定する。
- Project / Flow選択を左寄せにする。
- Save / Export / AIレビューは右寄せにする。
- 保存状態を視認できる位置に置く。

### 8.2 Toolbar

- アイコン中心にする。
- Tooltipを表示する。
- Undo / Redoは左側にまとめる。
- Zoom / Grid / SnapはCanvas操作グループにまとめる。

### 8.3 Node Palette

- カテゴリごとに区切る。
- アイコン + Node名 + 短い説明で表示する。
- 将来検索欄を追加できる余白を持つ。

### 8.4 Property Panel

- 選択対象名を先頭に出す。
- 重要項目を上に配置する。
- AI専用メモは折りたたみ可能にする。
- Validation Errorを項目直下に表示する。

### 8.5 Status Bar

- 画面下部に固定する。
- 保存状態、エラー数、選択中要素、Zoomを表示する。
- 情報量を増やしすぎない。

## 9. Node UI

Nodeは種別ごとに視覚差を持たせる。

| Node | 表示方針 |
| --- | --- |
| start | 開始点として明確にする |
| end | 終了点として明確にする |
| process | 標準処理として最も汎用的にする |
| decision | 分岐と分かる形にする |
| hexagon | 外部・特殊処理を示す |
| balloon | 補足情報として軽く表示する |
| image | サムネイルまたは画像枠を表示する |

## 10. PrimeVue方針

PrimeVueは以下に使用する。

- Button
- Dialog
- Dropdown
- InputText
- Textarea
- Checkbox
- Tooltip
- Toast
- Menu
- DataTable

ただし、PrimeVue部品を直接散らばらせず、必要に応じてApp系Wrapperを使う。

## 11. Dark Theme方針

初期実装はLight Themeを主対象とする。

ただし、Token設計によりDark Themeを将来追加できるようにする。

Dark Theme対応時に調整が必要なもの:

- Canvas背景
- Grid色
- Node色
- Link色
- Panel背景
- Dialog背景
- Border色

## 12. アクセントカラー

アクセントカラーは主操作、選択中、フォーカスに使用する。

過剰に使わず、重要操作を明確にする。

## 13. 禁止事項

- 各Componentで直接色コードをばらばらに書く。
- 状態色を画面ごとに変える。
- Excel風のセル罫線UIにする。
- CanvasよりPanelが目立つUIにする。
- Error色とWarning色を混同する。

## 14. テスト観点

- 保存済み / 未保存 / エラー状態が視覚的に区別できる。
- 選択中Nodeが明確に分かる。
- Property PanelのValidation Errorが見える。
- DialogのPrimary / Cancelボタン位置が統一されている。
- Tokenを変更するとUI全体に反映される。

## 15. 完了条件

- Design Tokenが定義されている。
- Header / Toolbar / Palette / Canvas / Property Panel / Status BarのUI方針が明確である。
- 状態色と操作色が統一されている。
- PrimeVue利用方針が定義されている。
- AIが本書を読んでUI基盤実装に着手できる。
