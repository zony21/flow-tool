# 05_08_エラー設計

## 1. 目的

APIの共通エラー形式とHTTPステータスの使い分けを定義する。
画面側が一貫したエラー表示を行えるようにする。

## 2. 共通エラー形式

```json
{
  "code": "VALIDATION_ERROR",
  "message": "入力値を確認してください。",
  "details": [],
  "traceId": "xxx"
}
```

## 3. HTTPステータス

| Status | 用途 |
| --- | --- |
| 400 | 入力値不正 |
| 401 | 未ログイン |
| 403 | 権限不足 |
| 404 | 対象なし |
| 409 | 競合 |
| 500 | サーバ処理失敗 |

## 4. Validation Error

入力項目ごとのエラーをdetailsへ格納する。
画面は対象項目の下にメッセージを表示する。

## 5. Not Found

存在しないProject、Flow、FlowVersion、Templateを指定した場合は404を返す。

## 6. Conflict

Version番号重複、同時更新競合、EditorLock競合は409を返す。

## 7. Internal Error

内部例外やDB例外の詳細は返さない。
ログにはtraceIdと詳細を記録する。

## 8. テスト観点

- ステータスごとのErrorResponseが返ること
- 内部例外が画面へ返らないこと
- Validation詳細が項目単位で返ること
- 409が競合として扱えること

## 9. 完了条件

画面がAPIエラーを一貫して処理できる形式になっていること。
