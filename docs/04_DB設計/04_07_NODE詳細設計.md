# 04_07 NODE Detailed Design

## 1. Purpose

The `NODE` table stores structured processing units within a Flow. Transport Nodes may contain optional references to Transport master data.

## 2. Responsibilities and Relationships

```text
FLOW 1:N NODE
LANE 1:N NODE
STAGE 1:N NODE
NODE 1:N LINK
NODE N:1 TRANSPORT_COMMAND (optional)
NODE N:1 TRANSPORT_EQUIPMENT (optional)
NODE N:1 TRANSPORT_LOCATION (optional)
```

Location remains Project-level master data. A Node stores only its optional `location_id` reference and never duplicates Location name, type, or description.

## 3. Column Definition

| Column | Type | NULL | Description |
| --- | --- | --- | --- |
| node_id | TEXT | NO | Primary key |
| flow_id | TEXT | NO | Flow reference |
| lane_id | TEXT | YES | Lane reference |
| stage_id | TEXT | YES | Stage reference |
| node_type | TEXT | NO | Node type |
| name | TEXT | NO | Display name |
| description | TEXT | YES | Description |
| command_id | TEXT | YES | Transport command reference; Transport Flow only |
| equipment_id | TEXT | YES | Transport equipment reference; Transport Flow only |
| location_id | TEXT | YES | Project Transport Location reference; Transport Flow only |
| rw_type | TEXT | YES | `NONE`, `READ`, or `WRITE`; Transport Flow only |
| x | REAL | NO | X coordinate |
| y | REAL | NO | Y coordinate |

Indexes exist for the Transport reference columns. Application-level validation is authoritative for Transport master references; destructive foreign-key migration changes are outside this cycle.

## 4. Validation

### Normal Flow

```text
command_id = null
equipment_id = null
location_id = null
rw_type = NONE
```

### Transport Flow

When specified:

- `CommandId` references a valid non-deleted command.
- `EquipmentId` references valid non-deleted equipment in the same Project.
- `LocationId` references a valid non-deleted Location in the same Project.
- `RwType` is `NONE`, `READ`, or `WRITE`.

Command-category-specific required-field validation is a later implementation phase.

## 5. Save, Load, and Version Snapshot

The Flow Structure API accepts and restores `CommandId`, `EquipmentId`, `LocationId`, and `RwType` through Node request and response data.

All four values must be preserved in Node data stored in a Flow Version Snapshot. Resolved master display values are not included because their historical lifecycle requires a separate strategy.

## 6. Duplicate Flow

Same-Project duplication:

- regenerates Flow, Lane, Stage, Node, Link, and Comment IDs;
- remaps Link and Comment Node references;
- retains valid `CommandId`, `EquipmentId`, `LocationId`, and `RwType` values.

Cross-Project duplication must not blindly copy master IDs and is outside this implementation cycle.

## 7. Transport Template Relationship

The future Transport Template Generator must resolve the Location column from `Node.LocationId`. It must not infer Location from coordinates, Equipment, Lane, Stage, Project layout, or screen layout.

## 8. Test Conditions

Automated tests must confirm:

- Transport references survive save and load.
- cross-Project and deleted Location references are rejected.
- Normal Flow clears all Transport fields.
- changing a Transport Flow to Normal clears persisted Transport fields.
- Version Snapshot retains all four Transport attributes.
- same-Project duplication retains all four attributes while regenerating structural IDs.
- Node Detail displays Location only for Transport Flow and emits the selected `LocationId`.
- frontend save mapping prefers the newest explicit request values.
