# 04_07_NODE詳細設計

## 1. 本書の目的

NODEテーブルは、FlowVersion内の処理単位を保持する。開始、終了、処理、判定、画像、吹き出しなどのノードを構造化データとして管理する。

Transport Flowでは、搬送表生成に必要なコマンド、ロケーション、設備、R/WもNode属性として保持する。

## 2. テーブル概要

NODEはLaneとStageに属し、画面上の座標、サイズ、図形種別、処理説明、AIメモを保持する。

Transport Flowの場合は、AGF/AGVメーカーコマンド、ロケーション、設備、通信方向を保持する。

## 3. 採用理由

図形ではなくNodeとして意味を保存することで、AIが処理主体、工程、処理内容を正確に理解できる。

搬送表はNodeの実行順と属性から生成するため、Transport専用属性もNodeに紐付けて保存する。

## 4. 利用機能

フローエディタ、プロパティパネル、Mermaid出力、PDF出力、JSON出力、AI DSL出力で利用する。

Transport Flowでは搬送表出力でも利用する。

## 5. 関連画面

フローエディタ画面、ノードパレット、プロパティパネル。

Transport Flowではロケーション管理ダイアログ、AGF/AGVコマンド管理画面、設備管理画面とも関連する。

## 6. 関連API

GET/PUT /api/flow-versions/{flowVersionId}、将来Node個別API。

Transport属性はFlow構造保存APIのNode DTOに含めて保存する。

## 7. ER上の位置

FLOW_VERSION 1:N NODE、LANE 1:N NODE、STAGE 1:N NODE、NODE 1:N LINK。

Transport Flowの場合、NODEはTRANSPORT_COMMAND、LOCATION、EQUIPMENTを任意参照する。

## 10. PK

node_idをPKとする。

## 11. FK

flow_version_id、lane_id、stage_idを参照する。

Transport属性ではcommand_id、location_id、equipment_idを任意参照する。

## 12. Unique

同一FlowVersion内でdisplay_nameは重複可。内部参照はnode_idを使用する。

## 13. Index

- idx_node_version_lane_stage: flow_version_id, lane_id, stage_id, is_deleted
- idx_node_command: command_id
- idx_node_location: location_id
- idx_node_equipment: equipment_id

## 14. Default値

x=0、y=0、width=160、height=80、is_deleted=0。

rw_typeは未設定時NONE相当として扱う。

## 15. NULL可否

display_name、node_type、lane_id、stage_idはNOT NULL。

command_id、location_id、equipment_id、rw_typeはTransport Flowでのみ利用するためNULL可とする。

## 16. CHECK制約

width、heightは0より大きい。is_deletedは0/1。

rw_typeは以下のいずれかとする。

- NONE
- READ
- WRITE

## 17. 論理削除

Node削除時は関連Link削除確認を行う。通常は論理削除する。

## 18. 監査カラム

共通監査カラムを保持する。

## 19. 更新タイミング

追加、移動、サイズ変更、プロパティ変更、Transport属性変更、削除時。

## 20. 削除ルール

Linkが存在する場合、関連Linkも削除するか確認する。

Transport Flowで参照中のコマンド、ロケーション、設備が削除される場合は参照整合を確認する。

## 21. Versionとの関係

NODEはFlowVersion Snapshotの一部である。

Transport属性もSnapshotに含める。

## 22. Templateとの関係

Template適用時はnode_idを再採番し、Link参照を張り替える。

Transport Templateの場合、command_id、location_id、equipment_idの扱いはProject内マスタと照合する。

## 23. サンプルデータ

```sql
insert into NODE(node_id, flow_version_id, lane_id, stage_id, node_type, display_name, command_id, location_id, equipment_id, rw_type, x, y, width, height, display_order, is_deleted, created_at, created_by, updated_at, updated_by)
values('n-001','fv-001','l-001','s-001','PROCESS','P2へ移動','cmd-001','loc-p2',null,'NONE',100,120,160,80,1,0,'2026-07-03T00:00:00','u-001','2026-07-03T00:00:00','u-001');
```

## 24. SQL例

```sql
select * from NODE where flow_version_id='fv-001' and is_deleted=0 order by display_order;
```

## 25. パフォーマンス

FlowVersion単位の一括取得を基本とし、Lane/Stage絞り込みをIndexで支援する。

Transport属性は搬送表出力時にまとめて参照するため、command_id、location_id、equipment_idへIndexを設定する。

## 26. 将来拡張

NodeTypeマスタ、添付ファイル、AIレビュー結果、コード生成属性を追加可能。

Transport属性はPLCシーケンス表、WCS/RCS通信表、設備操作表などへ拡張可能。

## 27. テスト観点

Node追加、移動、Lane/Stage変更、Link整合、Version複製、Template適用を確認する。

Transport Flowでは、コマンド、ロケーション、設備、R/Wを保存・復元できることを確認する。

## 28. 完了条件

Nodeが意味情報と表示情報を保持し、AIが処理を誤解しない構造で保存できること。

Transport Flowでは、Node属性から搬送表を生成できること。
