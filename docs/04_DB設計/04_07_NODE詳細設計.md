# 04_07 NODE Detailed Design

## 1. Purpose

The `NODE` table stores structured processing units within a Flow, including start, end, process, decision, image, and annotation nodes.

In a Transport Flow, a Node stores only attributes that describe the action executed by the Node.

## 2. Node Responsibility

A Node belongs to a Flow, Lane, and Stage and stores:

- Node type
- Display name
- Description
- Coordinates
- Transport command reference
- Transport equipment reference
- R/W type

A Node must not store a physical Location reference.

## 3. Location Responsibility

A Location represents the physical position of client equipment or a facility point within a Project.

Location is Project-level master data and must be associated with the relevant equipment/facility structure, not directly with a Node.

The previous `NODE.location_id` design is obsolete.

## 4. Reason for the Design

A Node represents an action or processing step.

A Location represents a physical client facility position.

These responsibilities must remain separate to prevent duplicated or contradictory position data across multiple Nodes that refer to the same equipment or facility.

## 5. Related Functions

The `NODE` table is used by:

- Flow Editor
- Node Property Panel
- Mermaid export
- JSON export
- AI DSL export
- Version Snapshot
- Transport Template Generator

Location management is not a Node Property Panel function.

## 6. Related APIs

Transport Node attributes are saved through the Flow Structure API.

The Node DTO may contain:

- `commandId`
- `equipmentId`
- `rwType`

The Node DTO must not contain `locationId`.

## 7. Relationships

```text
FLOW 1:N NODE
LANE 1:N NODE
STAGE 1:N NODE
NODE 1:N LINK
NODE N:1 TRANSPORT_COMMAND (optional)
NODE N:1 TRANSPORT_EQUIPMENT (optional)
```

There is no direct relationship between `NODE` and `TRANSPORT_LOCATION`.

## 8. Table Definition

| Logical Name | Physical Name |
| --- | --- |
| Node | NODE |

## 9. Column List

| Column | Type | NULL | Description |
| --- | --- | --- | --- |
| node_id | TEXT | NO | Primary key |
| flow_id | TEXT | NO | Flow reference |
| lane_id | TEXT | YES | Lane reference |
| stage_id | TEXT | YES | Stage reference |
| node_type | TEXT | NO | Node type |
| name | TEXT | NO | Display name |
| description | TEXT | YES | Description |
| command_id | TEXT | YES | Transport command reference; Transport Flow only |
| equipment_id | TEXT | YES | Transport equipment reference; Transport Flow only |
| rw_type | TEXT | YES | `NONE`, `READ`, or `WRITE`; Transport Flow only |
| x | REAL | NO | X coordinate |
| y | REAL | NO | Y coordinate |

`location_id` is not part of the target NODE design.

## 10. Primary Key

`node_id` is the primary key.

## 11. Foreign Keys

The target relationships are:

- `flow_id`
- `lane_id`
- `stage_id`
- `command_id`
- `equipment_id`

`location_id` must not be added as a Node foreign key.

## 12. Indexes

Recommended indexes:

- Flow/Lane/Stage lookup index
- `command_id`
- `equipment_id`

A Node Location index is not required.

## 13. Default Values

- `x = 0`
- `y = 0`
- `rw_type = NONE`

## 14. Validation

For Normal Flow:

- `command_id = null`
- `equipment_id = null`
- `rw_type = NONE`

For Transport Flow:

- `command_id` must reference an active command when specified.
- `equipment_id` must reference active equipment in the same Project when specified.
- `rw_type` must be `NONE`, `READ`, or `WRITE`.

No Node Location validation is required.

## 15. Version Snapshot

Node command, equipment, and R/W attributes are included in the Flow Version Snapshot.

Location is not included through Node data. Project-level Location master data has a separate lifecycle.

## 16. Duplicate Flow

When duplicating a Flow:

- Node IDs are regenerated.
- Lane and Stage references are remapped.
- Command and Equipment references may be retained when valid in the same Project.
- No Location reference is copied because Node does not store Location.

## 17. Transport Template Relationship

The Transport Template Generator uses Flow, Link, Node, and Node action attributes.

When a template requires a physical location, it must resolve the location through the Project-level equipment/facility model defined by the Transport Template Generator specification.

It must not read `Node.LocationId`.

## 18. Test Conditions

Confirm that:

- Node save and load preserve command, equipment, and R/W type.
- Normal Flow clears Transport Node attributes.
- Duplicate and Version Snapshot preserve supported Transport attributes.
- Node DTO and database mapping do not require Location.
- Node Detail does not display a Location selector or Location creation action.

## 19. Completion Condition

The Node structure correctly separates action information from physical client facility position information.

This design supersedes the previous design that stored `location_id` directly in `NODE`.
