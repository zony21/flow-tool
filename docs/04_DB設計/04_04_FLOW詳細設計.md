# 04_04_FLOW詳細設計

## 1. 本書の目的

本書は、FLOWテーブルの詳細設計を定義する。FLOWはProject配下の作業単位を表し、RFID読取り、搬送要求、ラベル印刷、AGF搬送などのフローを管理する。

## 2. テーブル概要

FLOWはプロジェクト内の設計対象フローの見出し情報を保持する。実際のLane、Stage、Node、Link、CommentはFLOW_VERSION配下にSnapshotとして保持する。

## 3. 採用理由

Flow本体とVersionを分けることで、フロー名やカテゴリなどの管理情報と、版ごとの構造データを分離できる。

AGF/AGV搬送フローでは通常フローと異なる入力項目、マスタ参照、搬送表出力を持つため、FLOWに種別を保持する。

## 4. 利用機能

フロー一覧、フロー作成、フロー複製、バージョン管理、出力導線で利用する。

Transport Flowの場合は、メーカーコマンド、ロケーション、設備、R/W、搬送表出力の表示制御にも利用する。

## 5. 関連画面

フロー一覧画面、フローエディタ画面、バージョン管理画面、出力画面。

Transport FlowではAGF/AGVメーカー管理画面、AGF/AGVコマンド管理画面、設備管理画面、ロケーション管理ダイアログとも関連する。

## 6. 関連API

GET /api/projects/{projectId}/flows、POST /api/projects/{projectId}/flows、GET/PUT/DELETE /api/projects/{projectId}/flows/{flowId}。

Flow作成・更新時はflow_typeを受け取る。

## 7. ER上の位置

PROJECT 1:N FLOW、FLOW 1:N FLOW_VERSION。

Transport Flowの場合、FLOWはTRANSPORT_MANUFACTURER、TRANSPORT_COMMAND、LOCATION、EQUIPMENTを参照するNode属性の親コンテキストとなる。

## 8. テーブル定義

| 論理名 | 物理名 |
| --- | --- |
| フロー | FLOW |

## 9. カラム一覧

| カラム | 型 | NULL | 説明 |
| --- | --- | --- | --- |
| flow_id | TEXT | NO | PK、GUID |
| project_id | TEXT | NO | PROJECT FK |
| flow_name | TEXT | NO | フロー名 |
| flow_type | TEXT | NO | NORMALまたはTRANSPORT |
| category | TEXT | YES | カテゴリ |
| description | TEXT | YES | 説明 |
| display_order | INTEGER | NO | 表示順 |
| is_deleted | INTEGER | NO | 論理削除 |
| created_at | TEXT | NO | 作成日時 |
| created_by | TEXT | NO | 作成者 |
| updated_at | TEXT | NO | 更新日時 |
| updated_by | TEXT | NO | 更新者 |
| deleted_at | TEXT | YES | 削除日時 |
| deleted_by | TEXT | YES | 削除者 |

## 10. PK

flow_idをPKとする。

## 11. FK

project_idはPROJECT.project_idを参照する。

## 12. Unique

同一project_id内でflow_nameの重複を許可するかは運用判断とする。初期はUniqueなし。

## 13. Index

- idx_flow_project_deleted_order: project_id, is_deleted, display_order
- idx_flow_project_updated: project_id, updated_at
- idx_flow_project_type: project_id, flow_type, is_deleted

## 14. Default値

- flow_type: NORMAL
- display_order: 0
- is_deleted: 0

## 15. NULL可否

flow_name、flow_type、project_id、監査カラムはNOT NULL。category、description、削除監査はNULL可。

## 16. CHECK制約

is_deletedは0または1。display_orderは0以上。

flow_typeは以下のいずれかとする。

- NORMAL
- TRANSPORT

## 17. 論理削除

削除時はis_deleted=1とし、配下FLOW_VERSIONは保持する。

## 18. 監査カラム

共通監査カラムを保持する。

## 19. 更新タイミング

Flow作成、名称変更、説明変更、表示順変更、種別変更、削除時に更新する。

## 20. 削除ルール

削除時はFlowのみ論理削除する。VersionやSnapshotは履歴として保持する。

## 21. Versionとの関係

FLOWは複数のFLOW_VERSIONを持つ。最新版はFLOW_VERSION側のis_latestで判定する。

## 22. Templateとの関係

TemplateからFlowを作成する場合、FLOWを作成後、初期FLOW_VERSIONと配下要素を生成する。

Template側にもFlowTypeを保持し、Transport Templateから作成した場合はTransport Flowとして作成する。

## 23. サンプルデータ

```sql
insert into FLOW(flow_id, project_id, flow_name, flow_type, category, display_order, is_deleted, created_at, created_by, updated_at, updated_by)
values('f-001','p-001','RFID読取り','NORMAL','包装',1,0,'2026-07-03T00:00:00','u-001','2026-07-03T00:00:00','u-001');
```

## 24. SQL例

```sql
select * from FLOW where project_id = 'p-001' and is_deleted = 0 order by display_order;
```

Transport Flowのみ抽出する場合:

```sql
select * from FLOW where project_id = 'p-001' and flow_type = 'TRANSPORT' and is_deleted = 0 order by display_order;
```

## 25. パフォーマンス

project_id、is_deleted、display_orderの複合Indexにより一覧表示を高速化する。

Transport Flow絞り込みはproject_id、flow_type、is_deletedの複合Indexで支援する。

## 26. 将来拡張

Flow単位権限、タグ、公開状態、レビュー状態を追加可能とする。

FlowTypeは将来、PLC Sequence、WCS/RCS Communicationなどの専用Flow種別追加に拡張可能とする。

## 27. テスト観点

Flow作成、一覧、論理削除、Project別絞り込み、最新版取得連携を確認する。

Transport Flow作成時にflow_typeがTRANSPORTとして保存され、フローエディタ表示制御へ反映されることを確認する。

## 28. 完了条件

FLOWとFLOW_VERSIONの責務分離が明確で、画面・API・DB実装へ展開できる状態であること。

FlowTypeにより通常フローとAGF/AGV搬送フローを安全に分離できること。
