# 04_18_インデックス設計

## 1. 本書の目的

本書は、AI Flow Designer のIndex設計を定義する。
Indexは一覧検索、JOIN、ORDER BY、最新版取得、未読件数取得などの性能を安定させるために設計する。

## 2. 基本方針

Indexは闇雲に追加しない。
画面検索条件、API検索条件、JOIN条件、ORDER BY、想定件数を根拠として追加する。

## 3. 採用理由

FlowVersion配下にはNode、Link、Commentが増えるため、flow_version_idを基点とした検索を高速化する必要がある。

## 4. 利用機能

全一覧画面、フローエディタ、出力処理、バージョン管理、通知。

## 5. 関連画面

プロジェクト一覧、フロー一覧、フローエディタ、バージョン管理、出力画面、通知表示。

## 6. 関連API

全GET API、Export API、Auth API。

## 7. ER上の位置

PROJECT、FLOW、FLOW_VERSION、LANE、STAGE、NODE、LINK、COMMENTを中心にIndexを設定する。

## 8. Index一覧

| Index | 対象 | 採用理由 |
| --- | --- | --- |
| idx_project_deleted_order | PROJECT(is_deleted, display_order) | 一覧表示 |
| idx_flow_project_deleted_order | FLOW(project_id, is_deleted, display_order) | Project配下Flow一覧 |
| idx_flow_version_latest | FLOW_VERSION(flow_id, is_latest, is_deleted) | 最新版取得 |
| idx_lane_version_order | LANE(flow_version_id, is_deleted, display_order) | レーン一覧 |
| idx_stage_version_order | STAGE(flow_version_id, is_deleted, display_order) | 工程一覧 |
| idx_node_version_lane_stage | NODE(flow_version_id, lane_id, stage_id, is_deleted) | エディタ表示 |
| idx_link_version | LINK(flow_version_id, is_deleted) | 接続一覧 |
| idx_link_from | LINK(from_node_id) | Node削除時確認 |
| idx_link_to | LINK(to_node_id) | Node削除時確認 |
| idx_comment_version | COMMENT(flow_version_id, is_deleted) | コメント一覧 |
| idx_project_member_user | PROJECT_MEMBER(user_id, is_deleted) | ユーザー別Project一覧 |
| idx_notification_user_read | NOTIFICATION(user_id, is_read, created_at) | 未読通知 |

## 9. 検索条件

主要検索条件はis_deleted除外、project_id指定、flow_id指定、flow_version_id指定である。

## 10. ORDER BY

一覧はdisplay_order、updated_at、created_atを利用する。

## 11. JOIN

JOINは親子IDで行う。JOIN対象カラムにはIndexを設ける。

## 12. 想定件数

1 FlowVersionあたりNode 100件、Link 200件、Comment 100件程度を初期想定とする。

## 13. 複合Index方針

単一Indexではなく、実際のWHERE + ORDER BYに合わせて複合Indexを優先する。

## 14. SQLite差異

SQLiteでは部分Indexや制約挙動に注意する。EF Coreで表現可能な範囲を優先する。

## 15. SQL Server差異

SQL Server移行時はinclude列、filtered indexを検討する。

## 16. テスト観点

- 主要一覧でIndexが使われること
- 最新版取得が高速であること
- Node削除時のLink検索が高速であること
- 未読通知件数が高速に取得できること

## 17. 完了条件

Indexの採用理由、検索条件、JOIN、ORDER BY、想定件数が明確であること。
