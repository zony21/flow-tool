# 04_02_ER図

## 1. 本書の目的

本書は、AI Flow Designer のER構造とリレーション方針を定義する。
ER図だけでなく、各リレーションの多重度、Cascade/Restrict、採用理由を明確にする。

## 2. ER概要

中心はProject、Flow、FlowVersionである。
FlowVersion配下にLane、Stage、Node、Link、CommentをSnapshotとして保持する。

```text
USER 1 ---- N PROJECT_MEMBER N ---- 1 PROJECT
PROJECT 1 ---- N FLOW
FLOW 1 ---- N FLOW_VERSION
FLOW_VERSION 1 ---- N LANE
FLOW_VERSION 1 ---- N STAGE
FLOW_VERSION 1 ---- N NODE
FLOW_VERSION 1 ---- N LINK
FLOW_VERSION 1 ---- N COMMENT
NODE 1 ---- N LINK(FROM)
NODE 1 ---- N LINK(TO)
NODE 1 ---- N COMMENT(optional)
TEMPLATE 1 ---- N TEMPLATE_ITEM
FLOW_VERSION 1 ---- N EXPORT_HISTORY
FLOW_VERSION 1 ---- 0..1 EDITOR_LOCK
USER 1 ---- N USER_SETTING
USER 1 ---- N NOTIFICATION
```

## 3. 主要リレーション

| 親 | 子 | 多重度 | 削除方針 | 採用理由 |
| --- | --- | --- | --- | --- |
| PROJECT | FLOW | 1:N | Restrict/論理削除 | Flow履歴を不用意に消さないため |
| FLOW | FLOW_VERSION | 1:N | Restrict/論理削除 | Versionは設計履歴として保持するため |
| FLOW_VERSION | LANE | 1:N | Cascade候補 | Version Snapshot内の構成要素であるため |
| FLOW_VERSION | STAGE | 1:N | Cascade候補 | Version Snapshot内の構成要素であるため |
| FLOW_VERSION | NODE | 1:N | Cascade候補 | Version Snapshot内の構成要素であるため |
| FLOW_VERSION | LINK | 1:N | Cascade候補 | Version Snapshot内の構成要素であるため |
| FLOW_VERSION | COMMENT | 1:N | Cascade候補 | Version Snapshot内の構成要素であるため |
| NODE | LINK | 1:N | Restrict/アプリ制御 | LinkのFROM/TO整合性を明示制御するため |
| USER | PROJECT_MEMBER | 1:N | Restrict | 参加履歴を保持するため |

## 4. Cascade方針

Version配下のSnapshot構成要素は、FlowVersionを物理削除する場合にのみCascade候補とする。
通常運用ではFlowVersionを論理削除するため、Cascade削除は頻繁に発生しない。

## 5. Restrict方針

Project、Flow、Userなどの親データは履歴性が強いため、子データが存在する場合は物理削除を制限する。
画面上では削除済み扱いにし、通常検索から除外する。

## 6. N:M方針

PROJECTとUSERはPROJECT_MEMBERでN:Mを表現する。
将来的にFlow単位権限が必要な場合はFLOW_MEMBERまたはFLOW_PERMISSIONを追加する。

## 7. Versionとの関係

Lane、Stage、Node、Link、CommentはFlowVersionに属する。
最新版だけを上書きするのではなく、Version作成時にSnapshotを複製する。

## 8. Templateとの関係

TemplateはFlowVersionに近い構造を持つが、実体Flowではない。
Template適用時はIDを再採番し、LinkやCommentの参照を新しいNode IDへ張り替える。

## 9. テスト観点

- ProjectからFlowを辿れること
- Flowから複数Versionを辿れること
- VersionからLane/Stage/Node/Link/Commentを辿れること
- Linkが存在するNodeを参照していること
- Template適用時に参照張替えが行われること

## 10. 完了条件

本ER設計は、API設計、Entity設計、Migration設計へ展開できるリレーションが明確になった場合に完了とする。
