# 04_01_DB設計方針

## 1. 本書の目的

本書は、AI Flow Designer のDB設計における共通方針を定義する。
本システムでは、画面上の図形ではなく、Project、Flow、Lane、Stage、Node、Link、Comment、Version、Templateなどの構造化データをSingle Source of Truthとして扱う。

## 2. DB設計の基本思想

DBにはフロー図の画像そのものを正として保存しない。
保存するのは、AIと人間が同じ意味で解釈できる構造化データである。

中心構造は以下とする。

```text
PROJECT
  └ FLOW
      └ FLOW_VERSION
          ├ LANE
          ├ STAGE
          ├ NODE
          ├ LINK
          └ COMMENT
```

## 3. 採用理由

ExcelやVisioのような図形ベース資料では、AIはノードの意味、担当機器、工程、接続条件を推測する必要がある。
DBで構造化データを正として保持することで、AIは推測ではなく、テーブル間の関係からフローを理解できる。

## 8. 主キー方針

主キーは原則としてGUIDを利用可能な設計とする。
SQLite初期実装ではTEXT GUID、SQL Server移行後はuniqueidentifierへ移行しやすい設計とする。

## 9. 外部キー方針

参照整合性をDBレベルでも保持する。
削除時の動作は業務意味に応じてCascadeとRestrictを使い分ける。
設計履歴に関係するデータは不用意にCascade削除しない。

## 10. 論理削除方針

設計情報は履歴性が重要であるため、主要テーブルは論理削除を基本とする。
物理削除は一時データ、ロック、通知などに限定する。

## 11. 監査カラム方針

主要テーブルには以下を保持する。

- created_at
- created_by
- updated_at
- updated_by
- is_deleted
- deleted_at
- deleted_by

## 12. Version設計方針

VersionはSnapshotとして扱う。
Undo/Redoは画面操作履歴であり、Versionとは責務を分離する。
FlowVersionには、その時点のLane、Stage、Node、Link、Commentを含める。

## 13. Template設計方針

TemplateにはLane、Stage、Node、Link、Commentを含める。
Template適用時はIDを再採番し、Node参照やLink参照を張り替える。

## 14. Index方針

Indexは以下を基準に設計する。

- 一覧検索条件
- JOIN条件
- 表示順ソート
- 最新版取得
- 論理削除除外

## 15. Migration方針

EF Core Migrationを使用する。
Migration名は目的が分かる英語名とし、DB変更とEntity変更を一致させる。
Rollback可能性を考慮し、破壊的変更ではデータ退避を設計する。

## 16. SQL Server移行方針

初期DBはSQLiteとするが、SQL Server移行を前提に以下を意識する。

- GUID型
- DateTime型
- Boolean型
- FK制約
- CHECK制約
- Index差異
- Identity差異

## 17. テスト観点

- ERの中心構造が維持されること
- FlowVersionがSnapshotとして保存されること
- 表示名変更で内部参照が壊れないこと
- 論理削除済みデータが通常検索から除外されること
- SQLiteとSQL Serverで移行可能な定義であること
