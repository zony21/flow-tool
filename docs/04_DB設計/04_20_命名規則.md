# 04_20_命名規則

## 1. 本書の目的

本書は、AI Flow Designer のDB命名規則を定義する。
Entity、テーブル、カラム、Index、FK、Migrationの命名を統一し、実装者とAIが構造を誤解しないようにする。

## 2. テーブル命名

テーブル名は英大文字スネークケースを基本とする。

例:

- PROJECT
- FLOW
- FLOW_VERSION
- NODE
- LINK
- PROJECT_MEMBER

## 3. カラム命名

カラム名は英小文字スネークケースを基本とする。

例:

- project_id
- flow_version_id
- display_order
- is_deleted
- created_at

## 4. PK命名

PKは `{table_singular}_id` を基本とする。

例:

- project_id
- flow_id
- node_id

## 5. FK命名

FKカラムは参照先PK名と同じ名称を使う。

例:

- project_id
- flow_id
- flow_version_id

## 6. Index命名

Indexは `idx_{table}_{purpose}` とする。

例:

- idx_flow_project_deleted_order
- idx_node_version_lane_stage

## 7. Unique命名

Unique制約は `uq_{table}_{columns}` とする。

## 8. FK制約命名

FK制約は `fk_{child}_{parent}_{column}` とする。

## 9. Migration命名

Migration名は英語PascalCaseで目的を示す。

例:

- InitialCreate
- AddTemplateTables
- AddExportHistory

## 10. Boolean命名

Boolean相当は `is_` または `can_` を使う。

例:

- is_deleted
- is_latest
- can_export

## 11. 日時命名

日時は `_at` で終える。

例:

- created_at
- updated_at
- deleted_at
- exported_at

## 12. ユーザー参照命名

ユーザーIDを保持する場合は `_by` で終える。

例:

- created_by
- updated_by
- deleted_by

## 13. テスト観点

- テーブル名が統一されていること
- カラム名がスネークケースであること
- FK名が参照先と一致すること
- Index名が用途を表すこと

## 14. 完了条件

DB設計、Entity、Migration、SQLが本命名規則に従っていること。
