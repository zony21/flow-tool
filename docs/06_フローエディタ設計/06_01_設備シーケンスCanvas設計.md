# 06_01_設備シーケンスCanvas設計

## 1. 本書の目的

本書は、AI Flow Designer の Flow Editor における Canvas 表示・ノード配置・設備列・工程分類の設計方針を定義する。

AI Flow Designer は汎用的な自由配置フローチャートツールではない。

目的は、WCS / PLC / AGF / 作業者 / 外部システムなど、設備・システム間の責務と処理連携を、人間とAIが同じ粒度で理解できるシステムフローとして作成することである。

## 2. 基本思想

Flow Editor は、設備シーケンス型Canvasとして扱う。

横方向は設備・システムを表す。

縦方向は人間が見やすく整理するための処理配置を表す。

処理順そのものは座標ではなく Link で表す。

```text
横方向 = 設備・システム
縦方向 = 見た目上の処理配置
処理順 = Link
```

## 3. 用語定義

|内部名|画面表示|意味|
|---|---|---|
|Stage|設備|包装機PLC、AGF、WCS、RCS、作業者などの処理主体|
|Lane|工程分類|RFID読取り、包装開始、搬送処理、異常処理などの分類帯|
|Node|図形|処理、判定、開始、終了、帳票・データなど|
|Link|接続線|処理順、通信、制御、依存関係|

画面上では Stage / Lane / Node / Link という英単語を原則表示しない。

ただし、実装上の型名・API DTO・DB項目名では既存構造との整合性を優先し、内部名を維持してよい。

## 4. SSOTとの関係

SSOTとして意味を持つのは以下である。

|意味|正となるデータ|
|---|---|
|処理主体|Node.stageId|
|工程分類|Node.laneId|
|処理内容|Node.name / Node.description / Node.nodeType|
|処理順|Link.sourceNodeId / Link.targetNodeId|

Node.x / Node.y は View Hint である。

AI DSL、処理順判定、責務判定、仕様正当性判定に Node.x / Node.y を使ってはならない。

## 5. Canvas構造

Canvas は以下の構造で表示する。

```text
┌──────────┬──────────┬──────────┬──────────┐
│工程分類  │設備A      │設備B      │設備C      │
├──────────┼──────────┼──────────┼──────────┤
│RFID読取  │          │          │          │
├──────────┼──────────┼──────────┼──────────┤
│包装開始  │          │          │          │
├──────────┼──────────┼──────────┼──────────┤
│搬送処理  │          │          │          │
└──────────┴──────────┴──────────┴──────────┘
```

左列は工程分類を表す。

上列は設備を表す。

図形は必ずいずれかの設備列に所属する。

## 6. 設備列仕様

設備列は Stage から生成する。

Stage は横方向に sortOrder 昇順で並べる。

設備列の表示例は以下とする。

```text
┌──────────┐
│  設      │
│ 包装機PLC│
│ 設備     │
└──────────┘
```

将来的には Stage に設備アイコン、画像、色を持たせてもよい。

現時点ではDB変更を必須とせず、先頭文字アイコンなどの簡易表現でよい。

## 7. 工程分類仕様

工程分類は Lane から生成する。

Lane は縦方向の分類帯として sortOrder 昇順で並べる。

工程分類は処理順そのものではない。

処理順は Link が正である。

工程分類は人間がフローを読みやすくするための分類であり、例は以下とする。

- RFID読取り
- 包装開始
- 搬送処理
- 異常処理
- 完了処理

## 8. ノード配置仕様

ノードの横位置は設備列に固定する。

ノードの縦位置はユーザーが配置・調整できる。

```text
X = stageId から算出
Y = node.y を使用
```

Node.x は保存してもよいが、表示時は stageId を優先してX座標を再算出する。

これにより、設備列とノードの横位置を常に一致させる。

## 9. ノード追加仕様

ノード追加は図形パレットからCanvasへのドラッグ&ドロップを基本操作とする。

ドロップ時の判定は以下とする。

|項目|判定|
|---|---|
|stageId|ドロップ位置の設備列|
|laneId|ドロップ位置の工程分類帯|
|x|stageId から再算出|
|y|ドロップ位置のY座標|

設備列外へドロップされた場合、ノードは追加しない。

## 10. ノード追加時のY配置ルール

シーケンスフローでは、Y方向は処理の読みやすさに強く関わる。

そのため、設備ごとに単純に上から詰める配置は禁止する。

### 10.1 同じ設備に追加する場合

同じ設備の既存ノードを確認し、重なる場合は既存ノードの下にずらす。

```text
設備A
  処理1
  処理2
  処理3 ← 追加
```

### 10.2 別設備に追加する場合

別設備であれば、既存ノードと同じY軸への配置を許可する。

これは設備間通信や並行処理を表現するために必要である。

```text
設備A        設備B
  処理1       受信1
```

同じY軸に配置できることにより、Excelで作成する設備別システムフローに近い見た目を実現する。

### 10.3 既存ノードより上への配置

ユーザーが明示的に上部へドロップした場合は許可してよい。

ただし、既存ノードと重なる場合は同一設備内のみY方向に補正する。

## 11. ノード移動仕様

ノード移動時は以下とする。

|移動方向|仕様|
|---|---|
|横方向|最寄りの設備列へ補正し、stageId を更新する|
|縦方向|ユーザー操作を維持し、node.y を更新する|

横方向は完全自由配置にしない。

縦方向は20px程度の軽いスナップを許可する。

## 12. ズーム・パン方針

現状のCanvasでは、工程分類・設備グリッドは通常DOMで描画し、ノードはVue Flowで描画する。

この構造でVue Flowだけズーム・パンすると、ノードだけが移動し、設備列・工程分類とズレる。

そのため、当面はVue Flowのズーム・パンを無効とする。

ズーム・パンを再導入する場合は、以下のいずれかを満たす必要がある。

1. 工程分類・設備グリッドもVue Flow内の同一座標系で描画する
2. Vue Flowのtransform値を取得し、外部DOMのグリッドへ同じtransformを適用する

この対応なしにズーム・パンを有効化してはならない。

## 13. 接続線仕様

接続線は Link から生成する。

接続線は処理順・通信・制御関係を表す。

同一設備内の縦接続、別設備間の横接続の両方を許可する。

将来的には以下を検討する。

- 直線
- 折れ線
- 矢印方向の明確化
- ラベル表示
- 条件分岐ラベル

## 14. 図形種別

nodeType は以下を基本とする。

|nodeType|画面表示|用途|
|---|---|---|
|start|開始|処理開始点|
|process|処理|通常処理|
|decision|判定|条件分岐|
|document|帳票・データ|帳票、ファイル、データ入出力|
|wait|待機|待機、保留、応答待ち|
|end|終了|処理終了点|

図形内に nodeType 名は表示しない。

図形内はノード名のみ表示する。

## 15. 画面構成

Flow Editor画面は以下の構成とする。

```text
上部: フロー名、保存、出力、バージョン操作
左側: 図形パレット
中央: 設備シーケンスCanvas
右側: 選択時のみ図形詳細または接続線詳細
Dialog: 設備/分類設定
```

設備/分類設定は常時表示しない。

図形詳細・接続線詳細は未選択時に表示しない。

## 16. 実装上の注意

FlowCanvas の座標定数は、EditorStore の追加・移動処理と大きく乖離させてはならない。

特に以下は整合させる。

- stageWidth
- nodeOffsetX
- rowHeight
- nodeRowOffset
- nodeSpacingY

将来的にはこれらを共通定数化することが望ましい。

## 17. 完了条件

- ノードが必ず設備列上に表示される
- 別設備であれば同じY軸にノードを置ける
- 同じ設備内では重なりを避けられる
- 工程分類と設備ヘッダーが重ならない
- Vue Flowのズーム・パンで設備列とノードがズレない
- 図形パレットからドラッグ&ドロップで追加できる
- ノード選択時のみ図形詳細が表示される
