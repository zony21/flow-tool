# 10_14_CICD方針

## 1. 目的

本書は、AI Flow DesignerにおけるCI/CD方針を定義する。

本リポジトリでは、当面mainへ直接反映する運用とする。
ただし、mainへ直接反映する場合でも、将来の自動テスト・自動ビルド・品質確認に耐える構成を前提とする。

## 2. 基本方針

- mainを正とする。
- ブランチ運用は行わない前提で設計する。
- 将来GitHub Actionsで自動検証できるようにする。
- Frontend / Backend / Docsの検証を分ける。
- 設計書と実装の対応関係を崩さない。

## 3. CI対象

| 対象 | 内容 |
| --- | --- |
| Frontend | install / typecheck / build / test |
| Backend | restore / build / test |
| Docs | Markdown構成確認 / リンク確認 |
| Export | Mermaid / PDF / AI DSL変換テスト |

## 4. Frontend CI

想定処理:

```text
npm ci
npm run typecheck
npm run build
npm run test
```

確認観点:

- TypeScript型エラーがない。
- Vue buildが通る。
- Store / AdapterのUnit Testが通る。
- Lint導入後はLintも通る。

## 5. Backend CI

想定処理:

```text
dotnet restore
dotnet build
dotnet test
```

確認観点:

- .NET 8でBuildできる。
- Domain / Application / Repository Testが通る。
- SQLiteを使ったIntegration Testが通る。
- Migrationの破綻がない。

## 6. Docs CI

Docsは実装品質の前提である。

確認観点:

- READMEの索引とdocsフォルダが一致している。
- draft / temp / old / 作業メモ系ファイルが残っていない。
- 章番号とファイル番号が一致している。
- Markdownとして最低限壊れていない。

## 7. Export CI

Export機能はAI Flow Designerの中核である。

確認観点:

- サンプルFlowからMermaidが生成できる。
- サンプルFlowからAI DSLが生成できる。
- サンプルFlowからPDF Document Modelが生成できる。
- SnapshotからExportできる。

## 8. CD方針

初期段階ではCDは必須としない。

将来以下を検討する。

- Frontend静的ホスティング
- Backend APIデプロイ
- DB Migration自動適用
- Preview環境
- Release Tag作成

## 9. main直接反映時の注意

mainへ直接反映するため、以下を徹底する。

- 反映単位を小さくする。
- READMEとdocs索引を同時に更新する。
- 不要ファイルを残さない。
- 設計書は薄いメモではなく正式文書として追加する。
- 完了していない章は「詳細化済み」と記載しない。

## 10. 完了条件

- 将来GitHub Actionsへ移行できる検証単位が定義されている。
- Frontend / Backend / Docs / ExportのCI対象が分離されている。
- main直接反映運用でも品質を崩さないルールが定義されている。
