# 10_08_DBアクセス方針

## 1. 目的

本書は、AI Flow DesignerのDBアクセス方針を定義する。

初期DBはSQLiteとするが、将来SQL Serverへ移行する。
そのため、DBアクセスはEntity Framework Coreを中心に構成し、SQLite固有の実装がApplication LayerやDomain Layerへ漏れないようにする。

## 2. 採用技術

- Entity Framework Core
- SQLite
- 将来SQL Server
- ASP.NET Core .NET 8

## 3. 基本方針

- DBアクセスはInfrastructure層に集約する。
- ControllerからDbContextを直接操作しない。
- Application LayerはRepository Interfaceを経由する。
- Domain LayerはEF Coreに依存しない。
- SQLite固有SQLを上位層に書かない。
- MigrationはDB変更履歴として管理する。

## 4. DbContext方針

DbContextはInfrastructure層に配置する。

例:

```text
FlowDesigner.Infrastructure/Data/FlowDesignerDbContext.cs
```

DbSet対象:

- Projects
- Flows
- Lanes
- Stages
- Nodes
- Links
- Comments
- FlowVersionSnapshots
- Templates
- Images
- Users
- Settings

## 5. Entity方針

EF EntityはDB永続化用の構造である。
Domain Modelと完全一致させる必要はない。

理由:

- DBの正規化都合とDomainの扱いやすさは異なるため。
- SnapshotはJSON文字列として保持する可能性があるため。
- SQLiteとSQL Serverで細かな型差異が出るため。

変換方針:

```text
Domain Model
  ↓ Mapper
EF Entity
  ↓ EF Core
Database
```

取得時:

```text
Database
  ↓ EF Core
EF Entity
  ↓ Mapper
Domain Model
```

## 6. Repository方針

Repository InterfaceはApplicationまたはDomain寄りに配置し、実装はInfrastructureに配置する。

例:

```text
FlowDesigner.Application/Repositories/IFlowRepository.cs
FlowDesigner.Infrastructure/Repositories/FlowRepository.cs
```

Repositoryの責務:

- Flow取得
- Flow保存
- Project取得
- Snapshot保存
- Template取得
- Import保存

Repositoryの禁止事項:

- HTTP Request / Response DTOを扱う。
- Controller向け表示整形を行う。
- Export変換を行う。
- 業務検証をすべてRepositoryに押し込む。

## 7. Transaction方針

以下の処理はTransactionを使用する。

- Flow保存
- Lane / Stage削除とNode移動
- Flow削除
- Template適用
- Snapshot作成
- Import実行

Flow保存では、Node / Link / Commentなど複数テーブルを更新するため、一部だけ保存される状態を許可しない。

## 8. Snapshot保存方針

Version Snapshotは、保存時点のFlow構造を再現できる形式で保持する。

保持方式は以下を想定する。

- Snapshot Header Table
- Snapshot Body JSON

理由:

- Snapshotは履歴再現が目的であり、最新Flowと同じ正規化構造へ分解すると再現性が下がるため。
- 将来の構造変更時も、当時のJSONを保持できるため。

ただし、検索や一覧に必要な項目はHeaderへ保持する。

## 9. SQLiteとSQL Serverの差異吸収

差異が出やすい項目:

- 日時型
- Boolean型
- JSON保持
- 自動採番
- 文字列照合順
- 外部キー制約
- Transaction挙動

対応方針:

- DateTimeはUTC保存を原則とする。
- BooleanはEF Coreの変換に寄せる。
- JSONは文字列列として扱い、Application側でSchema検証する。
- IDはGUIDまたはアプリケーション側採番を優先する。
- DB固有関数に依存しない。

## 10. Migration方針

MigrationはDB変更履歴として管理する。

原則:

- 手動SQLだけでDB変更しない。
- Migration名は変更内容が分かる名前にする。
- SQLite初期MigrationとSQL Server移行時Migrationの差異を管理する。
- 本番相当データが入った後は破壊的変更を避ける。

## 11. テスト観点

- Flow保存時に全関連テーブルが保存される。
- Linkが存在しないNodeを参照できない。
- Lane削除時に孤立Nodeが残らない。
- Snapshot作成後に元Flowを変更してもSnapshot内容が変わらない。
- SQLiteで動作するRepositoryがSQL Server移行時もApplication層に影響を出さない。

## 12. 完了条件

- DBアクセスがInfrastructure層に閉じている。
- ControllerからDbContextを直接呼ばない設計になっている。
- Repository Interface経由でApplication LayerがDBを扱う。
- Snapshotの再現性が担保されている。
- SQLiteからSQL Serverへの移行影響が限定されている。
