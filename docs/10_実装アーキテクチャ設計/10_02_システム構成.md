# 10_02_システム構成

## 1. 目的

本書は、AI Flow Designerのシステム構成を定義する。

本システムは、Vue3によるWebフロントエンド、ASP.NET Core .NET 8によるバックエンドAPI、EF CoreによるDBアクセス、SQLiteによる初期永続化で構成する。

将来的にはSQL Serverへ移行できる構成とする。

## 2. 全体構成

```text
[Browser]
  └─ Vue3 / TypeScript / PrimeVue / Vue Flow / Pinia
       |
       | HTTPS / JSON API
       v
[ASP.NET Core API]
  ├─ Controllers
  ├─ Application Services
  ├─ Domain Models
  ├─ Validators
  ├─ Export Services
  └─ Infrastructure
       |
       | EF Core
       v
[Database]
  ├─ SQLite 初期
  └─ SQL Server 将来
```

## 3. Frontend構成

Frontendは、ユーザーがFlowを編集するためのUIを提供する。

主な責務:

- Project選択
- Flow選択
- Lane / Stage / Node / Link / Comment編集
- Vue FlowによるCanvas表示
- Property Panelによる詳細編集
- Undo / Redo
- 一時保存状態表示
- API呼び出し
- Export要求
- AIレビュー要求

Frontendは表示用モデルを持つが、保存時にはAPIのRequest DTOへ変換する。
Vue Flowの内部Node / EdgeをそのままSSOTとして扱わない。

## 4. Backend構成

Backendは、SSOTの正当性を守る中心である。

主な責務:

- GitHub OAuth認証
- Project / Flow / Lane / Stage / Node / Link / Comment API
- 保存時バリデーション
- Version Snapshot作成
- Mermaid出力
- PDF出力
- JSON出力
- AI DSL出力
- Import検証
- 将来のAIレビュー連携

Backendは、Frontendから受け取ったデータをそのままDBに入れない。
必ずApplication Serviceで検証し、Domain Modelへ変換してから永続化する。

## 5. Database構成

初期DBはSQLiteとする。

DBは以下を保持する。

- Project
- Flow
- Lane
- Stage
- Node
- Link
- Comment
- Version Snapshot
- Template
- Image Metadata
- User / Auth関連
- Setting

将来SQL Serverへ移行するため、DB固有のSQLや型はInfrastructure層に閉じ込める。

## 6. 出力処理構成

出力処理はBackend側に集約する。

理由:

- SSOTを正として安定出力するため。
- ブラウザ依存の印刷崩れを避けるため。
- Version Snapshotから再出力できるようにするため。
- 将来バッチ出力や共有URL出力へ拡張しやすくするため。

出力処理の入力は以下とする。

- 保存済みFlow
- Version Snapshot
- Export Request Option

出力処理の出力は以下とする。

- Mermaid text
- PDF file
- JSON file
- AI DSL text

## 7. 通信方式

FrontendとBackendはJSON APIで通信する。

原則:

- Request / Response DTOを定義する。
- Domain EntityをそのままAPIへ露出しない。
- 画面表示用DTOと保存用DTOを分離する。
- Export要求は同期処理から開始し、重くなった場合は非同期ジョブ化できる設計とする。

## 8. 完了条件

- Browser / API / DBの構成が明確である。
- FrontendとBackendの責務が分離されている。
- 出力処理がBackend側に集約されている。
- SQLite初期構成とSQL Server将来構成が矛盾しない。
