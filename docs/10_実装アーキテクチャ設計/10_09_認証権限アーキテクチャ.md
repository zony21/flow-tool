# 10_09_認証権限アーキテクチャ

## 1. 目的

本書は、AI Flow Designerにおける認証・権限アーキテクチャを定義する。

本システムはGitHub OAuthを採用する。
Project、Flow、Version Snapshot、Template、Export、AIレビューなどの操作は、ユーザー単位またはProject単位で権限管理する。

## 2. 採用方針

- 認証方式はGitHub OAuthとする。
- Backendで認証状態を確定する。
- Frontendのログイン状態は表示制御用であり、権限判定の正ではない。
- API実行時はBackendで必ず権限確認する。
- Project単位の権限を基本とする。

## 3. 認証フロー

```text
User
  ↓ Login Click
Frontend
  ↓ Redirect
GitHub OAuth
  ↓ Callback
Backend Auth Endpoint
  ↓ User resolve / create
Session or Token Issue
  ↓
Frontend Auth Store Update
```

## 4. Backend責務

Backendは以下を担当する。

- GitHub OAuth Callback受信
- GitHub userId / login取得
- App User作成または更新
- Session / Token発行
- APIごとの認証確認
- Projectごとの権限確認

## 5. Frontend責務

Frontendは以下を担当する。

- ログインボタン表示
- ログアウト操作
- ログインユーザー表示
- 権限に応じたボタン表示制御
- API 401 / 403時の画面誘導

Frontendだけで権限確定しない。

## 6. 権限種別

| 権限 | 内容 |
| --- | --- |
| Owner | Project作成者。全操作可能 |
| Editor | Flow編集、保存、Snapshot作成可能 |
| Viewer | 閲覧、Export可能 |
| Exporter | Exportのみ可能。将来拡張 |

初期実装ではOwner中心でもよいが、DB設計とAPI設計は将来拡張を妨げない形にする。

## 7. 操作別権限

| 操作 | 必要権限 |
| --- | --- |
| Project作成 | ログイン済み |
| Project更新 | Owner |
| Flow作成 | Owner / Editor |
| Flow編集 | Owner / Editor |
| Flow閲覧 | Owner / Editor / Viewer |
| Snapshot作成 | Owner / Editor |
| Export | Owner / Editor / Viewer |
| Template登録 | Owner / Editor |
| 権限変更 | Owner |

## 8. API権限判定

API LayerまたはApplication Layerで以下を確認する。

- 認証済みか。
- 対象Projectが存在するか。
- 対象Projectに対して必要権限を持つか。
- FlowがProject配下に属しているか。
- Snapshotが対象Flowのものか。

## 9. DB保持情報

想定テーブル:

- USER
- PROJECT_MEMBER
- AUTH_PROVIDER_ACCOUNT

保持項目例:

- userId
- githubUserId
- githubLogin
- displayName
- avatarUrl
- projectId
- role
- createdAt
- updatedAt

## 10. セキュリティ観点

- GitHub OAuthの戻り先URLを固定する。
- StateパラメータでCSRFを防止する。
- 権限チェックをFrontendだけに依存しない。
- Export URLを発行する場合は期限または権限確認を行う。
- Viewerが編集APIを実行してもBackendで拒否する。

## 11. 完了条件

- GitHub OAuthの責務境界が明確である。
- Backendで認証・権限判定する設計になっている。
- Project単位の権限拡張が可能である。
- Frontend表示制御とBackend権限判定が分離されている。
