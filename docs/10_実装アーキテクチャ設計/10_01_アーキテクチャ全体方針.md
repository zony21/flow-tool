# 10_01_アーキテクチャ全体方針

## 1. 目的

本書は、AI Flow Designer全体の実装アーキテクチャ方針を定義する。

AI Flow Designerは、画面上の図形を保存するツールではない。
Project / Flow / Lane / Stage / Node / Link / Comment を中心とするSSOTを保存し、そこから表示、編集、出力、AIレビュー、将来のコード生成へ展開する設計ツールである。

そのため、アーキテクチャ上の最重要方針は、見た目と構造化データを分離し、構造化データを正として扱うことである。

## 2. 採用技術

| 区分 | 技術 |
| --- | --- |
| Frontend | Vue3 / TypeScript / PrimeVue / Vue Flow / Pinia / Vite |
| Backend | ASP.NET Core .NET 8 / Entity Framework Core |
| Database | SQLite、将来SQL Server |
| Auth | GitHub OAuth |
| Output | Mermaid / PDF / JSON / AI DSL |

## 3. 基本方針

- Frontendはユーザー操作、キャンバス編集、入力補助、状態表示を担当する。
- Backendは保存、検証、履歴、Export、認証、権限、将来のAIレビュー連携を担当する。
- DBはSSOTとVersion Snapshotを永続化する。
- Mermaid、PDF、AI DSL、JSONはすべてSSOTまたはSnapshotから生成する。
- Canvasの座標や見た目は重要だが、業務意味・責務・工程・接続条件より優先しない。
- Undo / Redoは画面操作履歴、Version Snapshotは保存版管理として分離する。

## 4. アーキテクチャ原則

### 4.1 SSOT優先

図形描画結果ではなく、保存済み構造化データを正とする。

誤った実装例:

- Canvas上のNode位置だけを見てMermaidを生成する。
- 表示用のVue Flow nodeをそのままDBへ保存する。
- PDF出力時に画面DOMをそのまま印刷する。

正しい実装例:

- Flow構造からVue Flow表示モデルを生成する。
- Flow構造からMermaidを生成する。
- Flow構造またはSnapshotからPDF用Document Modelを生成する。
- Flow構造からAI DSLを生成する。

### 4.2 派生モデルの分離

Frontend表示、Mermaid、PDF、AI DSL、API生成、DB更新一覧は、それぞれ用途が異なる派生モデルである。

派生モデルは保存の正ではない。
保存の正は、Project / Flow / Lane / Stage / Node / Link / Comment と、そのVersion Snapshotである。

### 4.3 Backend検証

Frontendで入力制御を行っても、Backendで必ず構造検証を行う。

理由:

- API直接呼び出しでも不正データが入らないようにするため。
- 将来Import機能で外部データを取り込むため。
- AIレビューやコード生成の前提品質を担保するため。

### 4.4 将来移行前提

初期DBはSQLiteとするが、将来SQL Serverへ移行する。

そのため、Repository層より上位ではSQLite固有SQLに依存しない。
EF Coreで表現できないDB固有処理は、Infrastructure層に閉じ込める。

## 5. 完了条件

- Frontend / Backend / DB / Export の責務境界が明確である。
- SSOTと派生データの境界が明確である。
- SQLiteからSQL Serverへ移行する際の影響範囲が限定されている。
- Mermaid、PDF、AI DSLが共通入力から生成される設計になっている。
- AIだけが本書を読んでも実装構成の大枠を誤解しない。
