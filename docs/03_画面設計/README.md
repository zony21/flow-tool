# 03_画面設計

## 1. 本書の位置付け

本ファイルは、`03_画面設計` 配下の正式設計書の索引である。
過去版の短文メモ、更新メモ、前版維持のみの内容は正式設計書として扱わない。

## 2. 画面設計の目的

AI Flow Designer の画面設計では、利用者が視覚的にフローを作成できるだけでなく、AIが誤解しない構造化データを正しく入力・編集・確認できることを目的とする。

画面は以下を実現する。

- プロジェクトを管理できること
- フローを管理できること
- レーン、工程、ノード、接続、コメントを編集できること
- 構造化データとして保存できること
- Mermaid、PDF、JSON、将来AI DSLなどの出力へ接続できること

## 5. 不要ファイル削除ルール

`03_画面設計` 配下には正式設計書のみを残す。
以下のようなファイルは、現在存在するものについては古いファイルと判断し、原則として削除対象とする。

| 削除対象パターン | 削除理由 |
| --- | --- |
| `03_更新.md` | 更新履歴だけのメモであり、画面仕様・API・DB・テスト観点を持たないため |
| `03_更新概要.md` | 更新概要だけで、実装判断に使える正式設計書ではないため |
| `03_01_画面設計_概要.md` | `03_01_画面設計方針.md` に統合されるべき旧概要ファイルであり、役割が重複するため |
| `03_01_メイン画面設計.md` | メイン画面という粒度が曖昧で、現在の画面一覧・フローエディタ画面設計と対応しないため |
| 1〜2行だけの旧ファイル | 目的・レイアウト・項目・操作・API・DB・テスト観点がなく、正式設計書として利用できないため |
| `README.md` が短文または前版維持のみの場合 | 索引として機能せず、正式設計書の入口にならないため |

## 6. 残すファイルの条件

以下を満たすファイルは正式設計書として残す。

- ファイル名が画面一覧または設計対象と対応していること
- 本書の目的が明記されていること
- レイアウト仕様があること
- 表示項目またはUI部品仕様があること
- 操作仕様があること
- バリデーションまたは制約事項があること
- 関連APIが示されていること
- 関連DBが示されていること
- テスト観点があること
- 完了条件があること

## 7. 今後の改修への適用

今後、`03_画面設計` を改修する際は、まずファイル名と内容を確認する。
更新メモ、概要だけの旧ファイル、重複ファイルは正式設計書へ統合済みかを確認し、統合済みであれば削除する。

新しく増えたファイルについては、ファイル名だけで機械的に削除せず、内容を確認して削除・統合・正式化のいずれかを判断する。
