# 03_16_レスポンシブ設計

## 1. 本書の目的

本書は、AI Flow Designer のレスポンシブ設計を定義する。
本システムはPCでの設計作業を主用途とするが、画面幅が変わっても主要情報が破綻しないことを目的とする。

## 2. 画面概要

フローエディタは大画面利用を前提とする。
スマートフォンでの本格編集は初期対象外とし、参照または簡易確認を中心とする。

## 3. 利用者

- システム設計者
- レビュー担当者
- 実装者

## 4. 権限

レスポンシブ設計によって権限は変わらない。
ただし、狭い画面では編集機能の一部を非表示または利用不可にする可能性がある。

## 5. レイアウト

| 幅 | 方針 |
| --- | --- |
| 1200px以上 | 標準編集レイアウト |
| 900〜1199px | パネル折りたたみ対応 |
| 600〜899px | 参照中心、編集制限あり |
| 599px以下 | 本格編集非推奨、一覧・参照中心 |

## 6. 表示項目

画面幅が狭い場合は、以下を優先して表示する。

- 画面タイトル
- 主要一覧
- 保存状態
- エラー表示
- 主要操作

## 7. 入力項目

狭い画面では入力項目を縦積みにする。
フローエディタのプロパティパネルはDrawer表示に切り替える。

## 8. ボタン

ツールバーのボタンは幅に応じてアイコン化、またはメニュー内へ格納する。
破壊的操作は常にラベルを表示する。

## 9. ダイアログ

小さい画面ではダイアログ幅を画面幅に合わせる。
スクロール可能にし、実行ボタンが見切れないようにする。

## 10. バリデーション

レスポンシブ時もバリデーション表示位置を維持する。
項目下のエラーが画面外に隠れないようにする。

## 11. API

レスポンシブ設計によってAPIは変わらない。

## 12. DB

レスポンシブ設計によってDBは変わらない。

## 13. 状態遷移

画面幅変更時、選択状態や未保存状態は維持する。
パネル折りたたみで入力内容が失われてはならない。

## 14. エラー処理

画面幅が狭い場合でもToast、Dialog、InlineErrorを確認できること。

## 15. 未保存データ

画面幅変更やパネル開閉で未保存データを破棄しない。

## 16. テスト観点

- 1200px以上で標準レイアウトになること
- 900px程度でパネルを折りたためること
- 小画面でダイアログが見切れないこと
- 幅変更後も選択状態が維持されること
- 未保存データが消えないこと

## 17. 完了条件

本仕様は、主要画面が幅変更時に破綻せず、編集状態を維持できる方針が定義された場合に完了とする。
