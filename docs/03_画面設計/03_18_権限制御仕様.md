# 03_18_権限制御仕様

## 1. 本書の目的

本書は、AI Flow Designer の画面における権限制御仕様を定義する。
権限制御は、利用者が参照・作成・更新・削除・出力・管理操作を行えるかを画面上で一貫して制御するために使用する。

## 2. 画面概要

初期実装ではGitHub OAuthでログインしたユーザーを利用者として扱う。
将来的にはロール、プロジェクト単位権限、Flow単位権限を追加する。

## 3. 利用者

- 参照者
- 編集者
- 管理者
- テンプレート管理者
- 出力権限ユーザー

## 4. 権限

| 権限 | 内容 |
| --- | --- |
| View | 参照可能 |
| Edit | 編集可能 |
| Delete | 削除可能 |
| Export | 出力可能 |
| ManageTemplate | テンプレート管理可能 |
| Admin | 管理操作可能 |

## 5. レイアウト

権限がない操作は以下のいずれかで表現する。

- ボタン非表示
- ボタン非活性
- 読み取り専用表示
- 権限なしメッセージ表示

破壊的操作は非表示よりも非活性＋理由表示を優先する。

## 6. 表示項目

画面ごとに以下を制御する。

- 新規作成ボタン
- 編集ボタン
- 削除ボタン
- 保存ボタン
- 出力ボタン
- テンプレート管理ボタン

## 7. 入力項目

参照のみ権限では入力項目をreadonlyにする。
編集権限がない場合、値を変更できない。

## 8. ボタン

| ボタン | 必要権限 |
| --- | --- |
| 新規作成 | Edit |
| 保存 | Edit |
| 削除 | Delete |
| 出力 | Export |
| テンプレート作成 | ManageTemplate |
| 管理画面 | Admin |

## 9. ダイアログ

権限がない場合、作成・更新・削除系ダイアログを表示しない。
操作直前で権限不足が判明した場合はErrorDialogまたはToastで通知する。

## 10. バリデーション

画面側の権限制御だけに依存しない。
API側でも必ず権限チェックを行う。

## 11. API

- GET /api/auth/me
- GET /api/projects/{projectId}/permissions
- GET /api/flows/{flowId}/permissions

各更新APIでも権限チェックを行う。

## 12. DB

- USER
- ROLE
- USER_ROLE
- PROJECT_PERMISSION
- FLOW_PERMISSION

初期実装では一部テーブルを将来拡張として扱ってよい。

## 13. 状態遷移

```text
画面表示
  ↓
権限取得
  ├ Viewのみ
  ├ Edit可能
  ├ Export可能
  └ Admin
```

## 14. エラー処理

- 操作権限がありません
- このプロジェクトを編集できません
- このフローを出力できません
- 管理者権限が必要です

## 15. 未保存データ

編集中に権限が失効した場合、保存を停止し、現在の編集内容を破棄せず権限エラーを表示する。

## 16. テスト観点

- Viewのみで編集ボタンが使えないこと
- Edit権限で保存できること
- Delete権限なしで削除できないこと
- API側でも403になること
- 権限取得失敗時に安全側へ倒れること

## 17. 完了条件

本仕様は、画面とAPIの両方で権限制御が前提化され、ユーザー権限に応じたUI制御が定義された場合に完了とする。
