# 03_03_プロジェクト一覧画面

## 1. 本書の目的

本書は、プロジェクト一覧画面の詳細仕様を定義する。
プロジェクト一覧画面は、AI Flow Designer にログインしたユーザーが最初に操作する主要画面であり、設計対象となる大項目を管理する入口である。

## 2. 画面の役割

プロジェクトは複数のFlowを束ねる大項目である。
例として、包装ライン制御、物流センター、WCS開発などをプロジェクトとして管理する。

本画面では以下を行う。

- プロジェクト一覧表示
- プロジェクト新規作成
- プロジェクト編集
- プロジェクト削除
- フロー一覧画面への遷移

## 3. レイアウト

画面構成は以下とする。

```text
[ヘッダー]
[画面タイトル: プロジェクト一覧] [新規作成ボタン]
[検索・絞り込み領域]
[プロジェクト一覧テーブル]
[ページング]
```

## 4. 表示項目

一覧には以下を表示する。

| 項目 | 内容 | 備考 |
| --- | --- | --- |
| プロジェクト名 | プロジェクトの表示名 | 必須 |
| 説明 | プロジェクト説明 | 任意 |
| フロー数 | 配下Flow件数 | 自動集計 |
| 作成者 | 作成ユーザー | GitHubユーザー |
| 更新日時 | 最終更新日時 | 降順ソート標準 |
| 操作 | 詳細、フロー一覧、削除 | ボタン表示 |

## 5. 操作仕様

### 5.1 新規作成

新規作成ボタン押下でプロジェクト作成ダイアログを表示する。
入力項目は以下とする。

- プロジェクト名
- 説明
- 表示順

プロジェクト名は必須である。

### 5.2 編集

詳細ボタン押下でプロジェクト詳細画面または編集ダイアログを表示する。
初期実装ではダイアログでもよいが、将来的に詳細画面へ拡張できる設計とする。

### 5.3 フロー一覧遷移

フロー一覧ボタン押下で、対象プロジェクトのフロー一覧画面へ遷移する。
URLにはProject IDを含める。

### 5.4 削除

削除ボタン押下で確認ダイアログを表示する。
プロジェクト配下にFlowが存在する場合は、削除可否をDB/API設計に従って制御する。
初期方針では論理削除を基本とする。

## 6. バリデーション

| 項目 | 条件 |
| --- | --- |
| プロジェクト名 | 必須、最大100文字 |
| 説明 | 最大1000文字 |
| 表示順 | 数値、0以上 |

API側でも同じバリデーションを行う。

## 7. 状態管理

Piniaで以下を管理する。

- プロジェクト一覧
- 選択中プロジェクト
- 読込中状態
- 保存中状態
- エラー状態
- 検索条件

## 8. API連携

使用APIは以下とする。

- GET /api/projects
- POST /api/projects
- GET /api/projects/{projectId}
- PUT /api/projects/{projectId}
- DELETE /api/projects/{projectId}

## 9. DB連携

主にPROJECTテーブルを使用する。
フロー数表示ではFLOWテーブルを参照する。

## 10. エラー表示

想定エラーは以下とする。

- プロジェクト一覧の取得に失敗しました
- プロジェクトの作成に失敗しました
- プロジェクトの更新に失敗しました
- プロジェクトの削除に失敗しました
- ログイン状態が切れました

## 11. 権限制御

初期実装ではログイン済みユーザーが操作可能とする。
将来的には以下を制御する。

- 参照のみの場合、新規作成、編集、削除ボタンを非表示または非活性
- 管理者のみ削除可能
- プロジェクト単位の権限により一覧表示を制御

## 12. テスト観点

- プロジェクト一覧が表示されること
- 新規作成できること
- 必須未入力時にエラーが出ること
- 編集内容が保存されること
- フロー一覧へ遷移できること
- 削除確認ダイアログが表示されること
- APIエラー時にエラーメッセージが表示されること

## 13. 完了条件

本画面は、プロジェクトの一覧、作成、更新、削除、フロー一覧遷移が実装可能な粒度で定義され、API・DB・テスト仕様と対応している場合に完了とする。
