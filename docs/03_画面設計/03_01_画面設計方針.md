# 03_01_画面設計方針

## 1. 本書の目的

本書は、AI Flow Designer の画面設計における共通方針を定義する。
本システムは単なる作図ツールではなく、Project、Flow、Lane、Stage、Node、Link、Comment で構成される構造化データを正として管理する設計ツールである。
そのため、画面は見た目の編集だけでなく、AIが理解可能な構造化データを正しく作成・更新・確認できることを目的とする。

## 2. 画面設計の基本思想

画面設計では以下を重視する。

- 人が見て分かりやすいこと
- AIが誤解しないデータを入力できること
- ExcelやVisioに近い感覚で操作できること
- 図形と意味情報が常に対応していること
- 編集結果が構造化データとして保存されること
- 設計書・Mermaid・PDF・JSON出力へつながること

## 3. 採用UI方針

フロントエンドは Vue3、TypeScript、PrimeVue、Vue Flow、Pinia、Vite を使用する。
PrimeVueは入力部品、ダイアログ、ボタン、テーブル、メニューに使用する。
Vue Flowはキャンバス上のノード描画、接続線描画、ドラッグ、選択、ズーム、パンに使用する。
Piniaは画面状態、選択状態、編集中データ、Undo/Redo状態、保存状態を管理する。

## 4. レイアウト原則

画面は以下の原則で設計する。

- 左から右、上から下へ自然に作業できること
- 一覧画面は検索、一覧、操作ボタンを明確に分離すること
- 編集画面は中央に主要作業領域を配置すること
- プロパティ編集は右側パネルに集約すること
- 重要操作はツールバーまたは固定フッターに配置すること
- 削除や破壊的操作は確認ダイアログを必須にすること

## 5. ExcelライクUIの定義

AI Flow Designer のフローエディタはExcel風のレイアウトとする。
ただし、セル表計算ではなく、縦横のレーン構造を持つキャンバスとして扱う。

- 上方向に担当レーンを表示する
- 左方向に工程レーンを表示する
- レーンと工程の交点にノードを配置する
- ノード位置から担当と工程を視覚的に理解できる
- レーン幅、工程高さは将来調整可能とする

## 6. 画面共通構成

各画面は原則として以下の構成を持つ。

- ヘッダー領域
- タイトル領域
- 操作ボタン領域
- メインコンテンツ領域
- 補助パネルまたはダイアログ
- メッセージ表示領域

## 7. 操作共通方針

ユーザー操作は以下の方針とする。

- 作成、更新、削除、複製、出力は明示的な操作として提供する
- 保存前の変更状態を画面上で分かるようにする
- 画面遷移時に未保存変更がある場合は確認する
- エラー時は原因と次に行うべき操作を表示する
- 成功時は過剰な通知を避け、必要最低限のメッセージを表示する

## 8. 入力項目共通方針

入力項目は以下を基本とする。

- 必須項目には必須表示を行う
- 文字数上限をUIとAPIで一致させる
- IDは画面に直接入力させない
- 表示名と内部IDは分離する
- 選択項目はマスタまたは固定定義から選択させる
- AI向け項目は通常説明と区別する

## 9. ダイアログ共通方針

ダイアログは以下の用途で使用する。

- 新規作成
- 削除確認
- レーン削除時のノード移動選択
- 工程削除時のノード移動選択
- テンプレート選択
- 出力条件指定
- バージョン作成

破壊的操作では、実行ボタンを明確にし、キャンセルできることを保証する。

## 10. メッセージ共通方針

メッセージは以下に分類する。

- 成功メッセージ
- 警告メッセージ
- エラーメッセージ
- 確認メッセージ
- 入力補助メッセージ

内部例外やスタックトレースを画面に表示してはならない。

## 11. 権限との関係

初期実装ではGitHubログイン済みユーザーを利用者として扱う。
将来的に以下の権限差を画面に反映できるようにする。

- 参照のみ
- 編集可能
- 管理者
- テンプレート管理者
- 出力可能

## 12. 関連API

画面設計は以下のAPI設計と対応する。

- Auth API
- Project API
- Flow API
- FlowVersion API
- Lane API
- Stage API
- Node API
- Link API
- Comment API
- Export API
- Template API

## 13. 関連DB

画面設計は以下のDB設計と対応する。

- USER
- PROJECT
- FLOW
- FLOW_VERSION
- LANE
- STAGE
- NODE
- LINK
- COMMENT
- TEMPLATE

## 14. テスト観点

- 画面レイアウトが共通方針に従っていること
- 必須項目が分かること
- エラー時に分かりやすいメッセージが表示されること
- 未保存変更の警告が表示されること
- 破壊的操作に確認があること
- 構造化データに必要な項目を入力できること

## 15. 完了条件

本方針は、各画面設計書が本書の方針に従って詳細化され、画面、API、DB、テスト仕様へ展開できる状態になった場合に完了とする。
