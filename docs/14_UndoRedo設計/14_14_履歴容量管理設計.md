# 14_14_履歴容量管理設計

## 1. 目的

本書は、AI Flow DesignerのUndo / Redo履歴容量管理を定義する。

Undo / Redo履歴はFrontendメモリ上に保持するため、無制限に蓄積すると性能劣化やメモリ使用量増加につながる。

## 2. 基本方針

- maxHistoryCountを設定する。
- 初期値は100とする。
- 上限超過時は古いUndo履歴から削除する。
- Redo Stackも同じ上限を持つ。
- Commandに巨大データを持たせない。

## 3. maxHistoryCount

```ts
const DEFAULT_MAX_HISTORY_COUNT = 100;
```

将来、ユーザー設定で変更できるようにする。

## 4. 巨大Command対策

注意が必要なCommand:

- Template適用
- Replace Flow
- 大量Paste
- Layout適用

対策:

- 必要最小限のbefore / afterを保持する。
- File / Blobを保持しない。
- Canvas DOMを保持しない。

## 5. Flow切替時

Flowを切り替えた場合、Undo / Redo履歴はクリアする。

理由:

- Flow構造が変わる。
- ID参照の整合性が保証できない。

## 6. 保存成功時

保存成功時に履歴を消すかは運用方針による。

初期方針:

- undoStackは維持する。
- lastSavedCommandIdを保持する余地を残す。
- Backend Responseで大きく再同期した場合はクリアする。

## 7. テスト観点

- 101件目追加で最古履歴が破棄される。
- Flow切替で履歴がクリアされる。
- 保存成功では履歴が維持される。
- Backend再同期時に履歴をクリアできる。

## 8. 完了条件

- 履歴上限、破棄条件、Flow切替時の扱いが定義されている。
- 実装者が履歴容量管理を実装できる。
