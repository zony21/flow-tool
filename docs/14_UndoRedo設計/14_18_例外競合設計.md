# 14_18_例外競合設計

## 1. 目的

本書は、AI Flow DesignerのUndo / Redoにおける例外・競合設計を定義する。

Undo / RedoはFrontend編集中の状態を戻す処理であるが、Flow再読込、Snapshot復元、Backend保存後の再同期などにより、Commandが保持する対象IDが無効になる可能性がある。

## 2. 基本方針

- Command実行前に対象要素の存在を確認する。
- 対象が存在しない場合は安全に失敗させる。
- Flow再読込時はUndo / Redo Stackをクリアする。
- Snapshot復元時はUndo / Redo Stackをクリアする。
- Backend Responseで構造が大きく変わった場合はStackをクリアする。

## 3. 競合ケース

- Command対象Nodeが存在しない。
- Command対象Linkが存在しない。
- Lane削除後にMoveNodeCommandをUndoする。
- Snapshot復元後に古いCommandをRedoする。
- Backend保存でIDが再採番された。
- Template適用後にFlow再同期された。

## 4. 対応方針

| ケース | 対応 |
| --- | --- |
| 対象IDなし | Command実行を中止しStackをクリア候補にする |
| Flow再読込 | Stackクリア |
| Snapshot復元 | Stackクリア |
| Backend構造補正 | Stackクリア |
| 軽微なProperty差分 | Stack維持可能 |

## 5. Error表示

Undo / Redo失敗時はToastまたはStatus Barに表示する。

例:

```text
この操作は現在のFlow状態では元に戻せません。履歴を更新しました。
```

## 6. 禁止事項

- 存在しない対象へCommandを強制適用する。
- Snapshot復元後に古いRedoを許可する。
- Undo失敗後も壊れた履歴を残し続ける。

## 7. テスト観点

- 対象Node削除後のUndo失敗を安全に扱える。
- Flow再読込でStackがクリアされる。
- Snapshot復元でStackがクリアされる。
- Undo失敗時に通知が出る。

## 8. 完了条件

- Undo / Redo競合ケースと対応方針が定義されている。
- Stackクリア条件が明確である。
