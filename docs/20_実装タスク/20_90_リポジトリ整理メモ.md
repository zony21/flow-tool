# 20_90_リポジトリ整理メモ

## 1. 目的

本書は、AI Flow Designer の実装作業中に確認されたリポジトリ整理事項を記録する。

対象は以下とする。

- 追跡済み生成物の整理
- SQLite DB ファイルの除外
- docs API 例と実装APIルートの差分管理
- テスト不足箇所の今後対応

## 2. 現状

`.gitignore` では以下を除外対象としている。

- `node_modules/`
- `bin/`
- `obj/`
- `apps/backend/**/bin/`
- `apps/backend/**/obj/`
- SQLite DB 系ファイル

ただし、過去にGit管理へ追加済みのファイルは `.gitignore` だけでは追跡解除されない。

そのため、ローカル環境で `git status` に以下が `modified` として出る場合がある。

- `apps/backend/**/bin/**`
- `apps/backend/**/obj/**`
- `apps/frontend/node_modules/**`
- `*.db`
- `*.sqlite`
- `*.sqlite3`

## 3. 対応方針

生成物はソース管理対象外とする。

理由:

- build環境ごとに差分が出る
- PR/commit差分が汚れる
- 実装差分レビューの精度が落ちる
- SQLite DB はローカル実行状態に依存する

## 4. ローカルで実行する追跡解除手順

以下は、ファイル実体を消さずにGit追跡だけを解除する手順である。

```bash
git rm -r --cached apps/frontend/node_modules 2>/dev/null || true
git rm -r --cached apps/backend/**/bin 2>/dev/null || true
git rm -r --cached apps/backend/**/obj 2>/dev/null || true
git rm --cached *.db *.sqlite *.sqlite3 2>/dev/null || true
git rm --cached apps/backend/**/*.db apps/backend/**/*.sqlite apps/backend/**/*.sqlite3 2>/dev/null || true
```

PowerShellの場合は、対象ファイルを `git status` で確認しながら以下のように個別または一括で実行する。

```powershell
git rm -r --cached apps/frontend/node_modules
git rm -r --cached apps/backend/FlowDesigner.Api/bin
git rm -r --cached apps/backend/FlowDesigner.Api/obj
git rm -r --cached apps/backend/FlowDesigner.Application/bin
git rm -r --cached apps/backend/FlowDesigner.Application/obj
git rm -r --cached apps/backend/FlowDesigner.Domain/bin
git rm -r --cached apps/backend/FlowDesigner.Domain/obj
git rm -r --cached apps/backend/FlowDesigner.Infrastructure/bin
git rm -r --cached apps/backend/FlowDesigner.Infrastructure/obj
git rm -r --cached apps/backend/FlowDesigner.Tests/bin
git rm -r --cached apps/backend/FlowDesigner.Tests/obj
```

SQLite DB が追跡されている場合は、対象ファイル名を確認して個別に実行する。

```powershell
git rm --cached <対象DBファイル>
```

## 5. APIルート差分メモ

docs のAPI例には、以下のようなFlow単独ルート表現が残っている可能性がある。

```text
/api/flows/{flowId}
```

一方、現在の実装ではProject配下のルートを採用している。

```text
/api/projects/{projectId}/flows
/api/projects/{projectId}/flows/{flowId}
/api/projects/{projectId}/flows/{flowId}/structure
/api/projects/{projectId}/flows/{flowId}/versions
```

現時点では、実装ルートを正とする。

理由:

- Flow は Project に属する
- 権限判定は Project 単位で行う
- Router も `/projects/:projectId/flows/:flowId/...` を採用している
- ProjectId を含めることで、URL・API・権限チェックの責務が一致する

今後、docs側のAPI例はProject配下ルートへ統一する。

## 6. テスト不足メモ

現状、Project / Flow / Flow Structure / Version / Permission 403 の最小統合テストを追加済みである。

追加済み:

- Project作成
- Project更新
- Flow作成
- Flow更新
- Flow Structure保存
- Lane / Stage / Node / Link / Comment 保存
- Permission不足時の403確認
- Version作成
- Version比較
- Version復元
- Version異常系
- Revision競合
- Flow Structure保存時の必須Permission別403確認

引き続き、以下の実質テストが不足している。

- Controller属性によるPermission制御の統合確認
- Export / Template / Settings

次フェーズでは、まず以下を優先する。

1. Controller属性によるPermission制御の統合確認
2. Export / Template / Settings の基本テスト
3. Frontend側の主要Store / API clientテスト

## 7. 完了条件

- `git status` に生成物が出ない
- DB実体がGit差分に出ない
- APIルート方針が docs と実装で一致している
- Project / Flow / Structure の基本テストが追加されている
