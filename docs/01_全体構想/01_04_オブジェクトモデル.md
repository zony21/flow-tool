# 01_04_オブジェクトモデル

## 1. 本書の目的

本書は、AI Flow Designer のオブジェクトモデルを定義する。

オブジェクトモデルは、Frontend DTO、Backend Entity、DBテーブル、API入出力、Export、Version、Templateの前提となる。

## 2. 基本方針

本システムでは、Flow構造を明確なオブジェクトとして扱う。

画面上の図形や線を直接正とせず、Project、Flow、Lane、Stage、Node、Link、Commentを明確な責務を持つオブジェクトとして管理する。

## 3. 中核オブジェクト

|オブジェクト|役割|
|---|---|
|Project|複数Flowを束ねる管理単位|
|Flow|フロー設計の単位|
|Lane|担当主体|
|Stage|工程|
|Node|処理・判断・開始・終了など|
|Link|Node間の意味的接続|
|Comment|補足説明|
|Template|再利用可能な構造|
|Version|復元可能なSnapshot|
|UserSetting|ユーザー設定|
|EditorLock|編集ロック|
|ExportHistory|Export履歴|

## 4. 関係モデル

```text
Project
  └─ Flow
      ├─ Lane
      ├─ Stage
      ├─ Node
      │   └─ Comment(optional)
      ├─ Link
      └─ Comment(independent)
```

NodeはLaneとStageに所属する。LinkはFromNodeとToNodeを持つ。Commentは独立配置またはNode紐付けを許可する。

## 5. 集約の考え方

Backend実装では、Flowを中心とした集約を意識する。

Flow保存では以下を一括して扱う。

- Flow
- Lane
- Stage
- Node
- Link
- Comment

これらは同一Transactionで保存し、途中失敗時はRollbackする。

## 6. EntityとDTOの分離

EntityはDB永続化用のモデルである。

DTOはAPI入出力用のモデルである。

EntityをそのままAPI応答として返却しない。Service層でEntityからDTOへ変換する。

## 7. ID管理

主要オブジェクトはGUIDをIDとする。

Template適用時やVersion復元時は、必要に応じて新しいIDを採番する。

特にTemplate適用では、Template内のIDをそのままFlowへ流用しない。

## 8. ライフサイクル

ProjectはFlowより長いライフサイクルを持つ。

Flow配下のLane、Stage、Node、Link、CommentはFlow保存単位で管理する。

VersionはFlowのSnapshotとして保持する。

TemplateはFlowとは独立した再利用資産として保持する。

## 9. Versionとの関係

Versionは、Flow構造をSnapshot JSONとして保存する。

SnapshotにはLane、Stage、Node、Link、Commentを含める。

Undo / Redo履歴、選択状態、Hover状態、編集ロック状態はVersionに含めない。

## 10. Templateとの関係

Templateは再利用可能なFlow構造である。

TemplateにはLane、Stage、Node、Link、Commentを含めることができる。

適用時はID再採番と参照張替えを行う。

## 11. DBとの対応

オブジェクトモデルは04_DB設計の各テーブルに対応する。

|オブジェクト|主テーブル|
|---|---|
|Project|PROJECT|
|Flow|FLOW|
|Lane|LANE|
|Stage|STAGE|
|Node|FLOW_NODE|
|Link|FLOW_LINK|
|Comment|FLOW_COMMENT|
|Version|FLOW_VERSION|
|Template|TEMPLATE|

## 12. APIとの対応

APIでは用途別DTOを利用する。

例:

- ProjectListItemDto
- FlowDetailDto
- FlowSaveRequest
- FlowSaveResultDto
- TemplateDetailDto
- FlowVersionDetailDto

## 13. 制約事項

- Nodeは同一Flow内のLaneとStageに所属する
- LinkのFromNodeとToNodeは同一Flow内に存在する
- CommentがNode紐付けの場合、対象Nodeは同一Flow内に存在する
- ループ接続は許可する
- 論理削除を基本とする

## 14. テスト観点

- Project配下にFlowを作成できる
- Flow配下にLane、Stage、Node、Link、Commentを保存できる
- Linkが同一Flow内Nodeのみを参照する
- Template適用時にIDが再採番される
- Version SnapshotからFlow構造を復元できる

## 15. 完了条件

- 中核オブジェクトと責務が定義されている
- Entity、DTO、DB、APIの関係が明確である
- Flow保存、Version、Templateへ展開できる
