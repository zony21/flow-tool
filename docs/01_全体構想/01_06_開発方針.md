# 01_06_開発方針

## 1. 本書の目的

本書は、AI Flow Designer の開発方針を定義する。

開発方針は、Frontend、Backend、DB、設計書、Git運用、テスト、レビューの進め方を統一するための基準である。

## 2. 基本方針

本システムは段階的に設計・実装する。

全機能を一度に作り込むのではなく、Project、Flow、Editor、DB、API、Export、Version、Templateの順に、依存関係が明確な単位で実装する。

## 3. アーキテクチャ方針

Backendはレイヤを分離する。

```text
Controller
  ↓
Service
  ↓
Repository
  ↓
Entity / DbContext
```

Frontendは画面、Component、Composable、Store、API Clientを分離する。

EntityをFrontendへ直接返却せず、API DTOを利用する。

## 4. Frontend開発方針

FrontendはVue 3、TypeScript、PrimeVue、Vue Flow、Piniaで実装する。

主な方針:

- Page ComponentとCommon Componentを分離する
- Flowエディタ固有処理はComposableへ分離する
- Storeは機能単位で分割する
- API型はDTOとして定義する
- Vue Flow内部状態を保存の正にしない

## 5. Backend開発方針

BackendはASP.NET Core .NET 8で実装する。

主な方針:

- ControllerはHTTP入出力に集中する
- Serviceで業務処理を行う
- RepositoryでDB操作を行う
- ValidationをServiceまたは専用Validatorで行う
- Project権限はBackendで必ず確認する
- Flow保存はTransactionで行う

## 6. DB開発方針

DBはEF Core Migrationで管理する。

初期DBはSQLiteとし、将来SQL Serverへ移行可能な設計にする。

手作業DDLを正とせず、MigrationをGit管理する。

## 7. Git運用方針

作業は目的別ブランチで行う。

例:

```text
chatgpt/foundation-docs
chatgpt/enrich-04-db-design
```

mainへ直接反映せず、PRを通じて反映する。

## 8. Commit方針

Commit messageは変更内容が分かる名前にする。

例:

```text
docs: enrich system overview
docs: add db constraints design
feat: add project list api
fix: handle autosave conflict
```

## 9. レビュー方針

レビューでは以下を確認する。

- 設計方針と矛盾していないか
- EntityとDTOが混在していないか
- 権限制御がBackendで行われているか
- DB Transactionが必要箇所で使われているか
- 画面とAPIとDBの整合が取れているか

## 10. テスト方針

テストは以下の層で実施する。

- Unit Test
- Service Test
- Repository Test
- API Test
- Frontend Component Test
- E2E Test

特にFlow保存、AutoSave、Version復元、Template適用、Exportは重点的にテストする。

## 11. 品質基準

品質基準は以下とする。

- 設計書と実装が対応している
- 保存データが破損しない
- 権限なし操作ができない
- AutoSave失敗時に編集内容を失わない
- Export結果が構造化データと一致している
- Version復元後にFlow構造が整合している

## 12. ドキュメント方針

設計変更が発生した場合は、該当するdocsを更新する。

更新メモだけを残さず、正式設計書へ統合する。

不要なREADME、更新メモ、仮ファイルは削除する。

## 13. リリース方針

初期リリースでは、小規模利用を前提にSQLiteで運用する。

安定後、SQL Server移行、CI整備、環境分離、バックアップ運用を段階的に追加する。

## 14. 将来拡張方針

将来拡張は、Single Source of Truthを崩さない範囲で行う。

対象:

- AI DSL
- AIレビュー
- API仕様生成
- DB更新一覧生成
- リアルタイム共同編集
- コード生成

## 15. テスト観点

- Controller、Service、Repositoryが責務分離されている
- Frontend Storeが機能単位で分割されている
- MigrationでDB作成できる
- PR運用で変更を追跡できる
- 設計変更がdocsへ反映される

## 16. 完了条件

- 開発の基本方針が定義されている
- Frontend、Backend、DB、Git、Test、Documentの方針が揃っている
- 実装時の判断基準として利用できる
