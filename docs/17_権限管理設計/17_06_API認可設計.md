# 17_06_API認可設計

## 1. 目的

Backend API実行時の認可方針を定義する。

AI Flow Designerでは、Frontendの表示制御だけではなく、Backend API側で必ずPermissionを確認する。

## 2. 基本方針

- Frontend制御だけを信用しない
- 更新系・削除系・Export系APIでは必ずPermission確認を行う
- Project単位でRoleを取得する
- RoleからPermissionを取得する
- PermissionServiceで共通判定する
- 認可エラーは401と403を明確に分ける

## 3. 認証と認可の分離

| 区分 | 内容 | 失敗時 |
| --- | --- | --- |
| 認証 | Userがログイン済みか確認する | 401 Unauthorized |
| 認可 | Userが対象Projectで操作可能か確認する | 403 Forbidden |

## 4. Backend構成

| 部品 | 役割 |
| --- | --- |
| AuthMiddleware | GitHub OAuth後の認証状態を確認する |
| CurrentUserService | 現在のUser情報を取得する |
| PermissionService | Project単位のPermission判定を行う |
| RequirePermissionAttribute | Controller/Actionに必要Permissionを明示する |
| AuditLogService | 重要操作履歴を保存する |

## 5. PermissionService

### 5.1 目的

UserがProject内で指定Permissionを持つか判定する。

### 5.2 Interface案

```csharp
public interface IPermissionService
{
    Task<bool> CanAsync(
        Guid userId,
        Guid projectId,
        string permissionCode,
        CancellationToken cancellationToken = default);

    Task<IReadOnlyList<string>> GetPermissionsAsync(
        Guid userId,
        Guid projectId,
        CancellationToken cancellationToken = default);
}
```

### 5.3 判定処理

1. Userが有効か確認する
2. Projectが存在するか確認する
3. ProjectMemberを取得する
4. ProjectMember.STATUSがactiveか確認する
5. Roleを取得する
6. RolePermissionからPermission一覧を取得する
7. 指定Permissionが含まれるか確認する

### 5.4 falseになるケース

- Userが存在しない
- Userが無効
- Projectが存在しない
- ProjectMemberが存在しない
- ProjectMemberがactiveでない
- Roleが存在しない
- Roleに対象Permissionがない

## 6. RequirePermissionAttribute

### 6.1 目的

ControllerまたはActionに必要なPermissionを明示する。

### 6.2 使用例

```csharp
[RequirePermission(PermissionCodes.FlowUpdate)]
[HttpPut("projects/{projectId}/flows/{flowId}")]
public async Task<IActionResult> UpdateFlow(Guid projectId, Guid flowId, UpdateFlowRequest request)
{
    ...
}
```

### 6.3 ProjectId取得方針

ProjectIdは原則としてRoute Parameterから取得する。

標準名:

```text
projectId
```

Routeから取得できないAPIではRequest BodyまたはQuery Parameterから取得する。

ただし、認可対象がProjectに属するAPIでは必ずProjectIdを特定できる設計にする。

## 8. 参加Project一覧APIの扱い

Project一覧取得では、Userが参加しているProjectのみ返す。

このAPIでは個別ProjectIdが存在しないため、Project.Readを単純にAttribute指定しない。

処理方針:

1. CurrentUserを取得する
2. PROJECT_MEMBERから参加Projectを検索する
3. STATUS=activeのみ返す
4. Role情報も返す
5. Permission一覧も必要に応じて返す

## 9. 認可エラー

### 9.1 401 Unauthorized

以下の場合に返す。

- 未ログイン
- セッション無効
- GitHub OAuth認証情報が不正
- Userが無効

### 9.2 403 Forbidden

以下の場合に返す。

- ProjectMemberではない
- RoleにPermissionがない
- Projectがarchived状態で更新不可
- 最後のOwner保護に違反する

## 10. AuditLog対象

以下のAPIは成功時にAuditLogへ記録する。

- Project作成
- Project更新
- Project削除
- Member招待
- Member削除
- Role変更
- Flow削除
- Image削除
- Template削除
- Export実行
- Project設定変更

## 11. Controller実装方針

Controllerでは以下を守る。

- Permission判定を各Controller内に直書きしない
- 可能な限りRequirePermissionAttributeを使用する
- 複雑な条件はPermissionServiceまたはDomain Serviceで判定する
- Permission Codeは定数を使用する
- エラー形式は共通化する

## 12. Test観点

19_テスト仕様では以下を確認する。

- 未ログイン時は401
- Project未参加時は403
- Viewerの更新APIは403
- EditorのMember管理APIは403
- OwnerのRole変更APIは成功
- 最後のOwner降格は403
- 許可されたExportは成功

## 13. 完了条件

- API認可がPermission単位で整理されている
- Backendの共通判定部品が定義されている
- 401と403の使い分けが明確である
- API別Permission対応が定義されている
- 実装時にControllerへ展開できる粒度である
