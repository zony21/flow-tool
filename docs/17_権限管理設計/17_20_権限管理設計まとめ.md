## 3. 権限管理の基本思想

AI Flow Designerでは、認証と認可を分離する。

GitHub OAuthは「誰か」を判定するために利用する。

「どのProjectで何ができるか」は、AI Flow Designer側のProjectMember、Role、Permissionで判定する。

Frontendは利用者に分かりやすい操作制御を提供する。

Backendは安全性を担保する最終判断として、すべての重要APIでPermissionを確認する。

## 4. 初期Role

初期Roleは以下の3つとする。

| Role | 説明 |
| --- | --- |
| Owner | Project所有者。全操作可能 |
| Editor | Flow編集者。Project管理とMember管理は不可 |
| Viewer | 閲覧者。参照とExportのみ可能 |

Adminは初期実装では採用しない。

理由は、Roleを増やしすぎるとAPI認可、画面制御、テストケースが増え、初期実装精度が下がるためである。

将来の組織利用に備え、RoleとPermissionは分離しておく。

## 5. DB設計まとめ

権限管理では以下のテーブルを使用する。

| テーブル | 目的 |
| --- | --- |
| USER | GitHub OAuth認証済みUserを管理する |
| PROJECT_MEMBER | UserとProjectの参加関係、Roleを管理する |
| ROLE | Owner / Editor / Viewerを管理する |
| PERMISSION | 操作権限定義を管理する |
| ROLE_PERMISSION | RoleとPermissionの対応を管理する |
| PROJECT_INVITE | Project招待を管理する |
| AUDIT_LOG | 権限変更・重要操作履歴を管理する |

重要ルール:

- PROJECT_ID + USER_ID は一意
- Projectには必ず1人以上のOwnerが必要
- 最後のOwnerは削除・降格できない
- Role変更、Member削除、Project削除はAuditLogへ記録する

## 6. Permission設計まとめ

Permission Codeは以下の形式とする。

```text
Resource.Action
```

例:

```text
Project.Read
Flow.Update
Member.Invite
Export.Execute
```

BackendとFrontendで同じPermission Codeを使用する。

文字列直書きは避け、Backend/Frontendとも定数化する。

## 8. Backend実装まとめ

Backendでは以下を実装する。

- AuthMiddleware
- CurrentUserService
- PermissionService
- RequirePermissionAttribute
- AuditLogService

認可処理の基本流れ:

1. User認証状態を確認する
2. ProjectIdを特定する
3. ProjectMemberを取得する
4. Roleを取得する
5. RolePermissionを取得する
6. 必要Permissionを持つか確認する
7. 許可なら処理実行
8. 不許可なら403を返す

未ログインは401、権限不足は403とする。

## 9. Frontend実装まとめ

Frontendでは以下を実装する。

- Auth Store
- Project Permission Store
- Route Guard
- PermissionCodes定数
- Button/Menu/Dialog制御
- Viewer時のEditor読み取り専用化

Frontend制御はUX向上が目的であり、Backend認可の代替にはしない。

## 10. API認可まとめ

APIはPermission単位で認可する。

例:

| API | Permission |
| --- | --- |
| Project更新 | Project.Update |
| Project削除 | Project.Delete |
| Member招待 | Member.Invite |
| Role変更 | Member.ChangeRole |
| Flow更新 | Flow.Update |
| Imageアップロード | Image.Upload |
| Template適用 | Template.Apply |
| Export実行 | Export.Execute |
| Project設定変更 | Setting.ProjectUpdate |

Project一覧取得は、参加Projectのみ返す特殊APIとして扱う。

## 11. Testへ渡す観点

19_テスト仕様では最低限以下を確認する。

- 未ログイン時は401
- Project未参加時は403
- ViewerはFlow参照可能
- ViewerはFlow編集不可
- ViewerはExport可能
- EditorはFlow編集可能
- EditorはMember招待不可
- OwnerはRole変更可能
- 最後のOwner降格は不可
- Project設定変更はOwnerのみ可能
- APIで403を受けた場合、FrontendはPermission再取得を行う
