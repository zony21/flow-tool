# 17_02_ユーザーデータモデル設計

## 1. 目的

AI Flow Designerの認証済み利用者、Project参加者、Role、Permission、招待、監査ログを管理するためのデータ構造を定義する。

権限はUser単位ではなくProject参加単位で管理する。

## 2. 設計思想

GitHub OAuthで取得した情報はUserの識別に利用する。

Project内で何ができるかは、ProjectMember、Role、Permissionで判断する。

これにより、同じUserでもProject AではOwner、Project BではViewerのような運用を可能にする。

## 3. テーブル一覧

| テーブル | 目的 |
| --- | --- |
| USER | 認証済み利用者を管理する |
| PROJECT_MEMBER | Project参加者とRoleを管理する |
| ROLE | Role定義を管理する |
| PERMISSION | 操作権限定義を管理する |
| ROLE_PERMISSION | RoleとPermissionの対応を管理する |
| PROJECT_INVITE | Project招待を管理する |
| AUDIT_LOG | 権限変更・重要操作履歴を管理する |

## 4. USER

### 4.1 目的

GitHub OAuthで認証された利用者をアプリ内Userとして管理する。

### 4.2 カラム

| カラム | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| USER_ID | TEXT | ○ | User ID。UUID |
| LOGIN_PROVIDER | TEXT | ○ | 認証プロバイダ。初期値はGitHub |
| PROVIDER_USER_ID | TEXT | ○ | GitHub側のUser ID |
| PROVIDER_LOGIN | TEXT | ○ | GitHubログイン名 |
| DISPLAY_NAME | TEXT |  | 表示名 |
| EMAIL | TEXT |  | メールアドレス |
| AVATAR_URL | TEXT |  | アイコンURL |
| LAST_LOGIN_AT | DATETIME |  | 最終ログイン日時 |
| IS_ACTIVE | INTEGER | ○ | 有効状態。1=有効、0=無効 |
| CREATED_AT | DATETIME | ○ | 作成日時 |
| UPDATED_AT | DATETIME | ○ | 更新日時 |

### 4.3 制約

- LOGIN_PROVIDER + PROVIDER_USER_ID は一意
- IS_ACTIVE=0のUserはログインできてもアプリ利用不可

## 5. PROJECT_MEMBER

### 5.1 目的

UserがどのProjectにどのRoleで参加しているかを管理する。

### 5.2 カラム

| カラム | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| PROJECT_MEMBER_ID | TEXT | ○ | ProjectMember ID。UUID |
| PROJECT_ID | TEXT | ○ | Project ID |
| USER_ID | TEXT | ○ | User ID |
| ROLE_ID | TEXT | ○ | Role ID |
| STATUS | TEXT | ○ | active / invited / suspended |
| JOINED_AT | DATETIME |  | 参加日時 |
| INVITED_BY_USER_ID | TEXT |  | 招待者User ID |
| CREATED_AT | DATETIME | ○ | 作成日時 |
| UPDATED_AT | DATETIME | ○ | 更新日時 |

### 5.3 制約

- PROJECT_ID + USER_ID は一意
- Projectには必ず1人以上のOwnerが存在する
- 最後のOwnerは削除・Viewer変更不可

## 7. PERMISSION

### 7.1 目的

実行可能な操作をPermissionとして定義する。

### 7.2 Permission命名規則

形式:

```text
Resource.Action
```

例:

```text
Project.Read
Project.Update
Flow.Create
Flow.Update
Template.Apply
Export.Execute
```

### 7.3 カラム

| カラム | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| PERMISSION_ID | TEXT | ○ | Permission ID。UUID |
| PERMISSION_CODE | TEXT | ○ | Resource.Action形式 |
| RESOURCE_NAME | TEXT | ○ | Project / Flow / Template等 |
| ACTION_NAME | TEXT | ○ | Read / Create / Update等 |
| DESCRIPTION | TEXT |  | 説明 |
| CREATED_AT | DATETIME | ○ | 作成日時 |
| UPDATED_AT | DATETIME | ○ | 更新日時 |

## 10. AUDIT_LOG

### 10.1 目的

権限変更や重要操作を追跡する。

### 10.2 記録対象

- Project作成
- Project削除
- Member招待
- Member削除
- Role変更
- Project設定変更
- Export実行
- 画像削除

### 10.3 カラム

| カラム | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| AUDIT_LOG_ID | TEXT | ○ | AuditLog ID。UUID |
| PROJECT_ID | TEXT |  | Project ID |
| USER_ID | TEXT |  | 操作者User ID |
| ACTION_CODE | TEXT | ○ | 操作コード |
| TARGET_TYPE | TEXT | ○ | Project / Flow / Member等 |
| TARGET_ID | TEXT |  | 操作対象ID |
| BEFORE_JSON | TEXT |  | 変更前JSON |
| AFTER_JSON | TEXT |  | 変更後JSON |
| CREATED_AT | DATETIME | ○ | 作成日時 |

## 11. Entity設計方針

Backendでは以下のEntityを作成する。

- UserEntity
- ProjectMemberEntity
- RoleEntity
- PermissionEntity
- RolePermissionEntity
- ProjectInviteEntity
- AuditLogEntity

Entity名、DTO名、API Model名は04_DB設計、05_API設計、12_Backend設計と横断確認する。

## 12. 完了条件

- UserとProject権限が分離されている
- Project単位でRoleを保持できる
- RoleとPermissionが多対多で管理できる
- 招待状態をDBで追跡できる
- 重要操作をAuditLogに記録できる
- EF Core Entityへ変換可能な粒度でカラムが定義されている
