# 17_01_権限管理概要

## 1. 目的

AI Flow Designerの認証、利用者、Role、Permission、Project共有、画面制御、API認可の基本方針を定義する。

本章の目的は、複数ユーザーでProjectを扱う場合でも、FrontendとBackendで同じ権限判断を行い、意図しない編集・削除・Export・設定変更を防止することである。

## 2. 前提

AI Flow DesignerはGitHub OAuthで認証する。

ただし、GitHub認証で分かるのは「誰であるか」までであり、「どのProjectで何ができるか」は本アプリ側のProjectMember、Role、Permissionで管理する。

## 3. 設計思想

認証と認可は分離する。

- 認証: GitHub OAuthにより利用者を識別する
- 認可: Project単位のRoleとPermissionで操作可否を判断する

Frontendの表示制御は利用者体験向上のために行う。

ただし、Frontend制御は安全性の根拠にしない。

すべての更新系API、削除系API、設定変更API、Export APIはBackend側で必ずPermission確認を行う。

## 4. 管理対象

| 対象 | 役割 |
| --- | --- |
| User | 認証済み利用者を表す |
| ProjectMember | UserとProjectの参加関係を表す |
| Role | Project内での役割を表す |
| Permission | 実行可能な操作を表す |
| Invite | Project招待状態を表す |
| AuditLog | 権限変更・重要操作履歴を表す |

## 5. Role

初期Roleは以下とする。

| Role | 位置付け |
| --- | --- |
| Owner | Project所有者。全操作可能。Project削除、Member管理、Role変更が可能 |
| Editor | Flow編集者。Project設定やMember管理は不可 |
| Viewer | 閲覧者。参照と許可されたExportのみ可能 |

Adminは初期実装では採用しない。

理由は、OwnerとEditorの中間Roleを最初から増やすと、Role MatrixとAPI認可が複雑化し、初期実装精度が下がるためである。

将来、組織利用が必要になった場合はAdminを追加可能な構造にする。

| Project | 作成、参照、更新、削除、Member管理 |
| Flow | 作成、参照、更新、削除、Version作成 |

| Link | 追加、更新、削除 |

## 8. 認可処理の基本流れ

1. APIリクエストを受け取る
2. 認証済みUserを取得する
3. 対象Projectを特定する
4. ProjectMemberを取得する
5. Roleを取得する
6. Roleに紐づくPermissionを確認する
7. 許可されていれば処理を実行する
8. 不許可なら403を返す
9. 重要操作はAuditLogへ記録する

## 9. Frontend制御

Frontendでは以下を行う。

- Route Guardによる画面遷移制御
- Pinia Auth StoreによるログインUser保持
- Project Permission StoreによるProject別Permission保持
- Button/Menu/Dialogの活性・非活性制御
- Viewer時のEditor読み取り専用化

Frontendで非表示・非活性にしても、Backend API側の認可は省略しない。

## 10. Backend制御

Backendでは以下を行う。

- GitHub OAuth認証後のUser特定
- PermissionServiceによる共通判定
- ControllerまたはAction単位のPermission指定
- 更新系API実行前のProjectMember確認
- 403 Forbidden / 401 Unauthorizedの明確な返却
- Role変更、Member変更、Project削除の監査ログ保存

## 11. 完了条件

- 認証と認可の責務が分離されている
- Project単位でRoleを保持する方針が明確である
- Role Matrixにより主要操作の可否が判断できる
- Frontend制御とBackend認可の責務が明確である
- 後続のDB設計、API設計、画面設計へ展開できる
