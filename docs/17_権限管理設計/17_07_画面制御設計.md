# 17_07_画面制御設計

## 1. 目的

Frontend画面上で、利用者のRoleとPermissionに応じた操作制御を行うための設計を定義する。

画面制御の目的は、利用者に実行できない操作を見せない、または非活性にすることで誤操作を防ぐことである。

ただし、Frontend制御はセキュリティ境界ではない。

最終的な操作可否は必ずBackend APIで確認する。

## 2. 対象

| 対象 | 制御内容 |
| --- | --- |
| Route | 画面遷移可否 |
| Header | Save / Version / Export / AI Review等の表示制御 |
| Toolbar | Undo / Redo / Template / Search等の操作制御 |
| Node Palette | Node追加可否 |
| Flow Canvas | 編集可否、読み取り専用制御 |
| Property Panel | 入力可否 |
| Dialog | 実行ボタン活性制御 |
| Button | 表示・非表示・活性制御 |
| Menu | 操作メニュー表示制御 |

## 3. 基本方針

- Login User情報はAuth Storeで管理する
- Project別PermissionはProject Permission Storeで管理する
- 画面ではRole名ではなくPermissionで判定する
- Viewerの場合、Editorは読み取り専用にする
- 非活性にした理由を必要に応じてTooltip表示する
- Backend APIの403は画面側で適切に表示する

## 4. Pinia Store構成

### 4.1 Auth Store

目的:

ログインUserと認証状態を保持する。

保持項目:

```ts
interface AuthState {
  isAuthenticated: boolean
  userId: string | null
  providerLogin: string | null
  displayName: string | null
  avatarUrl: string | null
}
```

主な処理:

- ログイン状態取得
- ログアウト
- Current User取得

### 4.2 Project Permission Store

目的:

Project単位のRoleとPermissionを保持する。

保持項目:

```ts
interface ProjectPermissionState {
  projectId: string | null
  roleCode: 'OWNER' | 'EDITOR' | 'VIEWER' | null
  permissions: string[]
}
```

Getter:

```ts
const can = (permissionCode: string): boolean => {
  return permissions.includes(permissionCode)
}
```

## 5. Route Guard

### 5.1 目的

ProjectやFlowの画面に入る前に参照権限を確認する。

### 5.2 判定例

| Route | 必要Permission |
| --- | --- |
| /projects | ログイン済み |
| /projects/:projectId | Project.Read |
| /projects/:projectId/flows/:flowId | Flow.Read |
| /projects/:projectId/settings | Setting.ProjectUpdate |
| /projects/:projectId/members | Member.Read |

### 5.3 処理流れ

1. 認証状態を確認する
2. 未ログインならLoginへ遷移する
3. projectIdがある場合、Project Permissionを取得する
4. 必要Permissionを確認する
5. 不許可なら403画面へ遷移する

## 6. Editor読み取り専用制御

Viewerの場合、Flow Canvasは読み取り専用にする。

読み取り専用時の制御:

- Node移動不可
- Node追加不可
- Node削除不可
- Link追加不可
- Property Panel入力不可
- Save非活性
- Template適用不可
- Exportは可能

Editor / Ownerの場合は編集可能とする。

ただし、Projectがarchivedの場合はOwnerでも編集不可とする。

## 7. Button制御

| Button | Permission | 制御 |
| --- | --- | --- |
| Save | Flow.Update | ない場合は非活性 |
| Add Node | Node.Update | ない場合は非表示 |
| Delete Node | Node.Update | ない場合は非活性 |
| Add Link | Link.Update | ない場合は非活性 |
| Upload Image | Image.Upload | ない場合は非表示 |
| Apply Template | Template.Apply | ない場合は非活性 |
| Create Template | Template.Create | ない場合は非表示 |
| Export | Export.Execute | ない場合は非表示 |
| Project Settings | Setting.ProjectUpdate | ない場合は非表示 |
| Invite Member | Member.Invite | ない場合は非表示 |

## 8. Menu制御

Context MenuはPermissionにより表示項目を制御する。

Node右クリック例:

| Menu | Permission |
| --- | --- |
| 詳細表示 | Flow.Read |
| 編集 | Node.Update |
| 複製 | Node.Update |
| 削除 | Node.Update |
| コメント追加 | Comment.Update |

Viewerでは「詳細表示」のみ表示する。

## 9. Dialog制御

Dialogでは実行ボタンの活性をPermissionで制御する。

例:

| Dialog | 実行Button | Permission |
| --- | --- | --- |
| Project設定 | 保存 | Setting.ProjectUpdate |
| Member招待 | 招待 | Member.Invite |
| Role変更 | 変更 | Member.ChangeRole |
| Template保存 | 保存 | Template.Create |
| Imageアップロード | アップロード | Image.Upload |

## 10. API 403時の画面挙動

APIから403が返った場合は、画面側の権限情報が古い可能性がある。

処理:

1. Project Permissionを再取得する
2. 再取得後も不可ならエラーメッセージ表示
3. 編集中データがある場合は破棄せず保持する
4. 必要に応じて読み取り専用へ切り替える

表示文言例:

```text
この操作を実行する権限がありません。Projectの権限が変更された可能性があります。
```

## 11. UI実装方針

### 11.1 共通関数

```ts
const can = (permissionCode: string): boolean => {
  return projectPermissionStore.can(permissionCode)
}
```

### 11.2 使用例

```vue
<Button
  label="Save"
  :disabled="!can(PermissionCodes.FlowUpdate)"
/>
```

### 11.3 v-if と disabled の使い分け

| 制御 | 用途 |
| --- | --- |
| v-if | そもそも見せる必要がない管理系操作 |
| disabled | 存在は分かるが実行不可にしたい操作 |

例:

- Member招待はViewerに見せないためv-if
- SaveはEditor画面の文脈上存在するためdisabled

## 12. Test観点

19_テスト仕様では以下を確認する。

- ViewerはFlowを開けるが編集できない
- ViewerはExportできる
- EditorはFlow編集できるがMember招待できない
- OwnerはProject設定を変更できる
- 権限変更後に403を受けた場合、Permission再取得が行われる

## 13. 完了条件

- Route Guardの制御方針が明確である
- Pinia Store構成が定義されている
- Button/Menu/DialogのPermission制御が定義されている
- Viewer時の読み取り専用制御が明確である
- Backend認可と矛盾しないFrontend制御になっている
