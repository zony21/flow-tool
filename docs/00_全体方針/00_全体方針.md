# 00_全体方針

## 1. 本書の目的

本書は、AI Flow Designer 全体の設計方針を定義する。

本書は個別設計書の入口となる総括文書であり、システムコンセプト、設計思想、技術方針、品質方針、実装時の判断基準をまとめる。

詳細は `00_01_システムコンセプト.md`、`00_02_設計思想.md`、`00_03_命名規則.md`、`00_04_フォルダ構成.md`、`00_05_技術スタック.md` に分割して記載する。

## 2. プロジェクト概要

AI Flow Designer は、業務フロー、システムフロー、設備連携フローを構造化データとして設計するWebアプリケーションである。

本システムは、ユーザーが視覚的にFlowを編集できるだけでなく、そのFlowをAIが理解しやすい形式へ出力することを目的とする。

## 3. システム全体像

本システムは以下の構成で実装する。

- Frontend: Vue 3 / TypeScript / PrimeVue / Vue Flow / Pinia
- Backend: ASP.NET Core .NET 8
- DB: SQLite 初期運用、将来SQL Server移行
- Auth: GitHub OAuth
- Export: Mermaid、PDF、JSON、将来AI DSL

Frontendは画面操作とFlow描画を担当する。BackendはAPI、認証認可、DB保存、Export生成、Version管理を担当する。DBは構造化Flowデータを保持する。

## 4. Single Source of Truth

本システムの最重要方針は、構造化Flowデータを唯一の正とすることである。

画面上の図形や線は、構造化Flowデータを視覚化した結果であり、保存やExportの正ではない。

正となる情報は、Project、Flow、Lane、Stage、Node、Link、Commentとして保持する。

この方針により、同じFlowデータから画面表示、Mermaid、JSON、PDF、Version Snapshot、Templateを安定して生成できる。

## 5. 設計原則

本システムでは以下を設計原則とする。

- 表示とデータを分離する
- EntityをAPIへ直接返却しない
- DTOを通じてFrontendとBackendを連携する
- Flow編集はCommandとして扱う
- Undo / Redoは画面操作履歴として扱う
- Versionは復元可能なSnapshotとして扱う
- AutoSaveは編集内容保護として扱う
- Template適用時はIDを再採番する

## 6. 開発方針

初期開発では、実装範囲を明確に固定する。

初期対象は、Project管理、Flow管理、Flowエディタ、Node / Link / Lane / Stage / Comment編集、Undo / Redo、AutoSave、Version、Template、Export、GitHub OAuthとする。

AIチャット、リアルタイム共同編集、本格的なコード生成は将来拡張とする。

## 7. 非機能方針

非機能面では以下を重視する。

- Flowエディタが安定して動作すること
- 1000 Node規模のFlowを扱えること
- AutoSave失敗時に編集内容を失わないこと
- 権限のないProjectやFlowを参照できないこと
- SQLite初期運用からSQL Server移行できること
- 設計書だけで実装判断ができること

## 8. 品質方針

品質は、画面が動くことだけでは判断しない。

DB、API、DTO、画面、Export、Version復元、Template適用、権限制御が整合していることを品質基準とする。

設計書は、実装者が迷わず実装できる粒度まで記述する。

## 9. セキュリティ方針

認証はGitHub OAuthを初期採用する。

内部ユーザーはUSER_ACCOUNTで管理し、外部認証情報と内部ユーザーを分離する。

APIでは認証とProject権限を必ず確認する。Frontend上の非表示や非活性は補助であり、Backend認可を正とする。

Tokenや機密情報はログやExportへ出力しない。

## 10. 保守性方針

FrontendはComponent、Composable、Storeを分離する。

BackendはController、Service、Repository、Entity、DTOを分離する。

DBはMigrationで管理し、手作業DDLを正としない。

一時メモや更新メモは正式設計書へ統合し、不要なファイルとして残さない。

## 11. 拡張性方針

将来拡張を見越し、NodeType、Export形式、Template、AI向け出力は拡張可能にする。

ただし、初期実装では範囲を広げすぎない。拡張点を設計しつつ、初期対象は明確に固定する。

## 12. ドキュメント運用方針

設計書は `docs` 配下で管理する。

フォルダ番号とフォルダ名は原則変更しない。

各設計書は、目的、方針、実装への影響、関連画面、関連API、関連DB、テスト観点を可能な限り記載する。

## 13. AI活用方針

本システムでは、AIに画像としてフロー図を読ませるのではなく、構造化データをAIに渡すことを重視する。

AI向け出力では、Lane、Stage、Node、Link、Commentの意味情報を省略しない。

将来的に、設計レビュー、API仕様生成、DB更新一覧生成、通信一覧生成、テスト観点生成へ拡張する。

## 14. 他設計書との関係

本書は全体方針を示す。

詳細は以下で定義する。

- 02_要件定義: 機能要件、非機能要件、受入条件
- 03_画面設計: 画面、状態、Dialog、通知
- 04_DB設計: テーブル、制約、Index、Migration
- 05_API設計: API、DTO、エラー、認可
- 06以降: Editor、Export、AI構造化、実装設計

## 15. 実装時の遵守事項

実装時は以下を守る。

- 構造化Flowデータを正とする
- Vue Flow内部状態を保存の正にしない
- EntityをAPI応答として直接返さない
- Flow保存はTransactionで行う
- 権限制御はBackendで必ず行う
- Export前にValidationを行う
- Version復元前に現在状態を保存する
- 不要な一時ファイルをdocsに残さない

## 16. 完了条件

00_全体方針は以下を満たすことで完成とする。

- システムの目的と全体像が説明されている
- Single Source of Truth方針が明確である
- 技術方針と設計原則が定義されている
- 他設計書との関係が明確である
- 実装時の判断基準として利用できる
