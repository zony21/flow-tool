# 00_02_設計思想

## 1. 本書の目的

本書は、AI Flow Designer の設計思想を定義する。

設計思想は、画面、DB、API、Export、Version、Template、Undo / Redo、AutoSaveの各設計判断の根拠となる。

## 2. 基本思想

本システムでは、見た目としての図ではなく、意味を持つ構造化データを中心に設計する。

Flowは、Lane、Stage、Node、Link、Commentの集合であり、それぞれが明確な責務を持つ。

- Lane は担当主体を表す
- Stage は工程を表す
- Node は処理・判断・開始・終了などを表す
- Link は処理間の意味的接続を表す
- Comment は補足情報を表す

## 3. 表示とデータの分離

画面に表示される図形、線、色、位置は、構造化データを視覚化した結果である。

保存、Export、Version、Template、AI向け出力では、画面表示状態ではなくDBの構造化データを利用する。

この分離により、将来UIライブラリや描画方式が変わっても、データと成果物生成の正しさを維持できる。

## 4. 拡張性の考え方

NodeType、Export形式、Template形式、AI向け出力形式は将来拡張される前提で設計する。

ただし、拡張しやすさを優先しすぎて初期実装が曖昧にならないよう、初期対応範囲は明確に固定する。

初期対応範囲を固定し、その上でExtensionJsonやschemaVersionを用意する。

## 5. 競合防止の考え方

同一Flowの同時編集は初期開発では対象外とする。

Flow単位で編集ロックを取得し、ロック保持者のみが編集できる。Viewerやロック未取得者は読み取り専用で開く。

保存時はRevisionを利用して競合を検出する。

## 6. 履歴管理の考え方

Undo / Redo は画面操作の一時的な履歴であり、DBに永続化しない。

Version はユーザーが明示的に残す復元可能な履歴であり、FLOW_VERSIONにSnapshot JSONとして保存する。

AutoSave は編集中の損失を防ぐための保存であり、Versionとは責務を分離する。

## 7. Exportの考え方

Exportは、表示中の見た目をそのまま出す処理ではない。

Exportは、構造化Flowデータから目的別の成果物を生成する処理である。

これにより、Mermaid、JSON、PDFなどの出力で同じ意味情報を共有できる。

## 8. 完了条件

- 表示とデータの責務が分離されている
- Flow構造の各要素の意味が定義されている
- 編集ロック、Revision、Version、AutoSaveの責務が分離されている
- 画面、DB、API、Export設計へ展開できる
