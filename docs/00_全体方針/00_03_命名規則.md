# 00_03_命名規則

## 1. 本書の目的

本書は、AI Flow Designer の命名規則を定義する。

命名規則は、Frontend、Backend、DB、API、DTO、設計書、GitHubブランチ、Migrationの表記ゆれを防ぎ、実装者が迷わず同じルールで開発できる状態を作るための基準である。

## 2. 基本方針

- 意味が分かる名前を優先する
- 省略しすぎない
- 同じ概念には同じ単語を使う
- 画面名、API名、DB名、DTO名の対応を取りやすくする
- 日本語設計書と英語コードの対応を明確にする

## 3. 用語統一

本システムで使用する主要用語は以下で統一する。

|日本語|英語|用途|
|---|---|---|
|プロジェクト|Project|最上位管理単位|
|フロー|Flow|設計対象のフロー|
|レーン|Lane|担当主体|
|工程|Stage|処理工程|
|ノード|Node|処理単位|
|リンク|Link|処理間接続|
|コメント|Comment|補足情報|
|テンプレート|Template|再利用構造|
|バージョン|Version|復元可能履歴|
|出力|Export|成果物生成|
|編集ロック|EditorLock|排他編集制御|

## 4. Frontend命名

Vue ComponentはPascalCaseとする。

例:

```text
ProjectListPage.vue
FlowEditorPage.vue
NodePropertyPanel.vue
AppConfirmDialog.vue
```

ComposableはcamelCaseで `use` から始める。

```text
useFlowEditor.ts
useEditorLock.ts
useExportValidation.ts
```

Pinia StoreはcamelCaseで、ファイル名は `xxxStore.ts` とする。

```text
authStore.ts
projectStore.ts
flowEditorStore.ts
```

## 5. Backend命名

C#のClass、Record、EnumはPascalCaseとする。

```csharp
ProjectService
FlowEditorService
FlowSaveRequest
FlowSaveResultDto
```

メソッドはPascalCaseとする。

```csharp
CreateProjectAsync
SaveFlowAsync
RestoreVersionAsync
```

非同期メソッドは `Async` を付与する。

## 6. DTO命名

DTOは用途が分かる名前にする。

|種類|命名例|
|---|---|
|一覧項目|ProjectListItemDto|
|詳細|FlowDetailDto|
|作成要求|ProjectCreateRequest|
|更新要求|ProjectUpdateRequest|
|保存要求|FlowSaveRequest|
|結果|FlowSaveResultDto|

Entityを直接API応答に利用しない。

## 7. DB命名

DBテーブル名とカラム名はUPPER_SNAKE_CASEを基本とする。

例:

```text
PROJECT
FLOW
FLOW_NODE
FLOW_LINK
FLOW_VERSION
```

ID列は `{ENTITY}_ID` とする。

```text
PROJECT_ID
FLOW_ID
NODE_ID
LINK_ID
```

論理削除は `IS_DELETED` とする。

監査項目は以下で統一する。

```text
CREATED_AT
CREATED_BY
UPDATED_AT
UPDATED_BY
```

## 8. API命名

URLは複数形・小文字・ハイフン区切りを基本とする。

例:

```text
/api/projects
/api/projects/{projectId}/flows
/api/flows/{flowId}/versions
/api/flows/{flowId}/exports/json
```

Controller名は対象リソースに合わせる。

```text
ProjectsController
FlowsController
TemplatesController
ExportsController
```

## 9. 設計書ファイル命名

設計書は以下の形式を基本とする。

```text
{フォルダ番号}_{連番}_{内容}.md
```

例:

```text
03_08_Flowエディタ画面.md
04_06_NODE詳細設計.md
05_03_FlowAPI設計.md
```

更新メモや一時ファイルは残さず、正式設計書へ統合する。

## 10. Gitブランチ命名

ChatGPTで作成する設計書改修ブランチは以下を基本とする。

```text
chatgpt/{目的}
```

例:

```text
chatgpt/foundation-docs
chatgpt/enrich-04-db-design
```

## 11. Migration命名

EF Core Migrationは内容が分かる名前にする。

例:

```text
AddFlowCoreTables
AddEditorLockTable
AddExportHistoryTable
AddFlowIndexes
```

## 12. 禁止事項

- 同じ概念を別名で呼ぶ
- DTOとEntityを同じ名前にする
- 一時ファイルを正式設計書フォルダに残す
- `data`, `info`, `temp` のような曖昧な名前を多用する
- 日本語名と英語名の対応が不明なまま実装する

## 13. 完了条件

- Frontend、Backend、DB、API、設計書の命名方針が定義されている
- Project、Flow、Lane、Stage、Node、Link、Commentの用語が統一されている
- 実装者が命名で迷わない状態になっている
